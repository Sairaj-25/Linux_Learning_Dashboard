ifconfig
Network Interface
Display and configure network interfaces ⚠ DEPRECATED
1Command Overview

ifconfig (interface configuration) displays and configures network interfaces — IP addresses, netmasks, broadcast, MAC addresses, and interface statistics. It is part of the net-tools package which is deprecated on modern Linux distributions. The modern replacement is ip from the iproute2 package.

Despite being deprecated, ifconfig remains widely referenced in legacy documentation, older CI/CD scripts, Docker base images, and in interview questions where knowing the "old vs new" distinction is expected.

⚠️
Deprecated — Use ip Instead ifconfig is not installed by default on modern Ubuntu (20.04+), RHEL 8+, or Debian 10+. Install with apt-get install net-tools. In production scripts, always use ip commands instead.
2Syntax
BASH
ifconfig [interface] [options]

  No args     — Show all active interfaces
  interface   — Specific interface (eth0, ens33, lo, docker0)
  -a          — Show ALL interfaces including inactive

6Interview Questions & Answers
Q1Why is ifconfig deprecated and what should you use instead?
A:ifconfig is part of the net-tools package which hasn't been actively developed since 2001. It lacks support for modern Linux networking features: IPv6 (partially), network namespaces, VLANs, bonding, and advanced routing. The iproute2 suite (ip command) is the modern replacement — actively maintained, namespace-aware, supports all modern features. Use ip addr show instead of ifconfig.
7Common Mistakes & Pro Tips
❌
Mistake — Using ifconfig in new scripts Any script using ifconfig will break on fresh Ubuntu 20.04+ or RHEL 8+ systems where net-tools isn't installed. Always use ip addr in scripts.
✅
Pro Tip — Interview mention When asked about ifconfig, always proactively mention: "ifconfig is deprecated; I prefer ip addr show in production scripts. ifconfig is still useful for quick inspection on legacy systems or when net-tools is already installed." This shows maturity.
ip
Modern Network Management
ip addr · ip link · ip route — the unified iproute2 network command
1Command Overview

The ip command from the iproute2 package is the modern, unified network management tool for Linux. It replaces ifconfig, route, arp, and netstat with a consistent, feature-rich interface. It supports network namespaces (critical for Docker/Kubernetes), IPv6, VLAN tagging, bonding, and all modern Linux networking features.

In DevOps and cloud engineering, ip is used daily for: verifying assigned IPs on new cloud instances, managing routes, troubleshooting container networking, inspecting network namespaces, and writing portable network configuration scripts.

2Syntax
BASH
ip [OPTIONS] OBJECT [COMMAND]

  OBJECT   — addr | link | route | neigh | rule | tunnel | netns | maddress
  COMMAND  — show | add | del | set | flush | replace
  OPTIONS  — -4 (IPv4) | -6 (IPv6) | -s (stats) | -h (human) | -j (JSON) | -br (brief)

5Real-World Use Cases
  • Cloud instance verification: After launching EC2/GCP/Azure VM, ip addr show confirms the private IP assignment before configuring security groups or DNS records.
  • Kubernetes node networking: ip route show reveals pod CIDR routes added by CNI plugins (Calico, Flannel). Essential for troubleshooting pod connectivity.
  • Docker container debugging: ip netns exec allows inspecting the network namespace of a running container — equivalent to docker exec container ip addr.
  • Jumphost/bastion routing: ip route get DEST quickly shows which interface and gateway would be used to reach a target — invaluable for multi-homed server debugging.
6Interview Questions & Answers
Q1What is the modern replacement for ifconfig and what advantages does it have?
A:The ip command from iproute2. Advantages: (1) Network namespace support — critical for Docker and Kubernetes, (2) Full IPv6 support, (3) Supports advanced features: VLANs, bonding, bridging, tunnels, (4) Consistent syntax across all network objects, (5) JSON output (ip -j) for automation and parsing, (6) Actively maintained. Mapping: ifconfig → ip addr, route → ip route, arp → ip neigh.
Q2Scenario: A Kubernetes pod cannot reach the external internet. How do you debug the routing?
A:1. From the node: ip route get 8.8.8.8 — verify the host itself can route. 2. Check pod network: kubectl exec -it pod -- ip route show. 3. Verify CNI routes: ip route show on the node — CNI should have added pod CIDR routes. 4. Check iptables NAT rules: iptables -t nat -L POSTROUTING -n — ensure MASQUERADE/SNAT rule exists for pod CIDR. 5. Test from pod network namespace directly.
7Common Mistakes & Pro Tips
✅
Pro Tip — ip route get is underused ip route get 10.20.30.40 immediately shows which interface and gateway would handle traffic to that destination — saving minutes of routing table analysis. Essential for multi-homed servers.
💡
Shorthand ip a = ip addr show, ip r = ip route show, ip l = ip link show. The command accepts any unambiguous prefix of the subcommand.
ping
Network Connectivity Test
Send ICMP Echo requests to test host reachability and measure latency
1Command Overview

ping sends ICMP (Internet Control Message Protocol) Echo Request packets to a target host and waits for Echo Reply responses. It measures round-trip time (RTT), detects packet loss, and verifies basic IP connectivity. It is the most fundamental network diagnostic tool.

Critical caveat: a host not responding to ping does NOT mean it's unreachable — many cloud firewalls and security groups block ICMP by default. Always interpret ping results in context.

2Syntax & Key Flags
BASH
ping [OPTIONS] HOST

  HOST — IP address or hostname (e.g., google.com or 8.8.8.8)
FlagDescriptionExample
-c NSend exactly N packets then stop (Linux default: infinite)ping -c 4 google.com
-i SECSInterval between packets (default: 1 second)ping -i 0.2 host
-t TTLSet IP Time-To-Liveping -t 64 host
-s SIZESet packet size in bytes (default: 56)ping -s 1472 host
-W SECSWait time for response in secondsping -W 2 host
-I IFACESend packets from specific interfaceping -I eth1 host
-qQuiet — show only summary statisticsping -q -c 10 host
-fFlood ping — maximum speed (root only)sudo ping -f -c 1000 host
-4 / -6Force IPv4 or IPv6ping -4 google.com
4Practical Examples
BASH
# Basic connectivity check (stops after 4 packets)
ping -c 4 google.com
# PING google.com (142.250.80.46): 56 data bytes
# 64 bytes from 142.250.80.46: icmp_seq=0 ttl=118 time=12.4 ms
# --- google.com ping statistics ---
# 4 packets transmitted, 4 received, 0% packet loss, time 3005ms
# rtt min/avg/max/mdev = 11.8/12.4/13.1/0.5 ms

# Test internal gateway reachability
ping -c 3 192.168.1.1

# Quick test for scripting (exit 0 = reachable, exit 1 = not)
ping -c 1 -W 2 8.8.8.8 &>/dev/null && echo "Online" || echo "Offline"

# Check for packet loss with more samples
ping -c 100 -q database.internal
# 100 packets transmitted, 98 received, 2% packet loss

# Test MTU size (detect fragmentation issues)
ping -M do -s 1472 -c 3 gateway.ip   # -M do = don't fragment

# Monitoring script — alert if host goes down
while true; do
    if ! ping -c 1 -W 2 10.0.1.5 &>/dev/null; then
        echo "$(date): Host 10.0.1.5 DOWN!" | tee -a /var/log/ping_monitor.log
    fi
    sleep 30
done
6Interview Questions & Answers
Q1Ping fails to a host. Does this mean the host is down?
A:Not necessarily. Multiple explanations: (1) ICMP is blocked by firewall/security group — most common in cloud (AWS blocks ICMP by default), (2) Host is reachable but OS has ICMP disabled (net.ipv4.icmp_echo_ignore_all=1), (3) Network routing issue — packets can't reach the host, (4) Host is genuinely down. To distinguish: try connecting to a known open port with nc -zv host PORT or curl -I http://host. If those work, the host is up — ICMP is simply blocked.
Q2What does a high ping RTT but 0% packet loss indicate?
A:Network congestion or high geographic distance — packets are arriving but the network path is slow. High RTT (>100ms for local network, >300ms for intercontinental) with 0% loss indicates: (1) routing through many hops, (2) network congestion causing queuing delays, (3) CPU overload on the target (ICMP processing is low priority). For a local datacenter, RTT should be <1ms; for cross-region AWS, 50-100ms is typical.
7Common Mistakes & Pro Tips
❌
Mistake — Using ping without -c on Linux Linux ping runs infinitely without -c. In scripts, always use ping -c N or the script will hang forever waiting for Ctrl+C.
✅
Pro Tip — Ping exit code for scripting Ping exits with 0 if at least one reply was received, 1 if none received (unreachable), 2 for other errors. Use this in deployment health checks: ping -c 1 -W 2 $HOST || { echo "Host unreachable"; exit 1; }
netstat
Network Statistics
Display network connections, ports, routing tables, stats ⚠ DEPRECATED
1Command Overview

netstat displays network connections (TCP/UDP), listening ports, routing tables, interface statistics, multicast memberships, and masquerade connections. It is part of the deprecated net-tools package — the modern replacement is ss which is faster and provides more information.

Despite deprecation, netstat questions remain extremely common in interviews, and the flag combinations — especially -tulnp — are muscle memory for experienced Linux engineers.

-nNumeric (no DNS resolution)Faster output, shows raw IPs and ports
-pShow process name/PIDKnow WHICH process owns a port
-aAll connectionsLISTEN + ESTABLISHED + others
-rRouting tableShow kernel routing table
-sStatistics per protocolTCP/UDP/ICMP stats
-cContinuous refreshLive monitoring
4Practical Examples
BASH — The Essential netstat Patterns
# THE MOST IMPORTANT NETSTAT COMMAND — memorize this!
# Show all listening TCP/UDP ports with process names
sudo netstat -tulnp
# Proto Recv-Q Send-Q Local Address   Foreign Address  State      PID/Program
# tcp        0      0 0.0.0.0:22      0.0.0.0:*        LISTEN     1234/sshd
# tcp        0      0 0.0.0.0:80      0.0.0.0:*        LISTEN     5678/nginx
# tcp        0      0 127.0.0.1:5432  0.0.0.0:*        LISTEN     9012/postgres

# 45  TIME_WAIT

# Show routing table
netstat -rn

# Show connections to a specific remote port (e.g., database)
netstat -tn | grep :5432
6Interview Questions & Answers
Q1What does the -tulnp flag combination mean?
A:tcp + udp + listen + numeric + process. This single combination shows all listening TCP and UDP ports with their numeric addresses (no slow DNS lookups) and the process name/PID that owns each port. It's the first command to run when: deploying a new service (verify it's listening), debugging "port already in use" errors, or auditing which services are exposed on a server.
Q2What does a large number of TIME_WAIT connections indicate?
A:TIME_WAIT is a TCP state where the connection has been closed but the OS keeps the socket for 2×MSL (Maximum Segment Lifetime, typically 60-120 seconds) to handle late-arriving packets. Many TIME_WAIT connections indicate high connection turnover — the server is closing many short-lived connections. In web servers, this is normal with HTTP/1.0 (no keep-alive). Mitigation: enable HTTP keep-alive, use connection pooling, or tune net.ipv4.tcp_tw_reuse=1 in sysctl.
7Common Mistakes & Pro Tips
⚠️
Always use sudo with -p flag Without sudo, netstat -p shows PID/program only for processes owned by the current user. With sudo, it shows all processes. In production debugging, always run sudo netstat -tulnp.
ss
Socket Statistics — Modern netstat Replacement
Investigate sockets — faster, richer output than netstat
1Command Overview

ss (Socket Statistics) is the modern replacement for netstat. It reads directly from the kernel's netlink interface, making it significantly faster than netstat (which reads from /proc/net/). It shows more detailed TCP socket information including TCP internal state, timers, and congestion control data.

In production environments, ss is the preferred tool for socket inspection because of its speed on servers with thousands of connections, richer filtering capabilities, and support for all modern socket types including Unix domain sockets and netlink sockets.

6Interview Questions & Answers
Q1What is the key difference between ss and netstat?
A:ss reads directly from the kernel's netlink socket, while netstat parses /proc/net/tcp. Speed difference is dramatic at scale — on a server with 100,000 connections, netstat can take 30+ seconds while ss returns in under a second. Additionally, ss -i provides internal TCP data (congestion window, RTT, retransmission count) that netstat can't access — invaluable for TCP performance tuning.
Q2Scenario: A web service is getting connection errors. How would you use ss to diagnose?
A:1. ss -s — check TIME_WAIT and CLOSE_WAIT counts. High TIME_WAIT = connection churn; high CLOSE_WAIT = application not closing connections properly. 2. ss -tlnp — verify the service is still listening on the expected port. 3. ss -tn state established | wc -l — count active connections vs expected max. 4. ss -ti to see retransmission counts indicating packet loss. 5. ss -tlnp | grep LISTEN | awk '{print $4}' | cut -d: -f2 to list all listening ports for comparison with baseline.
7Common Mistakes & Pro Tips
✅
Pro Tip — ss state filtering is powerful ss -t state time-wait | wc -l counts TIME_WAIT sockets. Valid states: established, syn-sent, syn-recv, fin-wait-1, fin-wait-2, time-wait, closed, close-wait, last-ack, listen, closing. This granularity is not available in netstat without piping to grep.
traceroute
Network Path Tracing
Trace the route packets take to a destination — identify where delays occur
1Command Overview

traceroute maps the network path from your machine to a destination by sending probes with incrementally increasing TTL (Time-To-Live) values. Each router along the path decrements the TTL; when it reaches 0, the router sends an ICMP "Time Exceeded" message back — revealing its IP and the latency to reach it.

In production, traceroute is used to identify: which network hop is introducing latency, where packet loss is occurring, whether traffic is taking the expected path (e.g., through the intended VPN or VPC peering), and asymmetric routing issues.

3Key Flags
FlagDescriptionExample
-nNumeric only (skip reverse DNS — much faster)traceroute -n 8.8.8.8
-TUse TCP SYN instead of UDP (bypasses many firewalls)traceroute -T -p 80 host
-IUse ICMP Echo instead of UDPtraceroute -I host
-p PORTDestination port to probetraceroute -T -p 443 host
-m HOPSMax number of hops (default: 30)traceroute -m 15 host
-q NNumber of probes per hop (default: 3)traceroute -q 1 host
-w SECSWait time for response (default: 5s)traceroute -w 2 host
-i IFACESend from specific interfacetraceroute -i eth1 host
4Practical Examples
BASH
# Basic traceroute (UDP default)

# Faster: numeric only, skip DNS lookups

# traceroute to 8.8.8.8 (8.8.8.8), 30 hops max, 60 byte packets
#  1  10.0.1.1    0.5 ms  0.4 ms  0.5 ms     ← default gateway
#  2  203.0.113.1 2.1 ms  2.0 ms  2.2 ms     ← ISP router
#  3  * * *                                   ← firewall blocking probes
#  4  8.8.8.8    12.4 ms 12.3 ms 12.5 ms     ← destination

# Continuous — shows real-time loss% at each hop

# Detect where traffic is being dropped
traceroute -n -q 1 database.internal
6Interview Questions & Answers
Q1What do asterisks (***) mean in traceroute output?
A:Asterisks mean no response was received for that probe within the timeout period. This usually means: (1) A firewall is silently dropping ICMP Time Exceeded messages, (2) The router at that hop is configured to not respond to traceroute probes. Crucially, asterisks at intermediate hops do NOT necessarily indicate a problem — traffic may still be flowing through. Only asterisks at the FINAL destination indicate a real connectivity issue. Use traceroute -T -p 80 (TCP) to bypass ICMP-blocking firewalls.
Q2What is mtr and when is it better than traceroute?
A:mtr (Matt's Traceroute) combines traceroute and ping into a continuous interactive tool. It updates live and shows packet loss percentage and RTT statistics per hop over time. It's better than traceroute when diagnosing intermittent packet loss (traceroute is a one-time snapshot; mtr shows patterns over minutes). Install: apt-get install mtr. Use: mtr --report -n --report-cycles 100 HOST for a 100-cycle automated report.
nslookup
DNS Query Tool
Query Internet name servers interactively — basic DNS lookup
1Command Overview

nslookup queries DNS servers to resolve hostnames to IPs and perform reverse lookups (IP to hostname). It is simpler than dig and available on all platforms (Linux, Windows, macOS), making it a universal DNS diagnostic tool. For advanced DNS debugging, dig is preferred.

4Practical Examples
BASH
# Forward lookup — hostname to IP
nslookup google.com
# Server:    8.8.8.8     ← DNS server used
# Address:   8.8.8.8#53
# Name:      google.com
# Address:   142.250.80.46

# Reverse lookup — IP to hostname
nslookup 8.8.8.8

# Query a specific DNS server (bypass system default)
nslookup google.com 1.1.1.1

# Query specific record type
nslookup -type=MX gmail.com     # Mail exchange records
nslookup -type=TXT github.com   # TXT records (SPF, DKIM)
nslookup -type=NS example.com   # Name servers
nslookup -type=CNAME www.example.com  # CNAME alias
6Interview Questions & Answers
Q1What is the difference between nslookup and dig?
A:nslookup is simpler, cross-platform (works on Windows too), and provides basic DNS resolution. dig (Domain Information Groper) is more powerful — it shows the full DNS response including TTL values, all records returned, query time, and the authoritative server. dig output is more detailed and suitable for advanced DNS debugging. In scripts, dig +short is preferred over nslookup because its output format is more stable and parseable. For cross-platform needs (including Windows troubleshooting), nslookup wins.
dig
Advanced DNS Lookup
Domain Information Groper — detailed DNS query and debugging tool
1Command Overview

dig (Domain Information Groper) is the professional DNS diagnostic tool that provides complete, detailed DNS responses including TTL values, all record types, query time, and which server responded. It is the preferred DNS tool for engineers because its output is consistent and scriptable.

In DevOps and cloud engineering, dig is used for: verifying DNS propagation after domain changes, debugging split-horizon DNS in VPCs, testing DNS-based load balancing, checking TTL values before making changes, and validating SPF/DKIM/DMARC records for email deliverability.

4Practical Examples
BASH — Production DNS Debugging
# Full DNS query — shows TTL, authoritative server, query time
dig google.com
# ;; QUESTION SECTION:
# ;google.com.    IN  A
# ;; ANSWER SECTION:
# google.com.  300  IN  A  142.250.80.46   ← 300s TTL
# ;; Query time: 12 msec
# ;; SERVER: 8.8.8.8#53(8.8.8.8)

# Clean output for scripting — just the IP(s)
dig +short google.com
# 142.250.80.46

# Query from a specific DNS server (test internal vs external DNS)
dig @10.0.0.2 internal-service.company.com   # VPC DNS
dig @8.8.8.8 internal-service.company.com     # External DNS

# Check TTL before making changes
dig +noall +answer api.example.com
# api.example.com. 3600 IN A 203.0.113.10   ← TTL = 1 hour

# Query MX records (email routing)
dig -t MX gmail.com +short
# 5 gmail-smtp-in.l.google.com.
# 10 alt1.gmail-smtp-in.l.google.com.

# Trace full DNS delegation chain (authoritative lookup)
dig +trace github.com

# Reverse DNS lookup
dig -x 8.8.8.8 +short
# dns.google.

# TXT records — SPF, DKIM, domain verification
dig -t TXT _dmarc.google.com +short
dig -t TXT github.com +short | grep v=spf1

# Script: wait for DNS to propagate
until [[ "$(dig +short api.example.com @8.8.8.8)" == "203.0.113.10" ]]; do
    echo "DNS not propagated yet, waiting..."
    sleep 30
done
echo "DNS propagated! Proceeding with deployment."
6Interview Questions & Answers
Q1Scenario: After a deployment, users report DNS issues with the new service endpoint. How do you debug?
A:1. dig +short api.example.com — check what public DNS resolves to. 2. dig @VPC_DNS_IP api.example.com — check internal VPC DNS separately. 3. dig +noall +answer api.example.com — check TTL, ensure old IP isn't cached. 4. dig +trace api.example.com — verify full delegation chain from root. 5. Compare results from multiple resolvers (8.8.8.8, 1.1.1.1, your ISP). Different answers = propagation in progress. Same wrong answer = DNS record is wrong.
7Common Mistakes & Pro Tips
✅
Pro Tip — Always check TTL before DNS changes Before changing a DNS record, run dig +noall +answer hostname to check the current TTL. If TTL is 3600 (1 hour), plan for up to 1 hour of propagation time. Lower TTL to 300 (5 min) a day before making changes, then change the record, then raise TTL again after propagation.

6Interview Questions & Answers
Q1How do you test if an API endpoint is returning the correct response in a CI pipeline?
A:Health check pattern in CI: curl -sS --fail http://service/health | jq '.status == "healthy"'. The --fail flag makes curl exit with code 1 on HTTP 4xx/5xx responses (instead of showing the error body and exiting 0). Combined with -sS (silent but show errors), this creates a clean CI-friendly check. For detailed timing: curl -w "%{http_code}" -o /dev/null -sS URL extracts just the HTTP status code.
Q2How would you debug an SSL/TLS issue with curl?
A:1. curl -v https://host — verbose mode shows full TLS handshake, certificate chain, and cipher. 2. If self-signed cert: curl --cacert /path/to/ca.pem https://host (add custom CA). 3. For mTLS (mutual TLS): curl --cert client.pem --key client-key.pem https://host. 4. curl -k https://host temporarily skips verification to confirm it's a cert issue (NEVER use -k in production scripts). 5. openssl s_client -connect host:443 for detailed TLS debugging.
Q3How do you measure API response time with curl?
A:curl -sS -o /dev/null -w "%{time_total}\n" https://api.example.com/endpoint. The -w (write-out) flag provides: time_namelookup (DNS), time_connect (TCP connect), time_appconnect (TLS handshake), time_pretransfer (before first byte sent), time_starttransfer (time to first byte = TTFB), time_total (complete. This breakdown immediately reveals whether latency is in DNS, TCP, TLS, or server processing.
7Common Mistakes & Pro Tips
❌
Mistake — Using -k in production scripts curl -k disables SSL verification — a security vulnerability. It's fine for local testing but NEVER acceptable in production CI/CD scripts. Use --cacert with the proper CA certificate instead.
✅
Pro Tip — Use --fail in CI scripts By default, curl exits with 0 even on HTTP 4xx/5xx responses. Add --fail to make non-2xx responses fail with exit code 22: curl --fail -sS https://api/endpoint || { echo "API check failed"; exit 1; }

6Interview Questions & Answers
Q1When would you use rsync instead of scp?
A:rsync is preferred over scp when: (1) transferring large directories where only changed files should be synced (rsync sends delta only), (2) resuming interrupted transfers (rsync --partial --progress), (3) syncing files repeatedly on a schedule (rsync is idempotent), (4) preserving all metadata including symlinks, hard links, and extended attributes (rsync -aAX). scp always copies everything; rsync is intelligent about what to transfer.

6Interview Questions & Answers
Q1Explain SSH local port forwarding and give a production use case.
A:ssh -L LOCAL_PORT:REMOTE_HOST:REMOTE_PORT JUMP_HOST — creates a tunnel where connections to localhost:LOCAL_PORT are forwarded through the jump host to REMOTE_HOST:REMOTE_PORT. Production example: ssh -N -L 5432:prod-db.internal:5432 bastion.company.com — allows a developer to connect their local PostgreSQL client to localhost:5432 and access the production database without exposing the database to the internet. The database server only accepts connections from the bastion's private IP — enforcing network security.
Q2What is the difference between local and remote port forwarding?
A:Local (-L): Forward a local port to a remote service — "bring remote service to me." ssh -L 5432:db:5432 bastion — I access prod database via localhost. Remote (-R): Forward a remote port to a local service — "expose my service to the remote server." ssh -R 8080:localhost:8080 server — the server can access my local service on port 8080. Use case: expose a local Jupyter notebook to a team member via a public server. Remote forwarding requires GatewayPorts yes in the server's sshd_config.
Q3How do you debug SSH connection failures?
A:Client side: ssh -vvv user@host — triple verbose shows every step of connection, key negotiation, and authentication. Look for "Permission denied" (wrong key/user), "Connection refused" (SSH not running or port wrong), "No route to host" (network issue). Server side: journalctl -u sshd -f or tail -f /var/log/auth.log — shows failed login attempts and reasons. Common fixes: check ~/.ssh/authorized_keys permissions (must be 600), check ~/.ssh/ permissions (must be 700), verify key format matches (ed25519 vs RSA).
7Common Mistakes & Pro Tips
❌
Mistake — StrictHostKeyChecking=no in scripts -o StrictHostKeyChecking=no disables host key verification — opens you to MITM attacks. In CI/CD: scan host keys first with ssh-keyscan -H host >> ~/.ssh/known_hosts and commit known_hosts to your deployment config. Never use StrictHostKeyChecking=no in production scripts.
✅
Pro Tip — Use ~/.ssh/config extensively A well-maintained SSH config eliminates the need to remember IPs, keys, usernames, and jump hosts. It also enables scp, rsync, git to transparently use your custom settings. This is a significant productivity multiplier in production environments.
ftp
File Transfer Protocol
Legacy FTP client — unencrypted file transfer ⚠ INSECURE
1Command Overview

ftp is the traditional File Transfer Protocol client for uploading and downloading files from FTP servers. Critical security note: FTP transmits data, usernames, and passwords in plain text with NO encryption. It should never be used on untrusted networks.

🚨
INSECURE — Avoid in Production FTP sends credentials and data unencrypted over the network — any eavesdropper on the network path can capture your username, password, and all transferred data. Use SFTP (SSH File Transfer Protocol) or FTPS (FTP over TLS) instead. FTP is only relevant when: connecting to legacy FTP servers you don't control, or working with certain IoT/embedded systems.

Understanding FTP remains relevant for interviews because many legacy systems (CDNs, ISPs, manufacturing systems, government portals) still expose FTP endpoints. Knowing the secure alternatives and when FTP is unavoidable demonstrates production maturity.

6Interview Questions & Answers
Q1What are the security issues with FTP and what should you use instead?
A:FTP security issues: (1) No encryption — credentials and data in plaintext, (2) Vulnerable to packet sniffing, MITM attacks, credential capture, (3) Active mode FTP opens client-side ports — problematic through NAT/firewalls. Secure alternatives: SFTP (SSH File Transfer Protocol) — completely different protocol over SSH, fully encrypted, best choice; FTPS (FTP over TLS) — FTP with TLS encryption added; rsync over SSH — for sync/backup use cases; scp — for simple file copies. In 2024, new deployments should never use plain FTP.
7Common Mistakes & Pro Tips
⚠️
Binary mode is critical! Transferring non-text files (images, zip, executables, database dumps) in ASCII mode causes silent file corruption. Always run binary inside the ftp prompt before downloading binary files. This is the #1 FTP mistake that corrupts deployment artifacts.
