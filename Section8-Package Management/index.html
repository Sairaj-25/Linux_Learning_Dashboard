apt-get
Debian / Ubuntu
APT package handling utility — install, remove, and update packages
🐧 Ubuntu 🐧 Debian
1Command Overview

apt-get is the traditional command-line package manager for Debian-based distributions. It interacts with the APT (Advanced Package Tool) library to install, upgrade, and remove packages, resolving dependencies automatically from configured repositories.

In DevOps and AI/ML roles, apt-get is used in Dockerfiles to install system dependencies (libpq-dev, build-essential, CUDA tools), in CI/CD pipelines to provision build agents, and in Ansible playbooks to ensure reproducible server environments. The -y flag and DEBIAN_FRONTEND=noninteractive pattern is ubiquitous in automated provisioning.

💡
apt-get vs apt apt is the newer, friendlier CLI (combines apt-get + apt-cache). apt-get is preferred in scripts because its output format is stable. apt may change output in future releases, breaking script parsers.
2Syntax
BASH
apt-get [OPTIONS] COMMAND [package...]

  OPTIONS  — Flags modifying behavior  (-y, -q, --no-install-recommends ...)
  COMMAND  — Subcommand: install | remove | update | upgrade | dist-upgrade ...
  package  — One or more package names  (e.g., nginx python3-pip)

4Practical Examples
Beginner — Daily Operations
BASH
# ALWAYS update index before installing
sudo apt-get update

# Install a package
sudo apt-get install nginx

# Install multiple packages at once
sudo apt-get install -y python3 python3-pip python3-venv

# Remove package (keep config files)
sudo apt-get remove nginx

# Remove package AND config files (clean slate)
sudo apt-get purge nginx

# Upgrade all packages
sudo apt-get update && sudo apt-get upgrade -y

# Clean up orphaned dependencies
sudo apt-get autoremove -y && sudo apt-get autoclean
Intermediate — DevOps & Automation
BASH — Dockerfile / CI-CD Pattern
# Dockerfile best practice — one RUN layer, clean cache at end
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
        build-essential \
        libpq-dev \
        python3-dev \
        curl \
    && apt-get clean \
    && rm -rf /var/lib/apt/lists/*
# Removing /var/lib/apt/lists/* shrinks the image layer

# Install specific version (reproducible builds)
apt-get install -y nginx=1.24.0-1~jammy

# Simulate upgrade — see what will change
apt-get -s dist-upgrade

# Fix broken packages (common after interrupted installs)
sudo apt-get --fix-broken install

# Hold a package at current version (prevent auto-upgrade)
sudo apt-mark hold kubernetes-cni
apt-get upgrade          # kubernetes-cni stays pinned
sudo apt-mark unhold kubernetes-cni
Advanced — ML / Data Infra Provisioning
BASH — Provisioning ML Server
# Install CUDA toolkit + ML dependencies (Ubuntu 22.04)
sudo apt-get update
sudo apt-get install -y --no-install-recommends \
    cuda-toolkit-12-2 \

echo "deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.gpg] \

    sudo tee /etc/apt/sources.list.d/docker.list
sudo apt-get update
sudo apt-get install -y docker-ce docker-ce-cli containerd.io

# Non-interactive for CI pipelines (suppress all prompts)
DEBIAN_FRONTEND=noninteractive apt-get install -y tzdata
5Real-World Use Cases
  • Dockerfile dependency installation: Every Python/Node/Go container image uses apt-get install --no-install-recommends to keep image layers minimal — a key Docker best practice.
  • CI/CD agent provisioning: GitHub Actions, GitLab CI, and Jenkins pipelines install tools with apt-get install -y in setup steps before running build/test stages.
  • Ansible playbooks: The ansible.builtin.apt module wraps apt-get for idempotent package management across fleets of Ubuntu servers.
  • Kubernetes node setup: kubeadm, kubelet, kubectl are installed via apt-get with version pinning and apt-mark hold to prevent accidental cluster upgrades.
  • Security patching: apt-get update && apt-get upgrade -y in a nightly cron job or via unattended-upgrades for automated security patch application.
6Interview Questions & Answers
Q1What is the difference between apt-get update and apt-get upgrade?
A:update only refreshes the local package index — it downloads the latest package lists from all configured repositories but does NOT install or upgrade anything. upgrade then uses that updated index to upgrade all installed packages. You must always run update before upgrade — otherwise, you're upgrading based on a stale index.
Q2What is the difference between remove and purge?
A:remove uninstalls the package binaries but leaves configuration files in place — useful when you plan to reinstall later and want to preserve your settings. purge removes both binaries AND config files (including /etc configs) — use for a clean slate. In production, purge is used when migrating to a different solution.
Q3What is the difference between upgrade and dist-upgrade?
A:upgrade only upgrades packages that don't require installing new packages or removing existing ones. dist-upgrade is smarter — it handles changing dependencies, installs new required packages, and removes obsolete ones. Use dist-upgrade for major kernel or library updates, and for OS distribution upgrades. In Docker, dist-upgrade is used for security-complete image builds.
Q4Scenario: Your Dockerfile build fails because apt-get install can't find a package. How do you diagnose?
A:The root cause is almost always a stale package index in the build cache. Fix: ensure apt-get update and apt-get install are in the same RUN layer — RUN apt-get update && apt-get install -y pkg. If they're in separate layers, Docker caches the update layer and uses a stale index for the install. Also check: package name changed, package requires a PPA, or the base image is too old.
Q5How do you install a specific version of a package?
A:apt-get install nginx=1.24.0-1~jammy — use apt-cache policy nginx to first see available versions. For Kubernetes components: apt-get install -y kubelet=1.28.0-00 kubeadm=1.28.0-00 kubectl=1.28.0-00 then apt-mark hold all three to prevent accidental upgrade.
Q6Why do we run apt-get clean && rm -rf /var/lib/apt/lists/* in Dockerfiles?
A:Each RUN instruction creates a new Docker layer. The APT package cache in /var/cache/apt/archives/ and index files in /var/lib/apt/lists/ can add 100–500MB to an image. Cleaning them in the SAME RUN instruction (same layer) prevents that data from being committed to the image. If you clean in a SEPARATE RUN layer, the data is already committed to the prior layer and the cleanup has no effect on final image size.
7Common Mistakes & Pro Tips
❌
Mistake — Running apt-get without update first Installing without refreshing the index may install outdated versions or fail if a package was renamed. Always chain: apt-get update && apt-get install -y pkg.
❌
Mistake — Separate RUN layers for update and install in Docker Docker's layer cache will reuse the cached update layer with a stale index. Always combine them in one RUN statement.
❌
Mistake — Not setting DEBIAN_FRONTEND=noninteractive Some packages (tzdata, postfix) prompt for user input. In CI/CD and Docker this hangs indefinitely. Always export DEBIAN_FRONTEND=noninteractive in automated environments.
✅
Pro Tip — Use --no-install-recommends in Docker Reduces image size significantly. Recommended packages are often GUI tools or optional extras irrelevant in containers. Typical savings: 20–40% smaller layer.
✅
Pro Tip — Pin versions for reproducibility apt-get install nginx=1.24.0-1~jammy ensures every build or server gets the exact same version. Critical for compliance environments and Kubernetes cluster consistency.

apt-cache
Debian / Ubuntu
Query the APT package cache — search, inspect, and analyse packages
🐧 Ubuntu 🐧 Debian
1Command Overview

apt-cache queries the local APT package cache (stored in /var/cache/apt/) to search for packages, display metadata, check dependencies, and compare available versions. It is a read-only tool — it never installs or modifies anything.

In engineering roles it's used to discover package names, check which version will be installed, inspect dependency chains before installing software, and verify that a pinned version is available in the configured repositories.

2Syntax
BASH
apt-cache COMMAND [OPTIONS] [package...]

  COMMAND  — search | show | policy | depends | rdepends | showpkg | stats
  package  — Package name or search term

policy PKGShow installed/candidate versions + repository prioritiesapt-cache policy python3

rdepends PKGShow reverse dependencies — what depends ON this packageapt-cache rdepends libc6

4Practical Examples
BASH
# Search for packages related to machine learning
apt-cache search "machine learning"
apt-cache search tensorflow | grep python3

# Check exactly what version will be installed + source repo
apt-cache policy nginx
# Output shows: Installed, Candidate, and Version table with priorities

# See all available versions of a package
apt-cache policy python3-tensorflow

# Inspect full package metadata before installing
apt-cache show cuda-toolkit-12-2 | grep -E "Version|Size|Depends"

# See what a package depends on (before installing)

# Find what would break if you remove a package
apt-cache rdepends --installed libssl3

# List all python3 packages available
apt-cache pkgnames python3- | sort | head -20
6Interview Questions & Answers
Q1What does apt-cache policy tell you and when is it useful?
A:apt-cache policy PKG shows the currently installed version, the candidate version that would be installed by apt-get install, and a version table listing every available version from each repository with its priority. This is essential when debugging why the wrong version is being installed, verifying pinning is working correctly, or confirming that a required version exists in the configured repos before writing automation scripts.
Q2What is the difference between apt-cache search and apt-cache show?
A:search queries package names and descriptions using a keyword — returns a list of matching packages. show requires an exact package name and returns full metadata: version, installed size, download size, maintainer, dependencies, and description. Use search to discover packages, then show to inspect the one you want to install.
7Common Mistakes & Pro Tips
⚠️
Mistake — apt-cache search with stale index apt-cache queries the LOCAL cache. If you haven't run apt-get update recently, results may not reflect the latest packages. Always update before searching for newly added packages.
✅
Pro Tip — Use policy before installing in CI In CI scripts, run apt-cache policy PKG to log exactly which version will be installed. This creates an audit trail and catches version drift between builds.
yum
Red Hat / CentOS 6 & 7
Package manager for RPM-based systems — Yellowdog Updater Modified
🔴 RHEL 7 🔴 CentOS 7 🔴 Amazon Linux
1Command Overview

yum (Yellowdog Updater Modified) is the package manager for RPM-based distributions — RHEL 7, CentOS 7, Amazon Linux 1, and Oracle Linux. It resolves dependencies automatically from configured repositories (yum repos) and handles the full package lifecycle.

Despite being superseded by dnf in newer releases, yum remains critical knowledge because many enterprise environments and AWS EC2 instances still run CentOS 7 or Amazon Linux 2. On CentOS 8+, yum is actually an alias to dnf.

⚠️
EOL Notice CentOS 7 reached End-of-Life June 2024. However, yum knowledge remains essential for Amazon Linux 2, legacy RHEL 7 systems, and reading existing Ansible/Chef/Puppet code targeting these platforms.

5Real-World Use Cases
  • Amazon Linux 2 / EC2 provisioning: AWS still widely uses Amazon Linux 2 (yum-based). EC2 user data scripts and Systems Manager Run Command use yum for instance setup.
  • Air-gapped environments: yum install --downloadonly + createrepo to build offline local repos for secure government or financial networks.
  • Rollback after bad update: yum history undo LAST — unique yum feature that reverses an entire transaction, invaluable in production incident response.
  • EPEL for ML tools: Many ML-adjacent tools (htop, iotop, nmap) are in EPEL, not base repos. yum install epel-release && yum install htop.
6Interview Questions & Answers
Q1What is the difference between yum update and yum upgrade?
A:In yum, update and upgrade behave similarly — both upgrade installed packages to their latest versions. The subtle difference: upgrade also removes obsolete packages (packages that are superseded by newer names). In practice on CentOS 7, they are often used interchangeably, but upgrade is slightly more aggressive in handling package obsoletes. This is different from the apt-get behavior where the distinction is more pronounced.
Q2What is EPEL and why is it important?
A:EPEL (Extra Packages for Enterprise Linux) is a Fedora Special Interest Group repository that provides high-quality additional packages for RHEL/CentOS that are not in the default repos. It's important because many tools needed in DevOps and data engineering (htop, jq, sysstat, redis, many Python libraries) are only available in EPEL. Install with: yum install epel-release then access packages with yum install --enablerepo=epel PKG.
Q3Scenario: You updated nginx on a production server and it broke an application. How do you rollback?
A:yum's transaction history makes this straightforward. 1. yum history list — find the transaction ID for the nginx update. 2. yum history info <ID> — confirm what was changed. 3. sudo yum history undo <ID> — reverses the entire transaction, reinstalling the previous version. This is a key differentiator of yum over apt — built-in transactional rollback.
7Common Mistakes & Pro Tips
❌
Mistake — Running yum update without testing On production RHEL systems, an unconstrained yum update can upgrade the kernel and critical libraries, requiring a reboot and potentially breaking application compatibility. Always test updates in staging first and consider yum-versionlock.
✅
Pro Tip — yum provides is your friend If you know you need a specific binary or library but don't know the package name, yum provides '*/binary_name' is invaluable. Example: yum provides '*/python3.11'.
ℹ️
On CentOS 8+ / RHEL 8+ yum is now a symlink to dnf. All yum commands are supported but dnf is the true engine. Migrate scripts to dnf for new deployments.
dnf
Fedora / RHEL 8+ / CentOS 8+
Next-generation package manager — Dandified YUM (replaces yum)
🔴 RHEL 8+ 🔴 CentOS 8+ 🔴 Fedora 🔴 Rocky/Alma
1Command Overview

dnf (Dandified YUM) is the next-generation package manager for Fedora, RHEL 8+, CentOS 8+, Rocky Linux, and AlmaLinux. It replaces yum with better performance, improved dependency resolution (using libsolv), proper Python 3 support, and a cleaner API.

For DevOps and MLOps engineers working in enterprise environments or building RHEL-based container images, dnf is the standard tool. Red Hat OpenShift, Red Hat Ansible Automation Platform, and modern RHEL-based Kubernetes nodes all rely on dnf.

🚀
dnf vs yum dnf is largely command-compatible with yum — most commands work identically. Key improvements: faster dependency resolution, better error messages, automatic handling of weak dependencies, and a proper module system (dnf module) for handling multiple versions of language runtimes.

6Interview Questions & Answers
Q1What are DNF modules and how are they useful for a Data Engineer?
A:DNF modules (Application Streams) allow multiple versions of a software stack to coexist in the same repository. For example, you can switch between Python 3.8, 3.9, and 3.11 on RHEL 8 without adding external repos. A Data Engineer can dnf module enable python311 to get Python 3.11 on RHEL 8 which ships Python 3.8 by default. This is critical for reproducible ML environments on enterprise RHEL systems.
Q2How does dnf handle dependency conflicts better than yum?
A:dnf uses libsolv — a SAT (satisfiability) solver for dependency resolution. This gives it mathematically optimal solutions to dependency conflicts and significantly faster resolution on complex dependency graphs. yum used its own resolver which could fail in complex situations or produce non-optimal solutions. dnf also has the --allowerasing flag to allow replacing conflicting packages, and --nobest to relax the "best version" requirement.
7Common Mistakes & Pro Tips
❌
Mistake — Forgetting --nodocs and weak deps in containers For Red Hat UBI-based containers, add --nodocs --setopt=install_weak_deps=False to minimize image size. Also clean the dnf cache in the same layer: dnf clean all && rm -rf /var/cache/dnf.
✅
Pro Tip — Use dnf check-update in monitoring dnf check-update --security exits with code 100 if security updates are available. Use this in cron jobs or monitoring scripts to trigger alerts when servers need patching.
rpm
Red Hat / All RPM-based distros
RPM Package Manager — low-level package database and .rpm file operations
🔴 RHEL 🔴 CentOS 🔴 Fedora 🔴 Rocky 🔴 Amazon Linux
1Command Overview

rpm (RPM Package Manager) is the low-level tool for working with RPM packages. Unlike yum or dnf, it does NOT resolve dependencies automatically — you must have all required packages available. It operates directly on the RPM database and .rpm files.

In production, rpm is used for: querying the installed package database (what's installed, what version, what files), verifying package integrity after installation, extracting files from .rpm packages without installing, and importing GPG keys for repository security.

⚠️
Key Distinction rpm = low-level (no dep resolution). yum/dnf = high-level (auto dep resolution). In interviews, knowing when to use each is critical.

4Practical Examples
BASH
# Find which package owns a file (critical for debugging)

# List all files installed by a package

# Output: S.5....T.  /usr/sbin/nginx  ← size/checksum/time changed!
# Silence = no changes (package is intact)

# Verify ALL packages (find tampered files — security scan)
rpm -Va 2>/dev/null | grep '^..5'   # MD5 checksum changes

# Import GPG key (required before installing from a new repo)
sudo rpm --import https://packages.example.com/GPG-KEY.pub

# Extract files from RPM without installing (using rpm2cpio)
rpm2cpio package.rpm | cpio -idmv

# Custom query format — useful for scripting
rpm -qa --queryformat "%{NAME}\t%{VERSION}\t%{ARCH}\n" | column -t
6Interview Questions & Answers
Q1What is the difference between rpm and yum/dnf?
A:rpm is the LOW-LEVEL tool — it installs/removes/queries individual .rpm files but does NOT resolve or download dependencies automatically. If a package requires glibc-2.34 and it's not present, rpm fails. yum/dnf are HIGH-LEVEL tools that build on top of rpm — they resolve dependencies, download packages from repositories, and manage the installation order automatically. Think of rpm as a wrench and dnf as a full workshop.
Q2How do you find which package a file belongs to?
A:rpm -qf /path/to/file — the -qf (query file) flag queries the RPM database to find which installed package owns a given file. Example: rpm -qf /usr/bin/git returns git-2.40.1-1.el9. This is crucial for debugging — if a binary is corrupted or missing, this tells you which package to reinstall.
Q3Scenario: You suspect a binary was tampered with on a production server. How do you verify?
A:1. rpm -V nginx — verify the specific package. Output like S.5....T. means Size, MD5 checksum, and mTime changed — the binary was modified. 2. rpm -Va 2>/dev/null | grep "^..5" — scan all packages for MD5 checksum changes (potential tampering). 3. Compare with a known-good checksum from the vendor. This is a standard security incident response procedure on RHEL systems.
7Common Mistakes & Pro Tips
❌
Mistake — Using --nodeps carelessly rpm -ivh --nodeps pkg.rpm ignores dependency errors and can leave the system in a broken state where the installed package fails at runtime. Only use when you know all deps are already present from another source.
❌
Mistake — Using rpm -i instead of -U -i (install) fails if the package is already installed. -U (upgrade) installs OR upgrades — much safer as a default. Always prefer -Uvh over -ivh for individual RPM installation.
✅
Pro Tip — rpm -Va for security audits Regular rpm -Va scans catch file system tampering — a key component of RHEL security hardening and CIS Benchmark compliance. Pipe to grep "^..5" to filter only MD5 mismatches.

dpkg

Debian Package Manager — low-level .deb package operations
🐧 Ubuntu 🐧 Debian 🐧 Raspbian
1Command Overview

dpkg is the low-level Debian package manager — the equivalent of rpm for Debian-based systems. It installs, removes, and queries .deb packages directly without resolving dependencies from repositories. apt-get is a high-level wrapper built on top of dpkg.

In DevOps and AI/ML engineering, dpkg is used for: querying installed packages and their files, installing local .deb packages (like Google Chrome, VS Code, or custom ML tools), reconfiguring packages, and diagnosing broken package states.

💡
dpkg vs apt-get dpkg = low-level (no dep resolution, works with local .deb files). apt-get = high-level (dep resolution, downloads from repos). The relationship mirrors rpm vs yum/dnf.

  Main modes:  -i (install)  -r (remove)  -P (purge)  -l (list)  -s (status)
               -L (list files)  -S (search file)  --configure  --audit

4Practical Examples

sudo dpkg --configure -a

5Real-World Use Cases
  • Installing vendor .deb packages: Google Chrome, VS Code, Slack, Zoom, and many ML vendor tools (NVIDIA drivers, CUDA toolkit) are distributed as .deb files — installed with dpkg -i.
  • CI/CD pipeline diagnostics: dpkg -l | grep python quickly shows which Python versions are installed on a build agent — useful for debugging version mismatches.
  • Server migration: dpkg --get-selections backs up the full package list; dpkg --set-selections + apt-get dselect-upgrade restores it on a new server.
  • Broken package recovery: sudo dpkg --configure -a && sudo apt-get install -f is the standard fix for interrupted package installations — a common CI/CD failure mode.
6Interview Questions & Answers
Q1What is the difference between dpkg -r and dpkg -P?
A:-r (remove) uninstalls the package binaries and data files but leaves the configuration files (in /etc/) intact — useful if you plan to reinstall and want to preserve settings. -P (purge) removes everything including configuration files — used for a completely clean removal. You can check packages with orphaned configs using dpkg -l | grep "^rc" and clean them with apt-get purge.
Q2Scenario: You installed a .deb package with dpkg and got dependency errors. How do you fix it?
A:dpkg -i pkg.deb may fail with "dependency problems" because dpkg doesn't auto-download deps. The fix is: sudo apt-get install -f (or sudo apt-get --fix-broken install). This tells APT to examine the broken state and automatically install all missing dependencies. Then the originally attempted dpkg install can proceed. This pattern is standard for all vendor .deb installations.
Q3What does dpkg -l status code rc mean?
A:rc means the package was Removed but Config files remain. The first letter is the desired state (r = removed) and the second is the current status (c = config files present). These are ghost entries that can accumulate over time. Clean them with: dpkg -l | grep "^rc" | awk '{print $2}' | sudo xargs dpkg -P — this purges all orphaned config files.
7Common Mistakes & Pro Tips
❌
Mistake — Not following up dpkg -i with apt-get -f When dpkg fails due to missing dependencies, it leaves the package in a "half-installed" state. Always run sudo apt-get install -f immediately after any dpkg dependency failure to clean up the broken state.
✅
Pro Tip — dpkg -l status codes Know the two-character status codes: ii = fully installed, rc = removed with configs, iU = unpacked not configured, iF = half-configured (broken). Filter with dpkg -l | grep "^rc" to find cleanup candidates.
✅
Pro Tip — Inspect before installing Always run dpkg -I package.deb and dpkg -c package.deb before installing third-party .deb files to audit dependencies and see exactly what files will be placed on your system.
