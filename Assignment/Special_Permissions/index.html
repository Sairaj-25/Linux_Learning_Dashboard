SUID, SGID & Sticky Bit

PART I: STEP-BY-STEP PROCEDURE - SPECIAL PERMISSIONS LIFECYCLE

STEP 1: Identify the Security Requirement

Determine if a user needs temporary elevated privileges (SUID), collaborative group write access (SGID), or delete protection in shared folders (Sticky Bit).

Rationale: Standard rwx permissions are insufficient for collaborative or privileged execution scenarios. Special permissions solve specific security gaps without handing out sudo/root passwords.

STEP 2: Apply the Permission

Use chmod with either octal (4xxx, 2xxx, 1xxx) or symbolic (u+s, g+s, o+t) notation.

Rationale: Octal is absolute and overwrites; symbolic is relative and adds to existing permissions. Choose based on whether you are creating from scratch or modifying existing files.

STEP 3: Verification

Run ls -l and inspect the execute column for 's', 'S', 't', or 'T'.

Rationale: A capital 'S' or 'T' indicates the special bit is set, but the underlying execute bit is missing. This usually means the permission won't function as intended.

STEP 4: Security Auditing

Regularly search for rogue SUID/SGID root files across the system.

Rationale: Attackers often leave SUID root copies of /bin/bash as backdoors. Regular auditing finds these vulnerabilities.

PART II: COMMANDS AND EXPLANATIONS

chmod u+s / chmod 4xxx (SUID)

Syntax: chmod u+s /path/to/file or chmod 4755 /path/to/file

Purpose: Set User ID. When executed, the process runs with the privileges of the file's owner, not the user running it.

chmod u+s /usr/bin/passwd

chmod g+s / chmod 2xxx (SGID)

Syntax: chmod g+s /path/to/dir or chmod 2755 /path/to/dir

Purpose: Set Group ID. On files: executes with group owner's privileges. On directories: new files inherit the directory's group owner.

chmod g+s /shared/project

chmod o+t / chmod 1xxx (Sticky Bit)

Syntax: chmod o+t /path/to/dir or chmod 1777 /path/to/dir

Purpose: Sticky Bit. Restricts file deletion within a directory so only the file owner, directory owner, or root can delete it.

chmod o+t /tmp

Auditing Commands

# Find all SUID root files
find / -perm -4000 -user root 2>/dev/null

# Find all SGID files
find / -perm /2000 2>/dev/null

PART III: ALTERNATIVE COMMANDS & METHODS

1. Symbolic vs. Octal Notation
  • Symbolic (u+s): Relative. Adds SUID without altering existing rwx permissions. Safer for modifications.
  • Octal (4755): Absolute. Sets SUID and explicitly sets rwx to 755. Overwrites any existing ACLs or special modes. Good for scripting exact states.
2. Capital S vs Lowercase s

- s: SUID/SGID is set, AND the execute bit is set. The permission is functional.

- S: SUID/SGID is set, BUT the execute bit is missing. The permission is generally non-functional because the file cannot be executed anyway.

Same applies to t vs T for the Sticky Bit.

PART IV: ASSIGNMENT SOLUTIONS

Section A: Short Answer Questions

1. What are special permissions in Linux?

Special permissions are advanced access rights that override standard ugo/rwx rules to handle specific security scenarios like privilege escalation (SUID), collaborative group inheritance (SGID), and delete protection (Sticky Bit).

2. Name the three special permissions.

SUID (Set User ID), SGID (Set Group ID), Sticky Bit.

3. What is SUID?

Set User ID. It allows a user to execute a file with the permissions of the file's owner, rather than their own permissions.

4. What is SGID?

Set Group ID. On a file, it executes with the group owner's privileges. On a directory, it ensures new files inherit the directory's group owner, not the creator's primary group.

5. What is Sticky Bit?

A permission set on directories that prevents users from deleting or renaming files unless they are the file owner, directory owner, or root.

6. Octal values for SUID, SGID, Sticky Bit.

SUID = 4000, SGID = 2000, Sticky Bit = 1000.

7. Symbol representing Sticky Bit?

t (if execute is present) or T (if execute is absent) in the "other" execute column.

Section B: Conceptual Questions

1. How SUID works with an example.

The /usr/bin/passwd command is owned by root. A regular user needs to edit /etc/shadow to change their password, which they cannot normally read/write. Because passwd has SUID set (-rwsr-xr-x), when a user runs it, the process runs with root's UID, temporarily granting it write access to /etc/shadow specifically through the passwd program.

2. Difference between SGID on files and directories.

Files: The process runs with the privileges of the file's group owner.
Directories: When a user creates a file inside an SGID directory, the new file's group owner is forced to match the directory's group owner (not the user's primary group). This enables collaboration.

3. Why is Sticky Bit used in shared directories?

In a shared directory (like /tmp) with 777 permissions, User A could delete User B's file. Sticky Bit stops this. It restricts deletion rights so users can only delete files they own, preventing accidental or malicious data loss in collaborative spaces.

4. How special permissions improve system security.

They provide granular delegation. Instead of giving a user full sudo access to become root, SUID gives them root access only for a specific, vetted binary (like passwd). SGID prevents "group drift" in shared folders. Sticky Bit prevents denial-of-service via file deletion in shared spaces.

5. Compare normal and special permissions.

Section C: Practical Questions

1. Create a file and apply SUID.
touch /opt/myapp
chmod u+s /opt/myapp
# OR
chmod 4755 /opt/myapp
2. Create a directory and apply SGID.
mkdir /shared/project
chmod g+s /shared/project
# OR
chmod 2755 /shared/project
3. Apply Sticky Bit to a directory.
chmod o+t /shared/uploads
# OR
chmod 1777 /shared/uploads
4. Use chmod to set SUID/SGID/Sticky.
chmod 4755 file  # SUID
chmod 2755 dir   # SGID
chmod 1777 dir   # Sticky
5. Verify using ls -l.
ls -l /opt/myapp
# Output: -rwsr-xr-x 1 root root ... /opt/myapp
6. Create shared dir and test Sticky Bit.
mkdir /tmp/testdir
chmod 1777 /tmp/testdir
# As user1: touch /tmp/testdir/user1_file
# As user2: rm /tmp/testdir/user1_file
# Output: rm: cannot remove 'user1_file': Operation not permitted

Section D: Problem Solving

1. Identify special permission in -rwsr-xr-x.

SUID. The 's' in the owner execute position indicates SUID is set.

2. Identify special permission in drwxr-sr-x.

SGID. The 's' in the group execute position indicates SGID is set.

3. Identify special permission in drwxrwxrwt.

Sticky Bit. The 't' in the other execute position indicates Sticky Bit is set.

4. Convert 4755 and 2755 to symbolic form.

4755 → -rwsr-xr-x
2755 → -rwxr-sr-x

5. What does chmod 1777 /shared mean?

It sets the /shared directory permissions to drwxrwxrwt. Everyone has full read/write/execute access, but the Sticky Bit ('1' prefix and 't') means users can only delete files they own.

Section E: Viva Questions

1. Real use of SUID?

The passwd command. A user must modify /etc/shadow to change their password, but they have no write permission. The SUID bit on /usr/bin/passwd lets the command run as root momentarily to perform the edit.

2. Why is /tmp given Sticky Bit?

Because /tmp is world-writable (1777). Without the Sticky Bit, User A could delete User B's temporary files, causing application crashes or denial of service.

3. Can SGID affect newly created files? How?

Yes, if SGID is on the directory. Any new file created inside inherits the directory's group, not the creator's primary group.

4. What happens if SUID is applied to a directory?

In modern Linux kernels, nothing. SUID on directories is generally ignored by the Linux kernel (unlike some older Unix variants).

5. Difference between s and t?

s represents SUID (owner column) or SGID (group column). It runs the file with owner/group privileges.
t represents Sticky Bit (other column). It restricts deletion in directories.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What is the security risk of SUID shell scripts?

SUID shell scripts are extremely dangerous and most modern systems ignore the SUID bit on scripts. Attackers can manipulate environment variables (like PATH or IFS), use symbolic links, or send signals (like SIGINT) to trick the script into executing arbitrary commands as root. Compiled binaries (like C programs) are much harder to manipulate in this way.

Q2: What is the difference between find / -perm -4000 and find / -perm /4000?

- -4000 (Exact/Minimum match): Finds files where SUID is set. It doesn't care about the other permission bits; they can be anything.
- /4000 (Any bit match): Finds files where at least the SUID bit is set. If you used /7000, it would find files with SUID or SGID or Sticky bit.

Q3: How would you find all world-writable files that also have SUID set?
find / -perm -4000 -perm -o+w 2>/dev/null

This combination is a massive security risk, as it means any user can overwrite a binary that executes as root.

Q4: If a file has permissions -rwSrw-r--, what does the capital 'S' mean and is it a security issue?

The capital 'S' means SUID is set, but the owner execute bit is missing (rw- instead of rwx). Because the file is not executable, SUID has no effect. It's usually a configuration error, not a direct security issue, but it should be cleaned up to avoid confusion.

Q5: How does SGID interact with default ACLs?

SGID on a directory forces the group owner. If a Default ACL is also set on the directory, the Default ACL determines the permissions of new files, but SGID still determines the group ownership. The group set by SGID will be the group owner, and the permissions will come from the ACL mask and default ACL entries.
