PART I: STEP-BY-STEP PROCEDURE - USER LIFECYCLE

STEP 1: PLANNING & POLICY CONFIGURATION

Before creating users, define the environment by editing /etc/login.defs to set default password aging, UIDs, and umask policies.

Rationale: Enforcing security policies at the base level ensures that every subsequently created user adheres to organizational compliance (e.g., password rotation, minimum length).

STEP 2: GROUP CREATION

Create the primary/secondary groups required for the user's role using groupadd.

Rationale: Creating groups first allows you to assign users to them immediately upon creation, ensuring correct default ACLs and file ownership from the start.

STEP 3: USER CREATION

Create the user using useradd with appropriate flags for UID, shell, home directory, and primary group.

Rationale: The useradd command updates system files (/etc/passwd, /etc/shadow, /etc/group) and creates the home directory skeleton.

STEP 4: AUTHENTICATION SETUP

Set the initial password using passwd and configure specific aging rules using chage.

Rationale: A user account is locked until a password is set. The chage command forces the user to change their password on first login, ensuring only they know it.

STEP 5: MODIFICATION & GROUP ASSIGNMENT

Add the user to supplementary groups using usermod -aG or gpasswd -a.

Rationale: Users often need access to shared resources (e.g., docker, sudo, wheel, project-specific groups) beyond their primary group.

STEP 6: OFFBOARDING (LOCK/DELETE)

When a user leaves, lock the account immediately with usermod -L, archive data, then eventually delete with userdel -r.

Rationale: Locking prevents immediate access while you back up data. Deleting with -r removes the home directory, preventing orphaned data from consuming disk space or posing a security risk.

PART II: COMMANDS AND EXPLANATIONS

useradd

Syntax: useradd [options] LOGIN

Purpose: Creates a new user account. Updates system files and creates the home directory.

Example: useradd -m -s /bin/bash -g developers student1

passwd

Syntax: passwd [options] [LOGIN]

Purpose: Sets or changes the password for a user account. Also used to lock/unlock accounts (-l, -u) and force expiration (-e).

Example: passwd student1

usermod

Syntax: usermod [options] LOGIN

Purpose: Modifies an existing user account (UID, GID, shell, home directory, groups, lock status).

Example: usermod -aG sudo student1

userdel

Syntax: userdel [options] LOGIN

Purpose: Deletes a user account. The -r flag removes the home directory and mail spool.

Example: userdel -r student1

groupadd

Syntax: groupadd [options] GROUP

Purpose: Creates a new group. Updates /etc/group and /etc/gshadow.

Example: groupadd developers

gpasswd

Syntax: gpasswd [options] GROUP

Purpose: Administers /etc/group and /etc/gshadow. Used to add/remove users from groups and set group administrators.

Example: gpasswd -a student1 developers (Add), gpasswd -d student1 developers (Remove)

chage

Syntax: chage [options] LOGIN

Purpose: Changes user password aging information. Controls when a password expires and how long it is valid.

Example: chage -M 90 -W 7 student1 (Max days 90, Warn 7 days before)

PART III: ALTERNATIVE COMMANDS & METHODS

1. User Creation Alternatives
  • useradd : The standard, low-level binary. Requires explicit flags for home directories (-m) and shells. Consistent across all distros.
  • adduser : A high-level Perl script (on Debian/Ubuntu) that interactively prompts for password, shell, and user info. Automatically creates home dir. (Note: On RHEL/CentOS, adduser is just a symlink to useradd).
  • Manual Edit: Using vipw to edit /etc/passwd and /etc/shadow. Dangerous Not recommended for beginners, but useful for emergency recovery.
2. Group Management Alternatives
  • gpasswd -a user group : Traditional way to add a single user to a group.
  • usermod -aG group user : Modern, commonly used method. The -a (append) flag is critical; without it, the user is removed from all other supplementary groups not listed.
  • groupmems : Administers the supplementary members of a user's primary group. Less commonly used but very specific.
3. Locking/Unlocking Account Alternatives
  • usermod -L user / usermod -U user : Standard method. Prepends a '!' to the password hash in /etc/shadow.
  • passwd -l user / passwd -u user : Also locks the account by changing the hash.
  • chage -E 0 user : Expires the account immediately. Different from locking; the user cannot login at all, even with SSH keys.

Security Note: usermod -L only locks password authentication. The user might still login via SSH keys. To fully lock, use chage -E 0 or change the shell to /sbin/nologin.

PART IV: ASSIGNMENT SOLUTIONS

Section A – Short Answer Questions

1. What is a user account in an operating system? Why is it important?

A user account is an identity created for a person or process to interact with the operating system. Importance: It provides security isolation (preventing User A from deleting User B's files), resource control (disk quotas), accountability (logging who did what), and personalized environments (shell, dotfiles).

2. List the three types of users in Linux.

1. Root/Superuser: UID 0. Has absolute system control.
2. System/Users/Service Accounts: UIDs 1-999 (typically). Used by daemons (e.g., apache, mysql) to run with limited privileges.
3. Regular/Normal Users: UIDs 1000+ (typically). Actual human users logging in to work.

3. What is the difference between root user and normal user?

Root (UID 0): Has unrestricted access to all files, processes, and kernel interfaces. Can destroy the system with a single typo.
Normal User (UID 1000+): Has restricted access. Can only modify files they own and can only run commands permitted by their permissions/groups. Must use sudo for admin tasks.

4. What is a system user? Give one example.

A system user is a non-login account created to run specific services or daemons. This ensures that if a service (like a web server) is compromised, the attacker only gets the privileges of that system user, not root. Example: www-data (Apache/Nginx) or mysql.

5. What is a group in Linux?

A group is a collection of users. It is used to simplify permission management. Instead of assigning read/write access to a file for 10 individual users, you assign it to a group, and add those 10 users to the group.

12. What is the purpose of /etc/passwd file?

/etc/passwd is a plain text file that stores essential user account information. It must be readable by all users. It contains 7 colon-separated fields:
username:x:UID:GID:GECOS:Home_Dir:Shell
The 'x' indicates the password is stored in /etc/shadow.

13. What is /etc/shadow file and why is it important?

/etc/shadow stores the actual encrypted password hashes and password aging information. Importance: In older Unix systems, hashes were in /etc/passwd (which is world-readable), making them vulnerable to brute-force attacks. Moving hashes to /etc/shadow (readable only by root) drastically improved system security.

14. Explain password aging in Linux.

Password aging is a security mechanism that forces users to change their passwords periodically. It defines:
1. Max days: How long a password is valid.
2. Min days: Minimum days before a user can change their password again.
3. Warn days: Days before expiration the user is warned.
4. Inactive days: Days after expiration before the account is locked.

15. What is the use of chage command?

The chage (change age) command is used to view and modify password aging information for a user. Example: chage -l student1 lists current aging policies; chage -M 90 student1 sets the password to expire every 90 days.

Section D – Group Management

16. Write the command to: Add user to a group / Remove user from a group
# Add user to a supplementary group
usermod -aG groupname username
# OR
gpasswd -a username groupname

# Remove user from a supplementary group
gpasswd -d username groupname
17. What is the use of gpasswd command?

gpasswd is the administration tool for /etc/group and /etc/gshadow. It is used to:
- Add/remove users from groups (-a, -d).
- Set group administrators (-A).
- Set a group password (allows non-members to temporarily join the group using newgrp).

18. Explain /etc/group file.

/etc/group stores group membership information. It contains 4 colon-separated fields:
group_name:password:GID:user_list
- group_name: The name of the group.
- password: Usually 'x' (points to /etc/gshadow).
- GID: Group ID number.
- user_list: Comma-separated list of supplementary members.

19. What is /etc/gshadow file?

/etc/gshadow is the secure counterpart to /etc/group. It stores encrypted group passwords (rarely used) and group administrators. It is readable only by root, keeping group administrative privileges secure.

Section E – Configuration & Real-Time

20. What is /etc/login.defs file?

/etc/login.defs is the configuration file that defines the default site-specific configuration for the shadow password suite. It sets default policies for user creation and password constraints (applies only when creating users with useradd).

21. Write any four parameters from /etc/login.defs.

1. PASS_MAX_DAYS : Maximum number of days a password is valid.
2. PASS_MIN_DAYS : Minimum days between password changes.
3. PASS_WARN_AGE : Number of days warning before password expires.
4. UID_MIN / UID_MAX : Range of auto-assigned UIDs for regular users.
5. UMASK : Default file permission mask.

22. How does Linux enforce password security policies?

Linux enforces policies through a combination of:
1. /etc/login.defs: Sets aging defaults for new users.
2. PAM (Pluggable Authentication Modules): Specifically pam_pwquality.so (or pam_cracklib.so) enforces password complexity (length, uppercase, digits, symbols) at the moment of change.
3. chage: Enforces per-user aging and expiration limits.

23. Explain real-time use of user and group management in companies.

In enterprise environments:
1. Role-Based Access Control (RBAC): Groups map to roles (e.g., 'dev', 'qa', 'db-admin'). Users are added to groups, inheriting permissions to servers/databases.
2. Onboarding/Offboarding: Scripts automate user creation (with standard shells, groups, quotas). Offboarding immediately locks accounts via usermod -L to prevent data theft.
3. Auditing: Centralized logging (LDAP/AD) tracks who has sudo access; local groups like 'wheel' are strictly controlled.
4. Shared Directories: Project groups (e.g., 'project-x') share a directory with setgid (SGID) bits, ensuring all new files are readable/writable by the team.

Section G – Long Answer

29. Explain in detail the process of user creation and management in Linux.

When useradd <username> is executed, Linux performs several sequential actions:
1. UID Assignment: Reads /etc/login.defs to find the next available UID between UID_MIN and UID_MAX.
2. Group Creation: Creates a primary group with the same name as the user (unless -g is specified) and adds it to /etc/group.
3. File Updates: Adds an entry to /etc/passwd and a locked entry ('!') in /etc/shadow.
4. Home Directory: If -m is used, creates /home/username and copies skeleton files from /etc/skel (like .bashrc, .profile).
5. Activation: The account remains locked until passwd username is run to set the initial hash.
Management involves using usermod to adjust attributes, chage to handle lifecycle/expiration, and userdel -r to safely remove the user and their footprint when they depart.

30. Describe group management and its importance in system administration.

Group management is the administrative task of organizing users into logical collections to streamline access control. Every user has exactly one primary group (listed in /etc/passwd) and can belong to multiple supplementary groups (listed in /etc/group). Importance: 1) Efficiency: Assigning file permissions to a group (e.g., 'dev-team') is vastly easier than assigning them to 50 individual users. 2) Sudo Delegation: Adding a user to the 'wheel' or 'sudo' group grants administrative rights, avoiding the need to share the root password. 3) SGID Directories: Setting the SGID bit on a directory (chmod g+s) ensures that any new file created inside inherits the group ownership of the directory, enabling seamless collaboration. Tools like groupadd, gpasswd, and usermod -aG are essential for maintaining this structure securely.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What is the difference between a Primary Group and a Supplementary Group?

Primary Group: Every user must belong to exactly one primary group. It is specified in the GID field of /etc/passwd. When a user creates a new file, the file's group ownership is automatically assigned to the user's primary group (unless SGID is set).
Supplementary Group: Additional groups a user belongs to, listed in /etc/group. Used to grant extra permissions (e.g., accessing the docker socket, sudo rights, shared project folders). A user can belong to zero or many supplementary groups.

Q2: What does the 'x' in the password field of /etc/passwd indicate?

The 'x' indicates that the user's actual encrypted password is stored in the /etc/shadow file. This is a security measure; in older Unix systems, the encrypted hash was stored directly in /etc/passwd, which is world-readable, making it vulnerable to offline brute-force attacks.

Q3: How do you securely force a user to change their password on their next login?

Use the command chage -d 0 username. This sets the date of the last password change to "0", which means the password has expired. Upon the next login attempt via SSH or console, the user will be prompted to enter a new password before gaining access.

Q4: If you run `usermod -G developers student1` without the `-a` flag, what happens?

This is a common and dangerous mistake. Without the -a (append) flag, the -G flag overwrites all existing supplementary groups for the user. If student1 was previously in the 'sudo' and 'docker' groups, running usermod -G developers student1 will remove them from 'sudo' and 'docker', leaving them only in 'developers'. Always use usermod -aG to add a user to a group.

Q5: What is the purpose of /etc/skel? How is it used in user management?

/etc/skel (skeleton directory) contains default configuration files (like .bashrc, .bash_profile, .vimrc) that are automatically copied into a new user's home directory when created with the useradd -m command. It allows administrators to ensure all new users start with a standardized shell environment and aliases.

Q6: How does `passwd -l` differ from changing the user's shell to /sbin/nologin?

passwd -l (or usermod -L) locks the account by prepending a '!' to the password hash in /etc/shadow. This prevents password-based logins, but does not prevent the user from logging in via SSH keys if they are configured.
Changing the shell to /sbin/nologin (or /bin/false) in /etc/passwd completely blocks the user from obtaining an interactive shell session, regardless of the authentication method (password, SSH key, etc.). It is a more thorough way to disable an account.

Q7: What is PAM and how does it relate to user management?

PAM (Pluggable Authentication Modules) provides a flexible framework for authentication in Linux. Instead of applications hard-coding authentication logic, they call PAM. Administrators configure PAM in /etc/pam.d/ to enforce complex password rules (using pam_pwquality.so), limit root logins, configure LDAP/AD authentication, and set time-based access restrictions. It is the engine behind the scenes that enforces the rules set by commands like passwd.

Q8: What is the numeric UID of the root user? Can you change it?

The root user always has a UID of 0. Technically, you can change the UID of another user to 0 in /etc/passwd, which would grant that user root-level privileges. Dangerous This is highly discouraged. It breaks auditing, sudo rules, and system scripts that explicitly check for the 'root' name versus UID 0. There should only be one UID 0 account.
