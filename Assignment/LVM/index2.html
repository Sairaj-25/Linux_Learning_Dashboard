PART 1-4: COMPLETE LVM CREATION FLOWS FROM SCRATCH

The LVM creation process is identical after the partitioning stage. The key difference lies in Partitioning Tool (fdisk vs gdisk) and Filesystem Tool (mkfs.ext4 vs mkfs.xfs).

1) MBR + ext4 Flow MBR ext4

Step 1: Partitioning (MBR)
fdisk /dev/sdb
# Press 'n' (new), 'p' (primary), Enter (default part num), Enter (default start), +10G (size)
# Press 't' (type), '8e' (Linux LVM)
# Press 'w' (write)

Rationale: MBR requires type '8e' for LVM. Without it, pvcreate will warn that the partition type doesn't match.

Step 2: Create PV, VG, LV
partprobe /dev/sdb                  # Inform kernel of partition changes
pvcreate /dev/sdb1                  # Initialize Physical Volume
vgcreate vg_mbr_ext4 /dev/sdb1      # Create Volume Group
lvcreate -L 9G -n lv_data vg_mbr_ext4 # Create Logical Volume
Step 3: Filesystem (ext4) & Mount
mkfs.ext4 /dev/vg_mbr_ext4/lv_data # Create ext4 filesystem
mkdir /mnt/ext4_data                # Create mount point
mount /dev/vg_mbr_ext4/lv_data /mnt/ext4_data # Mount temporarily
# Permanent Mount:
blkid >> /etc/fstab                 # Get UUID
vi /etc/fstab                       # Add: UUID=your-uuid /mnt/ext4_data ext4 defaults 0 0
mount -a                            # Verify fstab

2) MBR + XFS Flow MBR XFS

Step 1: Partitioning (MBR)
fdisk /dev/sdb
# Same as above: 'n', 'p', 't' (8e), 'w'
Step 2: Create PV, VG, LV
partprobe /dev/sdb
pvcreate /dev/sdb1
vgcreate vg_mbr_xfs /dev/sdb1
lvcreate -L 9G -n lv_data vg_mbr_xfs
Step 3: Filesystem (XFS) & Mount
mkfs.xfs /dev/vg_mbr_xfs/lv_data   # Create XFS filesystem
mkdir /mnt/xfs_data
mount /dev/vg_mbr_xfs/lv_data /mnt/xfs_data
# Permanent Mount:
blkid >> /etc/fstab
vi /etc/fstab                       # Add: UUID=your-uuid /mnt/xfs_data xfs defaults 0 0
mount -a

3) GPT + ext4 Flow GPT ext4

Step 1: Partitioning (GPT)
gdisk /dev/sdc
# Press 'n' (new), Enter (default part num), Enter (default start), +10G (size)
# Press 't' (type), '8e00' (Linux LVM) - THIS IS CRITICAL FOR GPT
# Press 'w' (write)

Rationale: GPT uses gdisk and requires hex code '8e00' for LVM. fdisk cannot safely manipulate GPT tables.

Step 2: Create PV, VG, LV
partprobe /dev/sdc
pvcreate /dev/sdc1
vgcreate vg_gpt_ext4 /dev/sdc1
lvcreate -L 9G -n lv_data vg_gpt_ext4
Step 3: Filesystem (ext4) & Mount
mkfs.ext4 /dev/vg_gpt_ext4/lv_data
mkdir /mnt/gpt_ext4_data
mount /dev/vg_gpt_ext4/lv_data /mnt/gpt_ext4_data
# Update fstab with UUID as shown above

4) GPT + XFS Flow GPT XFS

Step 1: Partitioning (GPT)
gdisk /dev/sdc
# Press 'n', Enter, Enter, +10G
# Press 't', '8e00' (Linux LVM)
# Press 'w'
Step 2: Create PV, VG, LV
partprobe /dev/sdc
pvcreate /dev/sdc1
vgcreate vg_gpt_xfs /dev/sdc1
lvcreate -L 9G -n lv_data vg_gpt_xfs
Step 3: Filesystem (XFS) & Mount
mkfs.xfs /dev/vg_gpt_xfs/lv_data
mkdir /mnt/gpt_xfs_data
mount /dev/vg_gpt_xfs/lv_data /mnt/gpt_xfs_data
# Update fstab with UUID as shown above

PART 5-6: COMPLETE LVM EXTEND FLOWS (XFS)

Extension flow is identical for MBR and GPT once the new storage is added to the VG. The crucial difference lies in the filesystem resize command. XFS cannot be reduced and must be mounted to be grown.

5 & 6) Extending LVM with XFS (MBR/GPT)

Step 1: Add new physical disk and partition
# If MBR:
fdisk /dev/sdd  # Create partition, type 8e

# If GPT:
gdisk /dev/sdd  # Create partition, type 8e00

partprobe /dev/sdd
Step 2: Extend the Volume Group
pvcreate /dev/sdd1                     # Initialize new disk
vgextend vg_gpt_xfs /dev/sdd1          # Add to existing VG
vgs                                     # Verify free space (VFree)
Step 3: Extend the Logical Volume
lvextend -L +5G /dev/vg_gpt_xfs/lv_data  # Add 5G to LV
# OR use all free space:
lvextend -l +100%FREE /dev/vg_gpt_xfs/lv_data
Step 4: Grow the XFS Filesystem Critical Step
# XFS MUST be mounted to grow. It uses the mount point, NOT the device path!
xfs_growfs /mnt/gpt_xfs_data

Rationale: lvextend only expands the block device. xfs_growfs expands the filesystem to fill the new block space. For ext4, this would be resize2fs /dev/vg/lv, but XFS requires the mount point and an online mounted state.

PART 7: COMPLETE LVM REDUCE FLOW (ext4 / MBR)

WARNING: XFS CANNOT BE REDUCED. This flow applies strictly to ext4/ext3. Reduction must happen OFFLINE (unmounted).

7) Reducing ext4 LVM safely

Step 1: Unmount the filesystem
umount /mnt/ext4_data

Rationale: ext4 reduction is NOT supported online. You must unmount to ensure no data is being written while metadata sizes change.

Step 2: Force filesystem check
e2fsck -f /dev/vg_mbr_ext4/lv_data

Rationale: Mandatory step. Ensures filesystem integrity before manipulating block sizes.

Step 3: Reduce the FILESYSTEM FIRST Order Matters!
resize2fs /dev/vg_mbr_ext4/lv_data 5G

Rationale: If you reduce the LV first, you chop off the end of the filesystem, causing catastrophic corruption. You must tell the filesystem to shrink into a smaller boundary first.

Step 4: Reduce the LOGICAL VOLUME SECOND
lvreduce -L 5G /dev/vg_mbr_ext4/lv_data
# Type 'y' to confirm the danger

Rationale: Now that the filesystem is safely 5G, you can shrink the underlying block device to 5G to match.

Step 5: Remount and Verify
mount /dev/vg_mbr_ext4/lv_data /mnt/ext4_data
df -h /mnt/ext4_data

PART 8: COMMON MISTAKES & TROUBLESHOOTING

Mistake 1: Reducing LV before Filesystem (Catastrophic)

The Mistake: Running lvreduce -L 5G before resize2fs.

What Happens: LVM literally cuts off the end of the block device. The filesystem still thinks it's 9G and tries to access blocks that no longer exist. Result: Kernel panic, I/O errors, massive data corruption.

Troubleshooting/Fix: If you haven't written new data, you can often recover by immediately extending the LV back to its original size (lvextend -L 9G), then running fsck. If data was overwritten, you need backups.

Mistake 2: Forgetting to run resize2fs / xfs_growfs after lvextend

The Mistake: Running lvextend but forgetting the filesystem resize step.

What Happens: lvs will show the LV is 14GB, but df -h will show the mount is still 9GB. The space is allocated to the block device but unavailable to users.

Troubleshooting/Fix: Just run resize2fs /dev/vg/lv or xfs_growfs /mount. It is safe to do this later; no data is lost.

Mistake 3: Attempting to reduce an XFS Filesystem

The Mistake: Trying to run resize2fs or xfs_growfs to shrink XFS, or running lvreduce on an XFS LV.

What Happens: XFS does not support shrinking. If you reduce the LV underneath XFS, you will permanently corrupt the filesystem.

Troubleshooting/Fix: The only way to "shrink" XFS is to back up all data (xfsdump), destroy the LV (lvremove), recreate a smaller LV (lvcreate), format it (mkfs.xfs), and restore data (xfsrestore).

Mistake 4: Using xfs_growfs with device path instead of mount point

The Mistake: Running xfs_growfs /dev/vg_gpt_xfs/lv_data.

What Happens: Command fails. XFS tools require the mount point because they operate on the mounted VFS, not the raw block device.

Troubleshooting/Fix: Use the mount point: xfs_growfs /mnt/gpt_xfs_data.

Mistake 5: System drops to Emergency Mode after reboot (fstab issues)

The Mistake: Typo in /etc/fstab, missing mount point directory, or missing LVM volume.

What Happens: System halts boot process and drops to emergency shell.

Troubleshooting/Fix:
1. Enter root password.
2. Remount root as read-write: mount -o remount,rw /
3. Edit fstab: vi /etc/fstab and comment out the bad line.
4. Reboot: systemctl reboot

Mistake 6: Device is busy (lsof / fuser)

The Mistake: Trying to umount a filesystem while a user/process is inside it or using files.

What Happens: umount: /mnt/data: target is busy.

Troubleshooting/Fix:
Find who is using it: lsof /mnt/data or fuser -v /mnt/data.
Kill the process or ask the user to cd out of the directory, then unmount.

PART 9: INTERVIEW QUESTIONS & ANSWERS

Q1: Why must you reduce the ext4 filesystem BEFORE reducing the Logical Volume?

LVM doesn't know about the filesystem inside it; it only knows about blocks. If you shrink the LV first, LVM cuts off the end of the storage block. The filesystem still thinks it has that space and tries to write to sectors that no longer exist, resulting in immediate filesystem corruption. You must tell the filesystem to pack its data into a smaller area first, then shrink the boundary around it.

Q2: Can you shrink an XFS filesystem? What is the alternative?

No, XFS cannot be shrunk. It is architecturally designed for growth and high-performance allocation, lacking the ability to relocate data backward safely. The alternative to reclaim space is: 1) Backup data (xfsdump/tar), 2) Destroy the LV, 3) Recreate a smaller LV, 4) Format with XFS, 5) Restore data.

Q3: What is the difference between resize2fs and xfs_growfs?

resize2fs is used for ext2/3/4 filesystems. It can both grow and shrink. It takes the device path (e.g., /dev/vg/lv).
xfs_growfs is used for XFS. It can ONLY grow. It takes the mount point path (e.g., /mnt/data) and requires the filesystem to be mounted online.

Q4: You ran lvextend but df -h shows the old size. What did you forget?

You forgot to resize the filesystem. The Logical Volume is just a container; you must expand the filesystem inside it to utilize the new space. Run resize2fs /dev/vg/lv (for ext4) or xfs_growfs /mountpoint (for XFS).

Q5: What is the LVM partition type code for MBR and GPT?

For MBR (using fdisk), the partition type code for Linux LVM is 8e. For GPT (using gdisk), the hex code is 8e00.

Q6: How can you extend an LV without risking the typo of forgetting the filesystem resize?

Use the -r flag with lvextend. For example: lvextend -r -L +5G /dev/vg/lv. This tells LVM to automatically call the appropriate filesystem resize tool (resize2fs or xfs_growfs) after extending the LV.
