PART I: STEP-BY-STEP PROCEDURE - BASH CONDITIONAL FLOW

STEP 1: The Test Command Evaluation

Bash evaluates the condition inside [ ] or [[ ]]. This is technically calling the test builtin.

Rationale: Unlike other programming languages where conditions evaluate to True/False, Bash evaluates based on the exit status of the test command. An exit status of 0 means True (success), and non-zero means False (failure).

STEP 2: The then Branch

If the evaluated exit status is 0, the script flows into the block following then.

Rationale: then acts as the delimiter separating the condition from the execution block. It is syntactically required to tell Bash where the "if true" logic begins.

STEP 3: Handling False Outcomes (elif / else)

If the initial condition fails (non-zero exit status), Bash checks for elif (else if) conditions, or falls back to the else block if provided.

Rationale: This allows for complex decision trees and graceful handling of unexpected or default scenarios.

STEP 4: Closure with fi

Close the entire conditional structure with fi (if spelled backwards).

Rationale: Bash doesn't use curly braces {} for if blocks like C/Java. fi explicitly tells the interpreter that the conditional logic has ended and normal sequential execution should resume.

PART III: ALTERNATIVE COMMANDS & METHODS

1. Single Brackets [ ] vs Double Brackets [[ ]]
  • [ ]: POSIX standard. Works in sh. Prone to errors if variables contain spaces (requires heavy quoting). Does not support regex matching =~ or logical &&/|| inside the brackets.
  • [[ ]]: Bash keyword (not POSIX). Safer. Handles spaces in variables automatically. Supports advanced pattern matching and logical operators inside. Preferred for Bash scripts.
2. if/elif vs case statements

When checking a single variable against multiple distinct string values or number ranges, case statements are cleaner and faster to read than long if/elif chains.

case $animal in
  dog) echo "Woof";;
  cat) echo "Meow";;
  *)   echo "Unknown";;
esac
3. Arithmetic Evaluation (( ))

For pure integer math, (( )) is the best alternative. It uses standard C-language operators (>, <, ==), eliminating the need to remember -gt, -lt, etc.

if (( a > b )); then
  echo "A is greater"
fi

PART IV: ASSIGNMENT SOLUTIONS

Section A: Theory Questions

1. What is an if statement in Bash?

A conditional statement that allows a script to make decisions. It executes a block of code only if a specified condition evaluates to true (exit status 0).

2. Why do we use an if statement?

To control the flow of a script based on dynamic conditions (like user input, file existence, or variable values), allowing the script to react differently to different situations.

3. Basic syntax of an if statement.
if [ condition ]; then
  # commands
fi
4. Purpose of then?

then separates the test condition from the commands to be executed if the condition is true. It's syntactically required in Bash.

5. Difference between if and if-else?

if executes code only if the condition is true (nothing happens if false). if-else provides an alternative block of code to execute if the condition is false.

6. Use of elif?

elif (else if) allows checking multiple conditions in sequence. If the first if fails, it checks the elif condition, preventing deeply nested if statements.

7. Numeric comparison operators?

-eq, -ne, -gt, -ge, -lt, -le.

8. Purpose of fi?

fi marks the end of an if statement block.

9. Check file existence?
if [ -f "filename.txt" ]; then
  echo "Exists"
fi
10. Nested if statements?

Placing one if statement inside the block of another if statement. Used when decisions depend on multiple layers of conditions (e.g., if file exists, then check if it is writable).

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: Why are spaces around the brackets [ condition ] mandatory in Bash?

In Bash, [ is actually a command (equivalent to the test command), not just a syntax delimiter. The trailing ] is its required final argument. If you write [$var], Bash tries to parse it as a single word (e.g., [10]), which results in a "command not found" error. The spaces tell Bash to execute the [ command with separate arguments.

Q2: What is the difference between = and == when comparing strings?

Inside single brackets [ ], only = is POSIX standard for string comparison. == works in Bash but isn't universally portable. Inside double brackets [[ ]], both work identically. However, note that in arithmetic context (( )), == is the equality operator, while = is an assignment.

Q3: Why does if [ $var = "yes" ] fail when $var is empty, and how do you fix it?

If $var is empty, the statement expands to if [ = "yes" ]. The [ command sees = as the first argument and throws a "unary operator expected" error. Fix: Always quote variables: if [ "$var" = "yes" ]. When empty, it expands to if [ "" = "yes" ], which is valid syntax. Alternatively, use [[ $var = "yes" ]], which handles empty variables safely without quotes.

Q4: How do you check multiple conditions (AND / OR) in an if statement?
# Using POSIX single brackets (commands must be separate)
if [ $age -gt 18 ] && [ $country = "US" ]; then ...  # AND
if [ $day = "Sat" ] || [ $day = "Sun" ]; then ...   # OR

# Using Bash double brackets (operators go inside)
if [[ $age -gt 18 && $country = "US" ]]; then ...   # AND
if [[ $day = "Sat" || $day = "Sun" ]]; then ...     # OR
