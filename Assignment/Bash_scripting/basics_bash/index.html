PART I: STEP-BY-STEP PROCEDURE - SCRIPT LIFECYCLE

STEP 1: Script Creation & Shebang

Create a file with .sh extension. The very first line must be the "shebang" (#!/bin/bash), which tells the OS to use the Bash interpreter.

Rationale: Without the shebang, the system might default to sh, which lacks many Bash-specific features, leading to unexpected syntax errors.

STEP 2: Writing Logic & Commands

Write standard Linux commands, variables, loops, and conditionals inside the file. Add comments using # to explain complex logic.

Rationale: Comments make the script maintainable. Scripts are often read by others or by yourself months later.

STEP 3: Setting Execute Permissions

By default, text files lack execute permissions. Use chmod +x script.sh to make it executable.

Rationale: The Linux kernel will refuse to execute a file unless the execute bit is set for the user attempting to run it.

STEP 4: Execution & Debugging

Run the script using ./script.sh or bash script.sh. If errors occur, debug using bash -x script.sh.

Rationale: ./ requires execute permissions and uses the shebang. bash bypasses permissions and ignores the shebang. -x prints every command before execution, vital for troubleshooting logic errors.

PART II: COMMANDS AND EXPLANATIONS

#!/bin/bash (Shebang)

Purpose: Directive that tells the operating system which interpreter to use to parse the rest of the file. Must be the absolute path.

chmod +x

Syntax: chmod +x script.sh

Purpose: Grants execute permissions to the file for the owner, group, and others.

Core Utility Commands Used in Scripts

  • date: Prints current system date/time (useful for timestamping logs).
  • whoami: Outputs the current logged-in username (good for privilege checks).
  • df -h: Reports file system disk space usage (human-readable).
  • free -m: Displays RAM/Swap usage in megabytes.
  • uptime: Shows how long the system has been running.
  • tar -czf: Creates compressed gzip archives (backups).

PART III: ALTERNATIVE COMMANDS & METHODS

1. bash script.sh vs ./script.sh
  • bash script.sh: Explicitly invokes the Bash shell to run the script. Ignores the shebang. Does not require execute permissions on the file (only read permission).
  • ./script.sh: Runs the script as an executable program. Requires execute permissions. Respects the shebang line to determine the interpreter.
2. source script.sh (or . command)

Unlike the previous methods, source executes the script in the current shell environment. This means variables and functions defined in the script persist in your terminal after it finishes. The other methods spawn a subshell, which is destroyed when the script ends.

3. Automation Alternatives: cron vs systemd timers

While Bash scripts hold the logic, they need a scheduler to run automatically.

  • cron: Traditional, simple, time-based scheduler (e.g., run every hour).
  • systemd timers: Modern alternative. Offers more flexibility (e.g., run 5 minutes after boot, or run based on system events) and better logging via journalctl.

Section B: Short Answer Questions

1. What is Bash scripting?

Bash scripting is the process of writing a series of commands for the Bash shell to execute sequentially. It allows users to combine multiple Linux commands, variables, and logic structures into a reusable file to automate complex or repetitive tasks.

2. Why is Bash scripting useful in Linux?

It eliminates the need to type the same commands repeatedly, reduces human error, ensures consistency in task execution, and allows scheduling of unattended operations (like backups or updates).

3. Role of #!/bin/bash in a script?

It is the "shebang." It instructs the operating system to use the Bash interpreter located at /bin/bash to parse and run the script.

4. Command to run a Bash script.
chmod +x script.sh  # (First time only)
./script.sh
5. Two uses of Bash scripting.

1. Automating system backups using tar and rsync.
2. Monitoring system resources (CPU, RAM) and sending alerts via email.

6. How does Bash scripting reduce human errors?

By executing a pre-written, tested sequence of commands, it removes the chance of typos, forgotten steps, or incorrect flags that commonly occur during manual typing.

7. What is automation in Linux?

Automation is using scripts and tools (like cron or systemd) to perform system tasks without continuous human intervention, allowing tasks to run at specific times or in response to events.

8. Purpose of backup scripts?

To systematically copy and compress critical data to a safe location at regular intervals, ensuring data can be recovered in case of hardware failure, accidental deletion, or corruption.

9. How can Bash scripts help in system monitoring?

Scripts can parse outputs from df, free, or top, filter for thresholds (e.g., Disk > 90%), and trigger alerts (emails or logs) automatically when the system is under stress.

10. Use of the date command in Bash?

It outputs the current system date and time. In scripts, it is heavily used to generate dynamic filenames (e.g., backup_$(date +%F).tar.gz) and to timestamp log entries.

BACKUP_DIR="/backup"
DATE=$(date +%F)

Section E: Scenario-Based Questions

1. Admin performs the same backup daily manually. How can Bash help?

The admin can write a Bash script containing the tar or rsync commands to archive the necessary directories. Then, schedule this script using cron (e.g., 0 2 * * * /path/to/backup.sh) to run automatically at 2 AM daily, eliminating manual effort.

2. Company wants to monitor disk space automatically every hour.

Write a Bash script that checks disk usage using df and awk, and sends an email using mail or mutt if usage exceeds 90%. Schedule this script using a cron job with an hourly interval: 0 * * * * /path/to/disk_monitor.sh.

3. User wants to install multiple packages with one command.

Create a script (e.g., setup.sh) that lists all the apt install or yum install commands with the -y flag. The user simply runs ./setup.sh once, and all packages are installed unattended.

4. Team repeatedly checks CPU and memory manually. Explain improvement.

Build a monitoring script using top, free, and mpstat that outputs a clean, formatted report. The team can run one script instead of typing five different commands, or the script can append data to a central log file for historical analysis.

5. Admin wants to reduce human errors during repetitive tasks. How can Bash help?

By converting the task into a script, the admin ensures the exact same commands, flags, and sequences are used every time. This removes variables like fatigue, typos, or forgotten steps, guaranteeing consistent and error-free execution.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What is the difference between ./script.sh and source script.sh?

./script.sh executes the script in a subshell (a separate process). Any variables created or modified inside the script are destroyed when the script finishes and do not affect the parent terminal.

source script.sh (or . script.sh) executes the script in the current shell environment. Variables and functions defined in the script remain active in your terminal after execution. This is commonly used for applying changes to ~/.bashrc.

Q2: What happens if a Bash script lacks the shebang (#!/bin/bash)?

If you run it using bash script.sh, nothing happens because you explicitly invoked Bash. However, if you run it using ./script.sh, the system will use your default shell (defined by $SHELL, which might be sh, zsh, or dash). Since Bash contains features not present in POSIX sh, the script will likely throw syntax errors.

Q3: How do you debug a Bash script?

The most common method is using the -x flag: bash -x script.sh. This prints each command and its arguments to standard error before it is executed, allowing you to trace the flow and see exactly where logic or variable expansion fails.

Alternatively, you can add set -x at the top of the script to enable debugging for the whole file, or wrap specific sections between set -x and set +x.

Q4: Explain Exit Status codes in Bash scripting.

Every command executed in Bash returns an exit status (an integer between 0 and 255) to the system. An exit status of 0 indicates success. Any non-zero exit status (1-255) indicates failure. You can check the exit status of the last run command using the special variable $?. This is fundamental for chaining commands conditionally (e.g., command1 && command2).

Q5: What is the significance of set -e in a Bash script?

Adding set -e at the top of a script tells Bash to exit immediately if any command exits with a non-zero status (fails). Without it, Bash will execute the next line regardless of whether the previous line failed, which can be dangerous in scripts handling backups or deletions.
