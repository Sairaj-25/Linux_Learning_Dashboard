PART I: STEP-BY-STEP PROCEDURE

STEP 1: DISK DETECTION

Identify available disks on the system before partitioning.

Rationale: You must know the correct device name (e.g., /dev/sdb) to avoid modifying the wrong disk, which could cause data loss.

STEP 2: OPENING THE PARTITIONING UTILITY

Launch fdisk (or another partitioning tool) targeting the correct disk.

Rationale: fdisk is the standard interactive utility for MBR partition management on Linux.

STEP 3: CREATING THE PARTITION

Use the 'n' command inside fdisk to create a new partition. Choose partition type (primary/extended), partition number, starting sector, and size.

Rationale: The partition table must have an entry before the disk space can be formatted or used.

STEP 4: WRITING THE PARTITION TABLE

Use the 'w' command to write changes to disk and exit fdisk.

Rationale: Changes are only held in memory until written. 'w' commits them to the actual disk's partition table.

STEP 5: INFORMING THE KERNEL

Run partprobe or partx to inform the OS kernel of the new partition.

Rationale: The kernel may not immediately detect partition table changes. partprobe forces a re-read without requiring a reboot.

STEP 6: FORMATTING (CREATING A FILE SYSTEM)

Use mkfs (e.g., mkfs.ext4 or mkfs.xfs) to create a file system on the new partition.

Rationale: A raw partition without a file system cannot store files. The file system provides the structure for data organization.

STEP 7: CREATING A MOUNT POINT

Create a directory (e.g., /mnt/data) that will serve as the access point.

Rationale: In Linux, file systems are accessed through a directory tree. The mount point is the directory where the partition attaches.

STEP 8: MOUNTING THE PARTITION

Use the mount command to attach the file system to the mount point.

Rationale: Until mounted, the file system exists on the disk but is not accessible through the directory hierarchy.

STEP 9: VERIFYING THE MOUNT

Use df -h or mount | grep to confirm the partition is mounted correctly.

Rationale: Verification ensures the partition is accessible and has the expected size and file system type.

STEP 10: MAKING THE MOUNT PERMANENT

Retrieve the UUID using blkid, add an entry to /etc/fstab, and verify with mount -a.

Rationale: Temporary mounts are lost on reboot. /etc/fstab ensures the partition is automatically mounted every time the system boots.

PART II: COMMANDS AND EXPLANATIONS

lsblk

Syntax: lsblk [options]

Purpose: Lists information about all available or specified block devices in a tree-like structure. Shows names, sizes, types (disk/part), and mount points.

Example: lsblk

fdisk -l

Syntax: fdisk -l [device]

Purpose: Lists the partition tables for the specified device(s). Shows start/end sectors, sizes, types, and partition IDs.

Example: fdisk -l /dev/sda

fdisk /dev/sdX

Syntax: fdisk /dev/sdX

Purpose: Opens the interactive fdisk utility for the specified disk. Used to create, delete, modify, and view partitions on MBR-based disks.

Example: fdisk /dev/sdb

PART III: ALTERNATIVE COMMANDS

1. Disk Detection Alternatives
  • lsblk : Tree view, shows mount points (recommended for overview)
  • fdisk -l : Detailed partition table with sector info
  • ls -l /dev/sd* : Raw listing of disk device files
  • cat /proc/partitions : Kernel's view of partitions
  • parted -l : Works with both MBR and GPT, shows free space

Differences: lsblk is the most user-friendly. fdisk -l gives sector-level detail. parted -l is GPT-aware. /proc/partitions is the kernel's direct view.

2. Partitioning Tool Alternatives
  • fdisk : MBR only, interactive, most common for standard partitions
  • parted : Supports both MBR and GPT, can be scripted, handles larger disks (>2TB)
  • gdisk : GPT equivalent of fdisk, for UEFI/GPT systems
  • cfdisk : Curses-based (TUI) partitioning tool, more visual/user-friendly
  • sfdisk : Scriptable version of fdisk, useful for automation

Differences: fdisk is limited to MBR (max 2TB, 4 primary partitions). parted and gdisk support GPT (128+ partitions, disks >2TB). cfdisk offers a visual interface. sfdisk is best for automation.

3. Kernel Notification Alternatives
  • partprobe /dev/sdX : Most common, requests kernel re-read partition table of specific/all devices
  • partx -a /dev/sdX : Tells kernel about specific partitions; more granular control
  • kpartx /dev/sdX : Creates device mappings for partitionable devices
  • hdparm -z /dev/sdX : Forces kernel to re-read partition table
  • Reboot : The guaranteed but time-consuming method

Differences: partprobe is the standard. partx is more precise and can add individual partitions. Reboot is the last resort.

4. File System Creation Alternatives
  • mkfs.ext4 : Default Linux FS, journaling, widely supported
  • mkfs.xfs : High performance, great for large files, default in RHEL
  • mkfs.btrfs : Advanced features (snapshots, subvolumes, compression)
  • mkfs.ext3 : Older journaling FS, predecessor to ext4
  • mkswap : Creates swap space (not a regular file system)

Differences: ext4 is most compatible. XFS excels with large files and parallel I/O but cannot be shrunk. btrfs offers modern features but is more complex.

5. Mount Verification Alternatives
  • df -h : Shows mounted FS with space usage
  • mount | grep /dev/sdXN : Shows raw mount data for specific device
  • findmnt /mnt/point : Tree-formatted mount info (most readable)
  • lsblk /dev/sdXN : Shows mount point in device tree
  • cat /proc/mounts : Kernel's mount table directly

Differences: findmnt is the most readable. df -h includes space info. cat /proc/mounts is the raw kernel data.

PART IV: ASSIGNMENT SOLUTIONS

Section A & B: Short & Medium Answers

1. What is the purpose of disk partitioning?

Disk partitioning divides a physical disk into one or more logical sections called partitions. This allows: Separation of OS files from user data, multiple OSes on one disk (dual-boot), different file systems on the same disk, efficient disk space management, improved security through isolation, and simplified backup and recovery.

2. Which command is used to check available disks in Linux?

The primary command is: lsblk. Alternative: fdisk -l.

3. What is the use of the lsblk command?

lsblk lists information about all available block devices in a tree-like format. It displays device names, major/minor numbers, sizes, types (disk or partition), and mount points, providing a quick overview of storage devices and their partitions.

4. What does fdisk -l display?

fdisk -l displays the partition tables of all detected disks (or a specific disk). It shows: disk name, disk size, sector sizes, partition entries with start/end sectors, partition sizes, partition types, and partition IDs.

5. Which command is used to open the fdisk utility?

fdisk /dev/sdX (replace sdX with the target disk, e.g., fdisk /dev/sdb).

6. What does the n option do in fdisk?

The 'n' option creates a new partition. After pressing 'n', fdisk prompts for: partition type (primary p or extended e), partition number (1-4), first sector, and last sector or size (e.g., +5G for 5GB).

7. What is the difference between primary and extended partitions?

Primary partitions: Bootable partitions that can contain an OS or data. MBR allows a maximum of 4 primary partitions.
Extended partitions: A special type that cannot hold data directly but acts as a container for logical partitions. Only one extended partition is allowed per disk. It overcomes the 4-partition MBR limit.

8. How many primary partitions are allowed in MBR?

Maximum of 4 primary partitions. If more are needed, one primary partition must be replaced with an extended partition, which can contain multiple logical partitions.

9. What does the p command do inside fdisk?

The 'p' command prints the current partition table of the disk, showing all existing partitions with their device names, start/end sectors, sizes, types, and IDs.

10. What is the use of the w command in fdisk?

The 'w' command writes the partition table changes to the disk and exits fdisk. Without using 'w', all modifications made in the fdisk session are discarded. This is the commit/save operation.

11. Explain the steps to create a new partition using fdisk.

Step 1: Open fdisk targeting the disk: fdisk /dev/sdb
Step 2: Press 'n' to create a new partition.
Step 3: Choose partition type: 'p' for primary or 'e' for extended.
Step 4: Select partition number (1-4).
Step 5: Specify the first sector (press Enter for default/first available).
Step 6: Specify the last sector or size (e.g., +5G for 5GB, or Enter for all remaining space).
Step 7: Press 'p' to verify the new partition appears in the table.
Step 8: Press 'w' to write changes and exit.

12. How do you specify partition size while creating a partition?

When fdisk prompts for the "Last sector", you can specify size using the format: +<size><unit>. Examples: +500M = 500 megabytes, +5G = 5 gigabytes, +2T = 2 terabytes. If you press Enter without specifying, fdisk uses all remaining free space on the disk.

13. What is the role of first sector and last sector in partitioning?

First sector: Defines where the partition begins on the disk. It determines the starting point and alignment of the partition. The default is usually the first available free sector.
Last sector: Defines where the partition ends. Combined with the first sector, it determines the total size of the partition.
Together, they define the exact range of disk sectors allocated to the partition. The partition size = (last sector - first sector + 1) x sector size.

14. How can you verify a partition after creating it?

Methods to verify:
a) Inside fdisk: Press 'p' to print the partition table and confirm the new entry exists.
b) After exiting fdisk: Run lsblk to see the new partition in the device tree.
c) Run fdisk -l /dev/sdX to see detailed partition information.
d) Check if the device file exists: ls -l /dev/sdXN
e) After formatting: Use blkid /dev/sdXN to confirm file system type.

15. Why is the partprobe command used?

partprobe is used to inform the Linux kernel of changes to the partition table on a disk. After creating or deleting a partition with fdisk, the kernel's in-memory partition table may not be updated immediately. partprobe forces the kernel to re-read the partition table, making the new partition device file (e.g., /dev/sdb1) available without requiring a system reboot. Usage: partprobe /dev/sdb

16. What is formatting in Linux?

Formatting (or creating a file system) is the process of writing a file system structure onto a raw partition. This structure includes: Superblock (Metadata about the FS), Inode tables (Metadata about files), Data blocks (Where actual file contents are stored), and Journal (For journaling file systems like ext4, xfs). Without formatting, a partition is just raw storage with no structure to organize or retrieve files. The mkfs family of commands is used for formatting.

17. Explain how to format a partition using ext4 or XFS file system.

For ext4: mkfs.ext4 /dev/sdb1 (Creates ext4 with default parameters).
For XFS: mkfs.xfs /dev/sdb1 (Creates XFS with default parameters).
Optional parameters: mkfs.ext4 -L mydata /dev/sdb1 (sets volume label to "mydata").
IMPORTANT: Formatting destroys any existing data on the partition.

18. What is a mount point?

A mount point is a directory in the Linux file system hierarchy where a disk partition or file system is attached (mounted). Once mounted, the contents of the partition become accessible through that directory. For example, if /dev/sdb1 is mounted on /mnt/data, accessing /mnt/data shows the contents of that partition. The mount point directory must exist before mounting.

19. How do you mount a partition in Linux?

Step 1: Create a mount point directory: mkdir -p /mnt/data
Step 2: Mount the partition: mount /dev/sdb1 /mnt/data
Step 3: Verify the mount: df -h | grep /mnt/data or mount | grep /dev/sdb1
This is a temporary mount that will be lost after reboot. For permanent mounting, add an entry to /etc/fstab.

20. What is the purpose of the df -h command?

df -h displays disk space usage for all currently mounted file systems. The -h flag outputs sizes in human-readable format (KB, MB, GB) instead of 1K blocks. It shows: File system device, total size, used space, available space, percentage used, and mount point. Useful for verifying mount status and checking available disk space.

2. Describe the full workflow from disk detection to mounting a partition.

PHASE 1: DISK DETECTION - The kernel detects hardware during boot or hot-plug events. Device files are created in /dev/. Verify with lsblk.
PHASE 2: PARTITION CREATION - Use fdisk/parted to divide disk into usable partitions. Partition table (MBR/GPT) is written to the disk. Device files for partitions appear. Inform kernel with partprobe.
PHASE 3: FILE SYSTEM CREATION - Use mkfs commands to create a file system structure. This writes metadata (superblock, inode tables, journal). Verify with blkid.
PHASE 4: MOUNTING - Create a mount point directory. Attach the file system to the directory tree. The partition contents are now accessible. Verify with df -h.
PHASE 5: PERSISTENT CONFIGURATION - Retrieve UUID, edit /etc/fstab, test with mount -a. This ensures the partition is mounted automatically on every boot.

3. Explain how to make a partition mount permanent in Linux using /etc/fstab.

The /etc/fstab (File System Table) file controls automatic mounting at boot. Each line has six fields:
<device> <mount_point> <fs_type> <options> <dump> <fsck_order>

Fields Explained:
- device: UUID=xxxx-xxxx (recommended) or /dev/sdXN
- mount_point: Directory where the FS is mounted (e.g., /mnt/data)
- fs_type: File system type (e.g., ext4, xfs)
- options: Mount options (defaults = rw,suid,dev,exec,auto,nouser,async)
- dump: Backup utility flag (0 = no dump, 1 = dump)
- fsck_order: File system check order at boot (0 = skip, 1 = root, 2+ = others)

Steps: Get UUID via blkid, backup fstab, edit fstab, test via umount /mnt/data then mount -a, verify via df -h.

4. What is UUID and why is it important in mounting partitions?

UUID (Universally Unique Identifier) is a 128-bit identifier assigned to a file system when it is created (formatted). It is guaranteed to be unique across all systems.

Importance in Mounting:
a) PERSISTENCE: Device names like /dev/sdb1 are assigned dynamically by the kernel based on detection order. If a new disk is added, /dev/sdb might become /dev/sdc. UUIDs remain constant.
b) RELIABILITY: Using UUID in /etc/fstab ensures the system always mounts the correct file system, preventing data corruption or boot failures.
c) PORTABILITY: If a disk is moved to another system, its UUID remains the same.
d) CLONE SAFETY: When cloning disks, new UUIDs are generated for cloned file systems, preventing conflicts.

5. Explain the role of file system creation and mounting in disk management.

FILE SYSTEM CREATION (FORMATTING): A raw partition is simply a contiguous block of sectors with no organizational structure. File system creation writes the metadata and structural framework needed to store, organize, retrieve, and manage files (Superblock, Inode table, Data blocks, Journal). Without a file system, applications cannot store or retrieve data.

MOUNTING: Linux uses a single unified directory hierarchy starting from /. Mounting is the mechanism that connects a file system to this hierarchy. Without mounting, the file system exists on disk but is inaccessible to users and applications. Proper mount management ensures data is accessible where expected, FS options are enforced, and the system boots correctly.

TOGETHER: File system creation provides the structure; mounting provides the access. The complete workflow (detect -> partition -> format -> mount -> persist) transforms raw hardware into usable storage.

Scenario 1: You created a partition but it is not visible. What steps will you take?

Step 1: Run partprobe to inform the kernel of partition table changes: partprobe /dev/sdb
Step 2: Check if the partition device file now exists: ls -l /dev/sdb1
Step 3: If still not visible, verify the partition was actually written: fdisk -l /dev/sdb
Step 4: If partprobe fails, try partx: partx -a /dev/sdb
Step 5: As a last resort, reboot the system: reboot
ROOT CAUSE: The most common reason is that the kernel has not re-read the updated partition table. partprobe resolves this in most cases.

Scenario 2: A partition is created but not mounted. How will you fix it?

Step 1: Verify the partition exists and is formatted: lsblk /dev/sdb1 and blkid /dev/sdb1
Step 2: If blkid shows no output, format it: mkfs.ext4 /dev/sdb1
Step 3: Create a mount point if it does not exist: mkdir -p /mnt/data
Step 4: Mount the partition: mount /dev/sdb1 /mnt/data
Step 5: Verify the mount: df -h | grep /mnt/data
Step 6: To make it persistent across reboots, add to /etc/fstab.

Scenario 3: After reboot, your mounted partition disappears. What is the issue and solution?

ISSUE: The partition was mounted temporarily using the mount command. Temporary mounts are lost when the system reboots because they are not recorded in the persistent mount configuration file (/etc/fstab).

SOLUTION: Step 1: Get the UUID: blkid /dev/sdb1
Step 2: Edit /etc/fstab: vi /etc/fstab and add UUID=<uuid> /mnt/data ext4 defaults 0 2
Step 3: Test the configuration without rebooting: mount -a
Tip A malformed /etc/fstab can cause boot failure; always test with mount -a before rebooting.

When prompted for the "Last sector", type: +500M

Size Specification Format in fdisk:
+500M = 500 megabytes
+500K = 500 kilobytes
+500G = 500 gigabytes

NOTE: The '+' sign is essential. Without it, fdisk interprets the value as an absolute sector number rather than a relative size. 'M' for megabytes (MiB in practice for fdisk, which uses 1024-based units).

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What happens if you use 'q' instead of 'w' in fdisk after creating a partition?

If you press 'q' instead of 'w' in fdisk, the utility quits WITHOUT saving any changes. The new partition you created exists only in fdisk's memory and is never written to the disk's partition table. This is actually a safety feature - if you make a mistake, you can quit without saving. Always use 'p' to verify before 'w'.

Q2: Why is using UUID preferred over device names in /etc/fstab?

UUID is preferred because: Device names (/dev/sdb1) are assigned dynamically based on detection order and can change when disks are added, removed, or if the kernel probes in a different order. UUIDs are generated when the file system is created and remain constant regardless of hardware changes. Using device names in /etc/fstab can cause boot failures or wrong partitions being mounted if device names change.

Q3: What is the difference between fdisk and parted? When would you use each?

fdisk: Supports only MBR partition tables (max 2TB disk, 4 primary partitions). Interactive, user-friendly, and the most common tool for standard partitioning on smaller disks.
parted: Supports both MBR and GPT partition tables. Can handle disks larger than 2TB and GPT's 128+ partitions. Can also resize partitions without data loss. Can be scripted for automation.
Use fdisk for simple MBR partitioning on disks <= 2TB. Use parted for GPT partitioning, disks > 2TB, or when scripting is needed.

Q4: A system fails to boot after modifying /etc/fstab. How do you recover?

Step 1: Boot into emergency/rescue mode (add 'single' or 'emergency' to kernel boot parameters in GRUB, or boot from a live USB).
Step 2: Remount the root file system as read-write: mount -o remount,rw /
Step 3: Edit /etc/fstab and fix or comment out the problematic line: vi /etc/fstab
Step 4: Verify the remaining entries are correct.
Step 5: Reboot.
PREVENTION Always run 'mount -a' after editing /etc/fstab to test entries before rebooting. Keep a backup.

Q5: What is the significance of the dump and fsck fields in /etc/fstab?

Dump field (5th column): Used by the dump command to determine which file systems need to be backed up. 0 = ignore (no dump), 1 = include in dump backup. Most modern systems set this to 0 as dump is rarely used.
FSCK field (6th column): Determines the order in which fsck checks file systems at boot. 0 = skip checking, 1 = check first (should only be used for root /), 2 = check after root. File systems on the same disk are checked sequentially; on different disks, they may be checked in parallel for speed.

Q6: Can you mount a partition to a directory that already contains files? What happens to those files?

Yes, you can mount to a directory with existing files. The existing files are NOT deleted - they are simply hidden (masked) by the mounted file system. When you unmount the partition, the original files become visible again. The original files remain intact on the root (or parent) file system but are inaccessible while another file system is mounted over them.

Q8: What does 'mount -a' do and what does it NOT do?

'mount -a' mounts all file systems listed in /etc/fstab that have the 'auto' mount option (which is included in 'defaults') and are not already mounted. It does NOT: Mount file systems with the 'noauto' option, remount already mounted file systems, mount file systems not listed in /etc/fstab, or process swap entries (swapon -a handles those). It is primarily used to test /etc/fstab changes after editing.

# Step 2: Inform the kernel

# Step 3: Format as swap

# Step 4: Enable swap
swapon /dev/sdb1
# Step 5: Verify
swapon --show
# Step 6: Make permanent in /etc/fstab
UUID=<uuid> none swap sw 0 0
Q10: What is the significance of the 'defaults' mount option in /etc/fstab?

'defaults' is a shorthand that includes the following mount options: rw (read-write), suid (allow set-user-identifier bits), dev (interpret character/block special devices), exec (allow execution of binaries), auto (can be mounted with mount -a), nouser (only root can mount), and async (I/O done asynchronously). These are standard options suitable for most file systems. You can override individual options by specifying them explicitly (e.g., defaults,noexec,nosuid).

Q11: Explain the concept of journaling in file systems like ext4 and XFS.

Journaling is a technique that records pending changes to a log (journal) before committing them to the main file system. This provides crash recovery and faster recovery times (replaying the journal vs full fsck). Three modes exist:
1. journal (data=journal): Both data and metadata are journaled (safest, slowest).
2. ordered (data=ordered): Metadata is journaled; data is written before metadata commit (default, good balance).
3. writeback (data=writeback): Only metadata is journaled; no data ordering guarantee (fastest, least safe).
ext4 defaults to ordered mode. XFS uses metadata-only journaling with write barriers for data integrity.

Q12: You accidentally deleted a partition with fdisk. Can you recover it?

Yes, recovery is often possible if you haven't written new data over the partition's sectors:
Step 1: DO NOT create a new partition or write anything to the disk.
Step 2: Open fdisk on the same disk: fdisk /dev/sdb
Step 3: Create a new partition with the EXACT same starting sector as the deleted one.
Step 4: Use 'n' to create, specify the same start and end sectors.
Step 5: Write with 'w'.
Step 6: If the file system was not overwritten, it should still be intact and mountable.
Alternative tools: testdisk can scan for lost partitions and recover them automatically.
