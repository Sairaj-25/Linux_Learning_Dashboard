🗑️ DELETING PARTITIONS IN LINUX

Safe Unmounting, fstab Cleanup & fdisk Management

PART I: STEP-BY-STEP PROCEDURE - PARTITION DELETION LIFECYCLE

STEP 1: Identify and Unmount

Verify if the partition is mounted using df -h or findmnt. If mounted, unmount it using umount.

Rationale: Deleting a mounted partition will cause data corruption, kernel panics, or immediate system crashes if the system tries to read/write to a suddenly missing block device.

STEP 2: Resolve "Target is Busy"

If umount fails, find processes using the mount point with lsof or fuser and terminate them safely.

Rationale: Linux refuses to unmount filesystems in use to prevent data loss. Forcing without investigation risks corrupting open files.

STEP 3: Clean up /etc/fstab

Remove or comment out the corresponding entry in /etc/fstab.

Rationale: If the entry remains, the system will enter Emergency Mode on the next boot because it expects a device that no longer exists.

STEP 4: Delete via fdisk

Open fdisk /dev/sdX, use the d command, select the partition number, and write changes with w.

Rationale: This modifies the partition table on the disk, marking that space as unallocated.

STEP 5: Inform the Kernel

Run partprobe or udevadm settle to force the kernel to re-read the partition table.

Rationale: The kernel retains the old partition table in memory. Without signaling it, the deleted partition device file (e.g., /dev/sdb1) may persist until reboot.

STEP 6: Verify

Use lsblk or fdisk -l to confirm the partition is gone.

Rationale: Ensures the operation completed successfully before repurposing the freed disk space.

PART II: COMMANDS AND EXPLANATIONS

umount

Syntax: umount /dev/sdXN or umount /mount/point

Purpose: Detaches the filesystem from the directory hierarchy. Must be done before deletion.

umount /dev/sdb1
umount /mnt/data

lsof & fuser

Syntax: lsof +D /mount/point / fuser -vm /mount/point

Purpose: Lists open files (lsof) or processes (fuser) actively using the filesystem, which prevents unmounting.

fuser -vm /mnt/data
lsof +D /mnt/data

fdisk (Deletion)

Syntax: fdisk /dev/sdX

Purpose: Interactive utility to modify partition tables. The d command deletes an entry.

fdisk /dev/sdb
Command (m for help): d
Partition number (1-4): 1
Command (m for help): w

partprobe & udevadm settle

Syntax: partprobe /dev/sdX / udevadm settle

Purpose: Informs the OS kernel of partition table changes, updating device mappings immediately without a reboot.

mount -a

Syntax: mount -a

Purpose: Mounts all filesystems mentioned in /etc/fstab. Used to test fstab syntax after editing; if it fails silently, the config is likely correct.

PART III: ALTERNATIVE COMMANDS & METHODS

1. Unmounting: umount -l (Lazy) vs umount -f (Force)
  • umount -l (Lazy): Detaches the filesystem from the hierarchy immediately, but waits for processes to finish to clean up references. Safe for NFS.
  • umount -f (Force): Forces unmount. Can cause data corruption on local disks if processes are actively writing. Primarily meant for unreachable NFS shares.
2. Partition Deletion Tools: fdisk vs gdisk vs parted
  • fdisk: For MBR partition tables.
  • gdisk: For GPT partition tables.
  • parted: Handles both MBR and GPT. Can be scripted easily (parted /dev/sdb rm 1).
3. Informing the Kernel: partprobe vs partx vs udevadm settle
  • partprobe: Most common. Asks the kernel to re-read the partition table.
  • partx -d /dev/sdb1: Specifically tells the kernel to drop a partition device. Very precise.
  • udevadm settle: Waits for udev event queue to settle after changes. Less about forcing re-reads, more about waiting for the system to catch up.

PART IV: ASSIGNMENT SOLUTIONS

Section A: Short Answer Questions

1. Why check mounted partitions before deleting?

Deleting a mounted partition can cause data corruption, application crashes, and kernel errors because the OS assumes the block device is still available for I/O operations.

2. Command to list mounted partitions?

df -h, findmnt, or mount | grep sdX.

3. What does df -h show?

Displays free and used disk space for all currently mounted filesystems in human-readable format (KB, MB, GB).

4. Purpose of umount?

To logically detach a filesystem from the Linux directory tree, making it inaccessible and safe for modification or removal.

5. Can you delete a mounted partition?

Technically, fdisk might let you, but it is highly dangerous. The kernel will still try to use the now-deleted boundaries, leading to immediate corruption or a system crash. You must unmount first.

6. Error if partition is in use while unmounting?

umount: /mount/point: target is busy (or similar).

7. Command to find processes using a partition?

fuser -vm /mount/point or lsof +D /mount/point.

8. Use of lsof?

Lists Open Files. It identifies which processes have files open on a specific mount point, preventing unmounting.

9. What does fuser -vm do?

Displays the PIDs, users, and commands of processes accessing the specified file or filesystem, with verbose and human-readable output.

10. Role of /etc/fstab?

File System Table. Defines how and where partitions should be automatically mounted during boot.

Section B: Medium Answer Questions

1. Steps to unmount a partition.

1. Verify it is mounted: df -h. 2. Ensure no users/processes are inside it: cd /. 3. Run umount /mount/point. 4. Verify with df -h.

2. Handle "target is busy" error.

1. Run fuser -vm /mount/point to find the PID. 2. Kill the process: kill -9 PID. 3. Try umount again. 4. If desperate and safe, use lazy unmount: umount -l /mount/point.

3. Why remove fstab entries before deletion?

If you delete the partition but leave the fstab entry, the system will try to mount a non-existent device at boot. This drops the system into Emergency Mode/Recovery Shell, halting the boot process.

4. How to test /etc/fstab after editing?

Run sudo mount -a. If it outputs no errors, the syntax and device paths are valid.

5. Role of mount -a?

It reads /etc/fstab and mounts all filesystems defined there that are not currently mounted. It serves as a validation test for fstab changes.

6. Deleting a partition using fdisk.
sudo fdisk /dev/sdb
Command: d       # Delete
Partition #: 1   # Select partition
Command: w       # Write changes
7. partprobe vs udevadm settle?

partprobe actively tells the kernel to re-read the partition table. udevadm settle waits for the udev daemon to finish processing device events after a change, ensuring device files are created/removed properly.

8. Verify partition is deleted?

Run lsblk /dev/sdb or fdisk -l /dev/sdb. The partition should no longer be listed.

Section C: Long Answer Questions

1. Complete step-by-step process of deleting a partition.

Step 1: Identify. lsblk or df -h. Note the device name and mount point.
Step 2: Unmount. umount /dev/sdb1. Handle busy errors with fuser.
Step 3: Edit fstab. Remove the line from /etc/fstab. Run mount -a to test.
Step 4: Delete. fdisk /dev/sdb -> d -> 1 -> w.
Step 5: Update Kernel. partprobe /dev/sdb.
Step 6: Verify. lsblk.

2. Importance of unmounting/fstab removal before deletion.

Unmounting: Prevents data corruption. If deleted while mounted, the kernel might try to write buffered data to sectors that no longer exist, causing I/O errors and potential filesystem damage.
Fstab removal: Prevents boot failure. The local-fs.target systemd unit will wait indefinitely for a missing device, ultimately forcing the system into emergency mode.

3. How Linux handles disk changes after deleting a partition.

The partition table is stored on the disk, while the kernel maintains an in-memory representation. fdisk w updates the disk. partprobe forces the kernel to flush its memory and re-read the disk. The udev daemon detects the removal event and deletes the /dev/sdXN device file.

4. Troubleshooting if partition cannot be unmounted.

1. Check working directory: Ensure your shell isn't inside the mount point.
2. fuser -vm /mount: Find and kill processes.
3. lsof +D /mount: Find open files.
4. Check for nested mounts: Unmount sub-mounts first.
5. Last resort: umount -l (Lazy unmount).

5. Workflow from checking partitions to final verification.

Check (lsblk) -> Unmount (umount) -> Resolve Conflicts (fuser) -> Clean fstab (vi + mount -a) -> Delete (fdisk d) -> Update Kernel (partprobe) -> Verify (lsblk).

Section D: Practical / Command-Based Questions

1. List all mounted partitions.
df -h
2. Unmount by device name and mount point.
umount /dev/sdb1   # By device
umount /mnt/data   # By mount point
3. Check process using partition.
fuser -vm /mnt/data
4. Kill processes using partition.
fuser -km /mnt/data  # -k kills, -m specifies mount
5. Delete partition using fdisk.
fdisk /dev/sdb
d
1
w
6. Open, View, Delete, Save in fdisk.
sudo fdisk /dev/sdb  # Open
p                    # View/Print
d                    # Delete
w                    # Save/Write
7. Inform kernel about changes.
sudo partprobe /dev/sdb
8. Verify deletion.
lsblk /dev/sdb

Section E: Scenario-Based Questions

1. Unable to unmount due to "target is busy". Steps?

1. Use fuser -vm /mount to see the PIDs.
2. If it's a shell, cd / out of the directory.
3. If it's a service, stop it: systemctl stop servicename.
4. If it's a rogue process, kill it: kill -9 PID.
5. Retry umount.

2. Deleted partition still appears. Issue?

The kernel has not updated its in-memory partition table. Run partprobe /dev/sdX to force a re-read.

3. Boot error after deleting partition. Reason?

The deleted partition's entry remains in /etc/fstab. Boot into emergency mode, remount root as read-write (mount -o remount,rw /), edit fstab to remove the entry, and reboot.

4. Forgot to remove fstab entry before deletion. Problem?

System will fail to boot normally. It will drop into emergency mode because systemd cannot find the device defined in fstab.

5. Safely delete partition in production.

1. Ensure backups exist. 2. Schedule downtime if the partition is heavily used. 3. Unmount gracefully. 4. Remove fstab. 5. Delete via fdisk. 6. Run partprobe. 7. Verify with lsblk.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What is the difference between umount -f and umount -l?

-f (Force) immediately unmounts but risks data corruption if writes are pending; used mainly for dead NFS shares. -l (Lazy) marks the filesystem as unmounted immediately so new processes can't access it, but waits for existing processes to release it before cleaning up; much safer for local disks.

Q2: How do you delete a partition non-interactively (for scripting)?
parted /dev/sdb rm 1

This deletes partition 1 on /dev/sdb without entering an interactive menu.

Q3: You deleted a partition but the device file /dev/sdb1 still exists. Why?

udev relies on kernel events. If the kernel hasn't re-read the partition table, it still thinks the partition exists. Run partprobe or use partx -d -n 1 /dev/sdb to explicitly tell the kernel to drop the partition.

Q4: If you delete a swap partition, what must you also do?

Besides unmounting/removing from /etc/fstab, you must also run swapoff /dev/sdXN before deleting it, and remove any resume references in grub or initramfs if it was used for hibernate.

Q5: Explain how to recover if you accidentally delete the wrong partition.

If you haven't run partprobe or formatted over it, you can often recover by running fdisk and recreating the partition with the exact same starting sector as the deleted one. Since the data wasn't overwritten, the filesystem will usually reappear intact.
