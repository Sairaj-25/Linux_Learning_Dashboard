PART I: STEP-BY-STEP PROCEDURE - SECURE FILE TRANSFER

STEP 1: Assess the Transfer Requirement

Determine if you need a quick one-off copy (scp), an interactive browsing session (sftp), or a synchronized backup of changed files (rsync).

Rationale: Using the wrong tool wastes bandwidth. SCP overwrites everything, while RSYNC checks checksums/timestamps to transfer only deltas (incremental changes).

STEP 2: Authentication Setup

Ensure you have SSH credentials or, preferably, SSH keys configured to the remote server.

Rationale: All three tools (SCP, SFTP, RSYNC) tunnel through SSH. Password prompts will interrupt automated scripts, so SSH keys are essential for automation.

STEP 3: Executing the Transfer

Run the command. Always test destructive or large sync operations using a dry-run (e.g., rsync --dry-run) before committing.

Rationale: A typo in source/destination paths can result in data deletion. Dry-runs simulate the transfer without writing data, preventing catastrophic mistakes.

STEP 4: Verification

Verify file integrity using checksums (md5sum or sha256sum) or check file sizes and counts post-transfer.

Rationale: Network interruptions can corrupt files. Verification ensures the bit-stream on the destination exactly matches the source.

PART II: COMMANDS AND EXPLANATIONS

scp (Secure Copy Protocol)

Syntax: scp [options] source destination

Purpose: Fast, non-interactive encrypted copy between hosts over SSH.

Key Options:
-r: Recursively copy entire directories.
-P <port>: Specify a custom SSH port (Note capital P).
-C: Enable compression.
-p: Preserves modification times, access times, and modes.

# Local to Remote
scp file.txt user@remote:/path/

# Remote to Local
scp user@remote:/path/file.txt /local/path/

# Directory recursively
scp -r mydir user@remote:/path/

sftp (Secure File Transfer Protocol)

Syntax: sftp user@remote

Purpose: Interactive file transfer session over SSH. Allows browsing, uploading, and downloading.

Interactive Commands:
put <local_file>: Upload a file to the remote server.
get <remote_file>: Download a file from the remote server.
cd / lcd: Change remote / local directory.
ls / lls: List remote / local files.

sftp user@192.168.1.10
sftp> put report.txt
sftp> get archive.tar.gz
sftp> bye

rsync (Remote Sync)

Syntax: rsync [options] source destination

Purpose: Delta-transfer algorithm copies only the changed parts of files. Ideal for backups and syncing.

Key Options:
-a: Archive mode (equals -rlptgoD). Preserves permissions, owners, groups, symlinks, and recursively copies.
-v: Verbose output.
-z: Compress file data during transfer.
--delete: Delete extraneous files from the destination directory (makes it an exact mirror).
--dry-run: Perform a trial run with no changes made.

# Sync local to remote (using : for ssh)
rsync -avz /local/dir/ user@remote:/remote/dir/

# Sync with custom SSH port
rsync -avz -e 'ssh -p 2222' /local/dir/ user@remote:/remote/dir/

PART III: ALTERNATIVE COMMANDS & METHODS

1. scp vs rsync
  • scp: Brute force. Copies the whole file every time, regardless of whether it exists on the target. Good for quick, one-off small file transfers. Easier syntax for simple copies.
  • rsync: Intelligent. Checks file sizes and timestamps. If a 1GB file has 1MB of changes, rsync only transfers the 1MB delta. Much faster for repeated transfers and backups.
2. tar + ssh (The Streaming Alternative)

If you need to move a massive directory tree and don't care about resuming or delta-syncing, piping tar over ssh is sometimes faster than rsync because rsync has to calculate checksums before transferring.

# Archives locally, streams over SSH, extracts remotely
tar -czf - /local/dir | ssh user@remote 'tar -xzf - -C /remote/dir/'
3. wget / curl vs sftp

wget and curl are for HTTP/HTTPS/FTP downloads. While FTP supports upload (curl -T), it is unencrypted. sftp should always be used for interactive secure file management over untrusted networks.

Section B: Short Answer Questions

1. Define SCP.

SCP (Secure Copy Protocol) is a network protocol used to securely copy files and directories between hosts over an SSH connection.

2. Purpose of SFTP?

SFTP provides an interactive, secure command-line session to browse, upload, download, and manage files on a remote server, similar to FTP but entirely encrypted via SSH.

3. Explain the use of RSYNC.

RSYNC is used for efficiently synchronizing files and directories between two locations. Its "delta-transfer algorithm" ensures only the modified parts of files are transferred, minimizing network bandwidth usage.

4. Basic syntax of SCP.
scp [options] source_path destination_path
# e.g., scp file.txt user@host:/path/
5. What does the -r option do in SCP?

It enables recursive copying, allowing you to copy entire directories and their contents, rather than just individual files.

6. Purpose of the put command in SFTP?

The put command uploads a file from the local machine to the remote server during an active SFTP session.

7. Explain archive mode in RSYNC.

The -a (archive) flag is a bundle option equivalent to -rlptgoD. It recursively copies files while preserving symlinks, permissions, timestamps, groups, and owners, ensuring an exact replica of the source data.

8. What is incremental backup?

An incremental backup only saves the data that has changed since the last backup. RSYNC does this natively by comparing file sizes and modification timestamps, avoiding the need to copy unchanged files.

9. Why is RSYNC faster than SCP?

RSYNC uses a delta-transfer algorithm. If a 1GB file has 10MB of changes, SCP re-sends the entire 1GB, while RSYNC calculates the 10MB difference and only sends that delta. RSYNC can also compress data on the fly.

10. List any two common RSYNC options.

-a (Archive mode) and -z (Compress data during transfer).

Section C: Command Writing

1. Copy file.txt local to remote.
scp file.txt user@remote_ip:/path/to/destination/
2. Copy file remote to local.
scp user@remote_ip:/path/to/file.txt /local/destination/
3. Recursively copy directory.
scp -r /local/mydir user@remote_ip:/remote/path/
4. Connect to remote server using SFTP.
sftp user@remote_ip
5. Upload file using SFTP.
sftp> put local_file.txt
6. Download file using SFTP.
sftp> get remote_file.txt
7. Sync local folder to remote.
rsync -avz /local/folder/ user@remote_ip:/remote/folder/

Note: The trailing slash on the source folder is crucial in rsync.

8. Sync remote folder to local.
rsync -avz user@remote_ip:/remote/folder/ /local/folder/
9. Daily backup using RSYNC.
# Archive, verbose, compress, and delete files that no longer exist in source
rsync -avz --delete /home/user/data/ backup_server:/backup/daily/
10. Copy files locally using RSYNC.
rsync -avz /source/directory/ /destination/directory/

Section E: Scenario-Based Questions

1. Securely copy project.zip to remote Linux server.
scp project.zip admin@192.168.1.50:/home/admin/projects/
2. Create daily incremental backups of /home to backup server.
# Add this to a cron job
rsync -avz --delete /home/ backup_user@backup-server:/backups/daily_home/

sftp> put report.txt
4. Sync files between servers, transferring only changed data. Which tool and why?

Tool: RSYNC.
Why: RSYNC's delta-transfer algorithm compares the source and destination files. It only transmits the blocks of data that have actually changed, drastically reducing bandwidth and time compared to SCP, which would re-copy the entire file.

5. Interactive secure file transfer session to manage remote files.

Use sftp user@server. It allows you to navigate the remote file system (cd, ls), upload (put), download (get), and delete files interactively, which SCP cannot do.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What is the significance of the trailing slash in RSYNC source paths?

This is a classic RSYNC gotcha.
- rsync -avz /source/ /dest/: Copies the contents of the source directory into dest.
- rsync -avz /source /dest/: Copies the source directory itself into dest (resulting in /dest/source/).
Always be explicit with trailing slashes to avoid nested directory mistakes.

Q2: How does RSYNC's delta-transfer algorithm actually work?

When RSYNC finds a file that exists on both sides but differs in size/timestamp, it splits the source file into fixed-size blocks, calculates a rolling checksum (adler32) and a strong checksum (md5/sha1) for each block, and sends these to the receiver. The receiver calculates the same checksums on the existing destination file. By comparing the lists, the receiver can identify which blocks are missing and requests only those specific chunks, which the sender then transmits.

Q3: Why did OpenSSH 9.0 change the default behavior of the scp command?

Historically, scp used the legacy SCP protocol (rcp protocol over SSH), which is insecure and cannot properly handle filenames with special characters. As of OpenSSH 9.0, scp now uses the SFTP protocol by default, which is more robust and predictable. You can revert to the legacy protocol with -O, but it is discouraged.

Q4: What is the --delete flag in RSYNC, and when should you avoid it?

--delete tells RSYNC to remove files on the destination that do not exist on the source, creating an exact mirror.
Avoid it when: You are using the destination as a backup archive where deleted source files should be retained. In such cases, you should use backup options like --backup and --backup-dir to move deleted/changed files to a separate directory before overwriting/deleting them.

Q5: How do you resume an interrupted file transfer?

- RSYNC: Simply run the exact same command again. RSYNC will check the partially transferred file, calculate what's missing, and resume where it left off.
- SFTP: Use the reget (for downloading) or reput (for uploading) commands to append to existing partial files.
- SCP: Cannot resume. You must restart the transfer from the beginning.
