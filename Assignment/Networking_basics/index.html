PART I: STEP-BY-STEP PROCEDURE - NETWORK TROUBLESHOOTING

STEP 1: Interface Verification

Check if the network interface is up and has an IP address using ip addr or ifconfig.

Rationale: If the interface is DOWN or lacks an IP, no network communication is possible. This is the physical/data link layer check.

STEP 2: Local Stack Verification

Ping the loopback address: ping -c4 127.0.0.1.

Rationale: This verifies that the local TCP/IP stack is initialized and functioning correctly. If this fails, the OS networking is broken.

STEP 3: Gateway & Route Verification

Check the routing table with ip route and ping the default gateway.

Rationale: A missing default route means the system doesn't know where to send packets destined for outside the local subnet.

STEP 4: External Connectivity & DNS

Ping an external IP (e.g., ping 8.8.8.8). If successful, test DNS (ping google.com).

Rationale: If the IP ping works but the name ping fails, the issue is DNS, not network routing.

STEP 5: Port & Service Verification

Use ss -tlupn to check if the specific service port is listening.

Rationale: Even if the network is up, a service won't accept connections if its daemon isn't bound to the port.

PART II: COMMANDS AND EXPLANATIONS

ip addr / ifconfig

Syntax: ip addr show or ifconfig

Purpose: Displays or manages IP addresses, MAC addresses, and interface states (UP/DOWN).

ip addr show eth0

ping

Syntax: ping [options] destination

Purpose: Sends ICMP Echo Request messages to test network reachability and measure round-trip time.

ping -c 4 192.168.1.1  # Send 4 packets only

ip route

Syntax: ip route show

Purpose: Displays or manipulates the kernel's routing table, showing how packets are directed to different networks.

ip route

traceroute / tracepath

Syntax: traceroute host or tracepath host

Purpose: Tracks the path packets take across network hops to reach a destination, identifying where delays or blocks occur.

traceroute www.google.com

ss (Socket Statistics)

Syntax: ss [options]

Purpose: Dumps socket statistics, showing network connections, listening ports, and socket states. Modern replacement for netstat.

ss -tlupn
# -t: TCP, -l: Listening, -u: UDP, -p: Process, -n: No DNS resolution

grep

Syntax: command | grep pattern

Purpose: Filters text output to show only lines matching a specific pattern. Crucial for parsing verbose network outputs.

ss -tlupn | grep :22

PART III: ALTERNATIVE COMMANDS & METHODS

1. iproute2 vs net-tools (The Legacy Shift)
  • net-tools (ifconfig, netstat, route): Deprecated since 2001. Doesn't support modern Linux networking features like multiple IPs per interface or advanced routing.
  • iproute2 (ip, ss, ip route): The modern standard. Communicates directly with the kernel Netlink interface, making it vastly faster and more feature-rich.
2. traceroute vs tracepath
  • traceroute: Traditionally uses UDP packets or ICMP. Often requires root privileges to send raw packets. Can be blocked by firewalls dropping UDP high ports.
  • tracepath: Uses UDP but does not require root privileges. It also automatically discovers MTU (Maximum Transmission Unit) along the path. Preferred for unprivileged users.
3. Checking Ports: ss vs lsof

ss is faster for viewing socket states. lsof -i :22 is an alternative that lists open files (including network sockets) and provides very detailed process ownership info, but runs slower on servers with many open files.

PART IV: ASSIGNMENT SOLUTIONS

Section A: Objective Questions

1. Purpose of ifconfig?

To display or configure network interface parameters (IP address, netmask, MAC, status).

2. Command for IP info in modern Linux?

ip addr (or ip a).

3. What does 127.0.0.1 represent?

The loopback address. It represents the local machine itself, used for internal testing and inter-process communication.

4. Command to test network connectivity?

ping.

5. Function of ip route?

To display or modify the kernel's IP routing table.

6. Command to trace route to remote host?

traceroute or tracepath.

7. What does ss display?

Socket statistics (network connections, listening ports, socket states).

8. Traditional command for network connections?

netstat.

9. Default SSH port?

Port 22.

10. Command to filter output using patterns?

grep.

Section B: Short Answer Questions

1. Difference between ifconfig and ip addr?

ifconfig is an older, deprecated tool that only shows basic IP info and requires raw socket manipulation. ip addr is part of the modern iproute2 suite, uses Netlink to talk to the kernel, and supports advanced features like multiple IPs per interface and CIDR notation.

2. Use of ping -c4 127.0.0.1?

It sends exactly 4 ICMP echo requests to the local loopback interface. It verifies that the local TCP/IP network stack is installed, configured correctly, and functioning without relying on physical network hardware.

3. How to display routing info?

Use ip route show (modern) or route -n (legacy). The -n flag in route prevents DNS lookups, displaying IPs instead of hostnames for faster output.

4. Difference between tracepath and traceroute?

traceroute often requires root privileges to send raw ICMP/UDP packets and allows choosing packet types. tracepath does not require root, uses standard UDP, and automatically discovers MTU (Path MTU Discovery) along the route.

5. Explain ss -tlupna?

A comprehensive socket dump:
- -t: TCP sockets only
- -l: Listening sockets only
- -u: UDP sockets only
- -p: Show process using the socket
- -n: Don't resolve service names (show port numbers)
- -a: Show all sockets (both listening and non-listening)

6. Check if a specific port is listening using ss?

Use ss -tlpn | grep :<port>. For example, ss -tlpn | grep :80 checks if HTTP is listening.

7. Purpose of using grep with commands?

Network commands like ss or netstat output hundreds of lines. grep filters this output to display only lines containing the specific IP, port, or state you are troubleshooting, drastically reducing noise.

8. Role of netstat in troubleshooting?

netstat provides a snapshot of network connections, routing tables, interface statistics, and masquerade connections. It helps identify which ports are open, which services are connected to what, and if there are abnormal connection states (e.g., many SYN_RECV indicating a SYN flood attack).

Section D: Scenario-Based Questions

1. Unable to connect to internet. Step-by-step troubleshooting?

1. Check Interface: ip addr - Is the interface UP and has an IP?
2. Check Local Stack: ping 127.0.0.1 - Is TCP/IP working?
3. Check Gateway: ip route to find gateway, then ping <gateway> - Can I leave my network?
4. Check External IP: ping 8.8.8.8 - Is the internet reachable?
5. Check DNS: ping google.com - Is name resolution working?
6. Trace Path: tracepath 8.8.8.8 - Where exactly do the packets drop?

2. Server not accepting SSH. Verify if port 22 is active?
# Check if sshd is listening
ss -tlpn | grep :22

# If it doesn't show up, check the service status
sudo systemctl status sshd

# Verify firewall isn't blocking it
sudo ufw status | grep 22
3. Identify all active network connections.

Use ss -tunap.
- -t (TCP) and -u (UDP) cover the main protocols.
- -n prevents slow DNS lookups.
- -a shows both listening and established connections.
- -p shows the process PID/name, which is critical for knowing *what* is making the connection.

4. Cannot reach external website. How does traceroute help?

traceroute or tracepath sends packets with incrementing TTLs. Each router along the path decrements the TTL and sends an ICMP Time Exceeded message back when TTL hits 0. This maps the exact hop-by-hop path. If the trace stops at hop 3, you know the failure is between your 3rd router and the 4th. If it reaches the destination but with high latency at a specific hop, that router is the bottleneck.

5. Compare ss and netstat. Which is more efficient?

ss is vastly more efficient. netstat gathers info by reading /proc/net/tcp and iterates over all /proc/<pid>/fd directories to match processes, making it extremely slow on servers with many processes. ss uses the kernel's Netlink interface, pulling socket and process data directly from kernel memory in a single query. It is orders of magnitude faster.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: Why does ping sometimes fail even if the server is online and serving web traffic?

Many servers and cloud providers (like AWS) disable ICMP traffic (which ping uses) at the firewall level to prevent network discovery and DDoS attacks. HTTP/HTTPS traffic works because it uses TCP on ports 80/443, which are explicitly allowed, while ICMP is dropped.

Q2: What is the significance of the "State" column in ss output? What does ESTAB vs TIME-WAIT mean?

- ESTAB: The TCP 3-way handshake is complete, and data can flow freely.
- TIME-WAIT: The side that closed the connection waits for a period (typically 2*MSL) to ensure the remote end received the ACK and that delayed packets are flushed. Too many TIME-WAIT sockets can exhaust ephemeral ports.

Q3: What does the 0.0.0.0 address mean in the ss -tlpn output?

When a service listens on 0.0.0.0, it means it is accepting connections on all available IPv4 network interfaces. If it listens on 127.0.0.1, it only accepts local connections. For security, services not needing external access should be bound to 127.0.0.1.

Q4: How would you find which process is consuming the most network bandwidth?

ss and netstat only show connections, not bandwidth. To find bandwidth usage per process, use tools like nethogs (shows per-process bandwidth in real-time) or iftop (shows bandwidth per connection).

Q5: What is the difference between a routing table and an ARP table?

- Routing Table (ip route): Maps destination IP networks to the next hop (gateway) or interface. It operates at Layer 3 (Network Layer).
- ARP Table (ip neigh or arp -a): Maps IP addresses to MAC addresses on the local network. It operates at Layer 2 (Data Link Layer) to ensure frames are sent to the correct physical device on the local subnet.
