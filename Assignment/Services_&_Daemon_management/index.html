PART I: STEP-BY-STEP PROCEDURE - SERVICE LIFECYCLE

STEP 1: Installation & Verification

Install the service package and verify the unit file is loaded into systemd.

Rationale: Before managing a service, the systemd unit file (usually in /usr/lib/systemd/system/) must exist so systemd knows how to start it.

STEP 2: Starting & Status Check

Start the service in the current session using systemctl start and verify its state with systemctl status.

Rationale: Starting a service only affects the current runtime. If it fails to start, checking the status or journalctl -u <service> provides diagnostic logs.

STEP 3: Enabling at Boot

Use systemctl enable to create the symlink in the .wants directory.

Rationale: Starting a service doesn't make it persistent across reboots. Enabling it tells systemd to start it automatically during the boot process.

STEP 4: Configuration Changes

If you modify configuration files, use systemctl reload (if supported) or systemctl restart.

Rationale: reload applies new config without dropping connections (zero downtime). restart kills the process and starts it fresh, dropping active connections.

STEP 5: Hardening (Masking)

If a service is a security risk or constantly starts when unwanted, use systemctl mask.

Rationale: disable only removes the boot symlink. A rogue process or admin can still start it manually. mask links the service to /dev/null, making it impossible to start manually or automatically.

PART II: COMMANDS AND EXPLANATIONS

systemctl status

Syntax: systemctl status <service>

Purpose: The swiss army knife. Shows if the service is active/enabled, the PID, memory usage, cgroup info, and the last few journal log lines.

systemctl start / stop / restart

Purpose: Alters the runtime state immediately. start activates, stop deactivates, restart stops then starts.

systemctl reload

Purpose: Asks the service to re-read its configuration files without stopping the process. Best Practice for zero-downtime config updates.

systemctl enable / disable

Purpose: Alters the boot state. Creates or removes symlinks in the multi-user.target.wants directory.

# Enable AND start in one command
sudo systemctl enable --now sshd

systemctl mask / unmask

Purpose: Prevents a service from being started manually or automatically by linking it to /dev/null. Security Feature

systemctl list-units

Purpose: Lists units (services, sockets, targets) that systemd has loaded into memory.

systemctl list-units --type=service --all

PART III: ALTERNATIVE COMMANDS & METHODS

1. systemctl vs Legacy service / chkconfig
  • systemctl: The modern standard for systemd. Manages services, targets (runlevels), sockets, and timers natively.
  • service: Legacy SysVinit wrapper. Still exists on modern systems as a compatibility script that translates service sshd restart to systemctl restart sshd.
  • chkconfig: Legacy tool for enabling/disabling runlevels. Replaced entirely by systemctl enable/disable.
2. restart vs reload
  • restart: Kills the main PID and starts a fresh process. Causes temporary downtime. Required for major config changes (like changing SSH port).
  • reload: Sends SIGHUP to the main process, asking it to re-read config files. No downtime. Only works if the daemon supports it (e.g., Nginx, httpd). Not all services support reload.
3. Checking Logs: systemctl status vs journalctl

systemctl status shows the last 10 lines of logs. For deep troubleshooting, use journalctl -u <service-name> to see the full historical log for that specific unit.

PART IV: ASSIGNMENT SOLUTIONS

Section A: Short Answer Questions

1. What is systemd in Linux?

systemd is the default init system and service manager in modern Linux distributions. It starts and manages daemons, mounts filesystems, and handles system states (targets) efficiently.

2. What is a daemon? Give two examples.

A daemon is a background process that runs continuously, waiting for events or requests to handle. Examples: sshd (SSH server), httpd (Web server).

3. Role of PID 1 in Linux?

PID 1 is the very first process started by the kernel. It is responsible for starting all other userspace processes and adopting orphaned processes. In modern systems, this is systemd.

4. Difference between systemd and initd.

initd (SysVinit) starts services sequentially using shell scripts, making boot slow. systemd starts services in parallel, uses socket activation, manages dependencies natively, and tracks processes via cgroups.

5. Purpose of systemctl command?

To inspect and control the state of the systemd system and service manager (e.g., start, stop, enable services).

6. What does LOAD column represent?

It indicates whether the unit configuration file was successfully loaded into memory by systemd. Values are 'loaded', 'not-found', or 'masked'.

7. Meaning of active (running) state?

It means the service is currently active and its main process is continuously running in the background.

8. What does enabled mean in service status?

It means the service is configured to start automatically during the system boot process via a symlink in the .wants directory.

Section D: Long Answer Questions

1. Explain key features of systemd.

- Parallel Startup: Starts independent services concurrently.
- Socket Activation: Listens on sockets and starts the daemon on-demand, reducing boot time.
- cgroups: Tracks processes accurately, preventing daemon leaks.
- Targets: Replaces runlevels with flexible system states.
- Timers: Replaces cron with robust, dependency-aware scheduling.

2. Describe different service states in systemctl output.

- active (running): Service is running continuously.
- active (exited): Service ran successfully and exited (common for oneshot config scripts).
- active (waiting): Service is active but waiting for an event.
- inactive: Service is stopped.
- failed: Service crashed or failed to start.

3. Explain columns in systemctl list-units.

- UNIT: The name of the unit (e.g., sshd.service).
- LOAD: Whether the config file was loaded into memory.
- ACTIVE: The high-level state (active, inactive, failed).
- SUB: The low-level state (running, exited, dead).
- JOB: Pending jobs for the unit.
- DESCRIPTION: Human-readable description of the unit.

4. Differences: start vs restart, enable vs disable
CommandActionRuntime vs Boot
startBegins the service nowRuntime only
restartStops then starts the serviceRuntime only

Tip: Use sudo systemctl enable --now sshd to enable AND start simultaneously.

5. How systemd improves boot performance.

systemd uses parallel initialization. Unlike SysVinit, which waited for each script to finish, systemd starts independent services simultaneously. It also uses socket activation (inetd-style), where systemd listens on ports and only starts the heavy daemon when a connection arrives, saving resources at boot. systemd-analyze blame shows exactly what slows down boot.

Section F: Scenario-Based Questions

1. SSH not running. Troubleshooting steps?

1. Check status: systemctl status sshd (Look for failed status/error logs).
2. Verify config: sshd -t (Tests syntax of sshd_config).
3. Check logs: journalctl -u sshd -n 50.
4. Check socket/port: ss -tulpn | grep 22.

2. Service shouldn't start at boot. Command?
sudo systemctl disable <service-name>

If it must NEVER run: sudo systemctl mask <service-name>

3. Updated config, don't want to stop service. What to do?

Use sudo systemctl reload <service-name>. This sends a signal to the daemon to re-read its configuration without terminating the process, maintaining active connections.

4. Service completely blocked from starting. Reason?

The service is likely masked. systemctl mask links the service file to /dev/null, preventing it from starting even with sudo systemctl start. Check with systemctl status <service> (will show 'masked').

Section G: Bonus (Critical Thinking)

1. Why are daemons important in Linux systems?

Daemons provide continuous, unattended background services. They listen for network requests (sshd, httpd), schedule tasks (crond), manage hardware (udev), and log events (rsyslogd). Without daemons, Linux could not function as a server or handle background automation.

2. How does systemd handle dependencies better than init?

SysVinit used sequential scripts (S01, S02) with no true dependency resolution—it just guessed via numbering. systemd uses declarative After=, Requires=, and Wants= directives in unit files. This allows systemd to build a dependency graph and start independent services in parallel, vastly improving boot time and reliability.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What is the difference between a .service and a .socket unit in systemd?

A .service defines how to start and manage a daemon process. A .socket defines an IPC or network socket. systemd can listen on the socket and only start the associated .service when a connection arrives (Socket Activation). This saves resources at boot.

Q2: What is the difference between Requires= and Wants= in a unit file?

- Requires=: Hard dependency. If the required unit fails, this unit fails too.
- Wants=: Soft dependency. If the wanted unit fails, this unit still starts. Wants= is used for enabling services at boot (symlinks in .wants directories).

Q3: How do you change the default target (runlevel) in systemd?
# View default
systemctl get-default

# Set default to multi-user (runlevel 3)
sudo systemctl set-default multi-user.target

# Set default to graphical (runlevel 5)
sudo systemctl set-default graphical.target
Q4: A service is in a 'failed' state. How do you reset it and view the error?
# View detailed error
systemctl status <service>
journalctl -xe

# Reset the failed state (clears the error flag)
sudo systemctl reset-failed <service>
Q5: What are systemd targets? How do they differ from SysVinit runlevels?

Targets are synchronization points in systemd. While SysVinit had rigid 0-6 runlevels, systemd targets are flexible groups of units.
- runlevel 3 ≈ multi-user.target
- runlevel 5 ≈ graphical.target
- runlevel 1 ≈ rescue.target
Unlike runlevels, multiple targets can be active simultaneously, and you can isolate targets to switch system states dynamically.
