PART I: STEP-BY-STEP PROCEDURE - MANAGING PROCESS PRIORITY

STEP 1: Identify the Process Constraint

Use top or htop to identify CPU-hogging processes. Look at the %CPU and NI (Nice) columns.

Rationale: Before adjusting priorities, you must identify which process is starving other critical services of CPU time.

STEP 2: Starting New Processes with nice

When launching a non-urgent, resource-intensive task (like a backup or compilation), start it with a higher nice value (e.g., 10 or 19): nice -n 10 ./backup.sh.

Rationale: By making the process "nicer" to others, the Linux scheduler will only allocate CPU time to it when no other normal-priority processes need it, keeping the system responsive.

STEP 3: Adjusting Running Processes with renice

If a running process is causing lag, alter its priority on the fly using its PID: renice -n 5 -p 1234. For critical DBs needing a boost (as root): renice -n -5 -p 5678.

Rationale: It avoids the downtime of stopping and restarting a long-running service just to change its priority.

STEP 4: Verification

Verify the changes by checking the NI column in top or using ps -o pid,ni,cmd -p <PID>.

Rationale: Ensures the scheduling adjustment was successfully applied by the kernel.

PART II: COMMANDS AND EXPLANATIONS

nice

Syntax: nice -n

Purpose: Launches a new process with a specified niceness (scheduling priority). Default is 0 if omitted. Higher values = lower priority.

Example: nice -n 10 tar -czf backup.tar.gz /data (Starts tar with low priority)

renice

Syntax: renice -n -p

Purpose: Changes the nice value of an already running process.

Example: renice -n -5 -p 1234 (Increases priority of PID 1234. Requires root.)

Viewing Priority

Syntax: ps -eo pid,ni,cmd or top

Purpose: Displays the NI (Nice) value alongside process IDs and commands. In top, look for columns NI (Nice) and PR (Priority).

PART III: ALTERNATIVE COMMANDS & METHODS

1. htop for Interactive Priority Management

While renice is CLI-based, htop provides a visual way. You can press F7 (Nice -) to increase priority and F8 (Nice +) to decrease priority on the selected process. This is much more user-friendly for system administrators monitoring systems in real-time.

2. ionice vs nice

nice only affects CPU scheduling. A process with a high nice value can still freeze the system if it hogs Disk I/O. ionice adjusts the I/O scheduling priority. For heavy backup jobs, it's best practice to combine both: nice -n 19 ionice -c2 -n7 rsync -a /src /dest. This makes the process polite to both the CPU and the hard drive.

3. chrt (Advanced Scheduling)

nice modifies processes in the default scheduling policy (SCHED_OTHER). chrt allows you to change the scheduling policy entirely (e.g., to Real-Time policies like SCHED_FIFO or SCHED_RR). This is for highly specialized, latency-sensitive applications and is dangerous if misused.

Section D: Short Answer Questions

1. What is process priority in Linux?

It is a numeric value the kernel uses to determine how much CPU time a process receives relative to other processes. In user space, this is managed via "Nice" values, where a lower number means higher priority for CPU allocation.

2. Explain the purpose of the nice command.

The nice command is used to launch a program with a modified scheduling priority. It allows administrators to start non-urgent background tasks (like backups) so they don't consume CPU time needed by critical foreground services.

3. Explain the purpose of the renice command.

The renice command alters the scheduling priority of an already running process. This is useful for deprioritizing a runaway process that is consuming too many resources, or boosting the priority of a critical database process, without needing to restart them.

4. What is the default nice value?

0. Processes inherit the nice value of their parent, and standard terminal sessions start with a nice value of 0.

5. Why are negative nice values restricted to root user?

If normal users could assign negative nice values (high priority), they could monopolize the CPU, making the system unresponsive for other users and essential system processes. Restricting this to root ensures system stability.

Section F: Scenario-Based Questions

1. A backup job should not affect system performance. Which command should be used?

Start the backup with nice -n 19 to give it the lowest CPU priority. Also consider ionice -c2 -n7 to prevent disk I/O starvation.

2. A database process needs more CPU priority. Which command is suitable?

Use sudo renice -n -5 -p <PID_of_DB> to give it a negative nice value, increasing its CPU share over standard processes.

3. A heavy process is slowing down the system. How can its priority be reduced?

Find its PID using top, then use renice -n 19 -p <PID>. As a normal user, you can only renice your own processes to higher values.

4. Why should administrators avoid assigning very high priority to many processes?

Priorities are relative. If every process is given the highest priority (e.g., -20), then they all compete equally for CPU, effectively reverting to the default state. Worse, high-priority user processes can starve essential kernel threads, causing system hangs.

5. Which command helps monitor priority values in real time?

top or htop.

Section H: Long Answer Questions

1. Explain the nice command with examples.

The nice command launches a process with a custom scheduling priority (nice value), allowing administrators to control how much CPU time a new process gets relative to others. The range is -20 (highest priority) to 19 (lowest priority). Default is 0. Normal users can only set values 1-19, while root can set -20 to 19.
Examples:
1. Start a low-priority compilation: nice -n 10 make
2. Start an urgent data crunching task (as root): nice -n -15 ./data_crunch.bin

2. Explain the renice command with examples.

The renice command alters the priority of an already running process. This is essential for system tuning without service interruption. It uses the -p flag for PIDs, -u for users, and -g for groups.
Examples:
1. Lower priority of PID 1234: renice -n 15 -p 1234
2. Increase priority of PID 5678 (as root): renice -n -10 -p 5678
3. Change priority of all processes owned by user 'admin': renice -n 5 -u admin

4. Explain how Linux process scheduling uses nice values.

Linux uses the Completely Fair Scheduler (CFS). Instead of allocating fixed time slices, CFS assigns a proportion of CPU time. The "nice" value acts as a weight. Every +1 increment in nice value decreases the CPU proportion by roughly 10%, and every -1 increment increases it. For example, a process with a nice value of 0 gets significantly more CPU time than a process with a nice value of 19. CFS ensures that even the lowest priority process will eventually get some CPU time (no starvation), unless higher-priority real-time processes are continuously runnable.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What is the difference between the PR and NI columns in the top command?

NI (Nice): The user-space priority value (-20 to 19) set by the user via nice or renice.
PR (Priority): The kernel-space scheduling priority. For standard processes (SCHED_OTHER), the formula is PR = 20 + NI. Therefore, a nice value of -20 equals PR 0 (highest), and a nice value of 19 equals PR 39 (lowest). Real-time processes will show PR values like 'rt'.

Q2: Can a normal user renice their own process from 5 to 0? Why or why not?

No. A normal user can only increase the nice value (make the process "nicer" / lower priority). Moving from 5 to 0 is decreasing the nice value (increasing priority), which is strictly a root-only operation. This prevents users from gaming the scheduler to the detriment of others.

Q3: If you renice a multi-threaded process, does it affect all threads?

Yes. In modern Linux, if you change the nice value of a process (via its main PID), all threads within that process (visible via ps -L or top -H) inherit the new nice value. You can theoretically set different nice values for individual threads, but this is rarely done and requires targeting specific Thread IDs (TIDs).

Q4: What happens if two processes with nice values of -20 are fighting for the CPU?

Since both have the highest possible user-space priority, the CFS scheduler will split the CPU time exactly 50/50 between them. The nice value determines the *ratio* of CPU allocation, not absolute exclusivity. They will still starve normal (nice 0) processes, however.

Q5: Why is it generally recommended to use ionice alongside nice for backup jobs?

nice only makes the process polite about CPU time. A backup job that is CPU-nice (nice 19) might still aggressively read from the disk, causing the disk controller to starve interactive processes (like a web server or desktop UI) of I/O. ionice sets the I/O priority (Best-effort, Real-time, Idle), ensuring the backup does not saturate the disk bandwidth.
