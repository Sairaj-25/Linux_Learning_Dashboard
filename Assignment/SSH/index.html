PART I: STEP-BY-STEP PROCEDURE - SSH LIFECYCLE

STEP 1: Installation & Service Startup

Install the OpenSSH server package and start the sshd daemon. Enable it to start on boot.

Rationale: The SSH server must be running to accept incoming connections. Enabling it via systemd ensures the server remains accessible after a reboot.

STEP 2: Key Pair Generation

On the client machine, generate a public/private key pair using ssh-keygen.

Rationale: Key-based authentication is vastly more secure than passwords. The private key never leaves the client, while the public key is placed on the server.

STEP 3: Key Distribution

Copy the public key to the remote server using ssh-copy-id.

Rationale: This appends the public key to the server's ~/.ssh/authorized_keys file, establishing the trust relationship required for passwordless login.

STEP 4: Security Hardening

Edit /etc/ssh/sshd_config to disable root login, change the default port, and disable password authentication.

Rationale: Default configurations are primary targets for brute-force bots. Disabling root login limits damage if compromised, and changing the port reduces noise in logs.

STEP 5: Service Restart & Validation

Restart sshd to apply changes. Verify syntax before restarting to avoid lockout.

Rationale: Configuration changes are only read at service startup. A syntax error in sshd_config can cause the service to fail to start, locking everyone out.

PART II: COMMANDS AND EXPLANATIONS

ssh (Client)

Syntax: ssh user@hostname

Purpose: Initiates an encrypted remote login session to a server running the SSH daemon.

# Connect using default port (22)
ssh admin@192.168.1.10

# Connect using a custom port
ssh -p 2222 admin@192.168.1.10

ssh-keygen

Syntax: ssh-keygen -t <algorithm> -b <bits>

Purpose: Generates authentication key pairs (public and private). The private key stays local; the public key goes to remote hosts.

# Generate RSA 4096-bit key
ssh-keygen -t rsa -b 4096

# Generate Ed25519 key (Modern, Recommended)
ssh-keygen -t ed25519

ssh-copy-id

Syntax: ssh-copy-id user@hostname

Purpose: Automatically appends the client's public key to the server's ~/.ssh/authorized_keys file.

ssh-copy-id admin@192.168.1.10

systemctl (Service Management)

Syntax: systemctl [action] sshd

Purpose: Manages the SSH daemon service state.

sudo systemctl start sshd
sudo systemctl enable sshd
sudo systemctl status sshd
sudo systemctl restart sshd

PART III: ALTERNATIVE COMMANDS & METHODS

1. ssh-keygen Algorithms: RSA vs Ed25519
  • RSA: Legacy standard. Requires large key sizes (2048 or 4096-bit) for security. Slower during the handshake.
  • Ed25519: Modern Elliptic Curve algorithm. Much shorter keys, faster authentication, and more secure against side-channel attacks. Recommended for all new deployments.
2. Distributing Keys without ssh-copy-id

If ssh-copy-id is unavailable, you can pipe the key via SSH manually:

cat ~/.ssh/id_ed25519.pub | ssh user@host "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
3. Checking Ports: ss vs netstat

netstat is considered deprecated on modern Linux. ss (socket statistics) is faster and pulls data directly from the kernel.

# Using ss (Modern)
ss -tulpn | grep 22

# Using netstat (Legacy)
netstat -tulpn | grep 22

PART IV: ASSIGNMENT SOLUTIONS

Section A: Short Answer Questions

1. What is SSH?

SSH (Secure Shell) is a cryptographic network protocol used for secure data communication, remote command-line login, and remote command execution over an unsecured network.

2. Default port number used by SSH?

Port 22.

3. Purpose of the SSH protocol?

To provide a secure, encrypted channel over an insecure network, preventing eavesdropping, password sniffing, and data manipulation (Man-in-the-Middle attacks).

4. Difference between Telnet and SSH?

Telnet transmits data (including passwords) in plain text, making it highly insecure. SSH encrypts all traffic, ensuring confidentiality and integrity.

5. Which package provides SSH service?

openssh-server (Server) and openssh-clients (Client).

6. Purpose of the sshd service?

It is the SSH daemon that runs on the server, listening for incoming SSH client connections and authenticating users.

7. File containing SSH server config?

/etc/ssh/sshd_config

8. Command to check SSH service status?
sudo systemctl status sshd
9. Use of SSH key authentication?

To provide a more secure and convenient alternative to passwords. It uses asymmetric cryptography, making it nearly immune to brute-force attacks.

10. Purpose of known_hosts?

Stores the host keys of remote servers. It verifies the server's identity on subsequent connections, protecting against Man-in-the-Middle (MITM) attacks.

11. Purpose of authorized_keys?

Located in ~/.ssh/, it stores the public keys of clients allowed to log into the server without a password.

12. What is passwordless SSH login?

Authenticating to a remote server using public/private key pairs instead of a password, streamlining automation and increasing security.

Section C: Practical Tasks

1. Install and configure SSH server.
sudo apt update && sudo apt install openssh-server
sudo systemctl start sshd
sudo systemctl enable sshd
2. Start and enable SSH service.
sudo systemctl enable --now sshd

Tip: The --now flag starts and enables in one command.

3. Verify SSH connectivity.
ssh user@remote_ip
# Successful login confirms connectivity
4. Connect to remote server.
ssh admin@10.0.0.5
5. Generate SSH keys.
ssh-keygen -t ed25519
# Press Enter to accept default path and set passphrase (optional)
6. Configure passwordless login.
ssh-copy-id admin@10.0.0.5
# Test: ssh admin@10.0.0.5 (Should not ask for password)
7. Disable root login.
sudo sed -i 's/#PermitRootLogin yes/PermitRootLogin no/' /etc/ssh/sshd_config
sudo systemctl restart sshd
8. Restart after changes.
sudo systemctl restart sshd

Section D: Scenario-Based Questions

1. Change default SSH port from 22 to 2222.
# 1. Edit config
sudo vi /etc/ssh/sshd_config
# Find and change: Port 2222

# 2. Allow port in firewall
sudo ufw allow 2222/tcp
# OR (firewalld)
sudo firewall-cmd --add-port=2222/tcp --permanent
sudo firewall-cmd --reload

# 3. Handle SELinux (RHEL/CentOS)
sudo semanage port -a -t ssh_port_t -p tcp 2222

# 4. Restart service
sudo systemctl restart sshd

# 5. Test connection
ssh -p 2222 user@ip
2. Steps to configure passwordless authentication.

1. Generate a key pair on the client: ssh-keygen.
2. Copy the public key to the server: ssh-copy-id user@server.
3. Test the connection: ssh user@server. If configured correctly, you will log in without a password prompt (or only prompted for the key's passphrase if set).

3. Importance of SSH key pairs.

Keys provide vastly superior security over passwords. They are cryptographically strong, immune to dictionary/brute-force attacks, and enable secure automation (scripts, cron jobs) by eliminating human interaction.

4. What happens if sshd_config contains incorrect syntax?

If sshd_config has syntax errors, the sshd service will fail to restart, or it will abort to prevent loading an insecure/broken configuration. This results in a Denial of Service—nobody can SSH into the server until the file is fixed. Always validate config with sudo sshd -t before restarting.

PART V: INTERVIEW QUESTIONS AND ANSWERS

Q1: What happens on the very first SSH connection to a server?

The client connects and receives the server's public host key. Since it has never seen this key before, it prompts the user: "The authenticity of host X can't be established... Are you sure you want to continue connecting?". If the user types 'yes', the key is saved to ~/.ssh/known_hosts. On future connections, the server's key is compared to the saved key to detect Man-in-the-Middle attacks.

Q2: How do you troubleshoot an "SSH Connection Refused" error?

1. Check if sshd is running: systemctl status sshd.
2. Check if the port is listening: ss -tulpn | grep 22.
3. Check firewall rules: ufw status or iptables -L.
4. Check if the server IP is reachable: ping <ip>.
5. Check for SSH config syntax errors: sshd -t.

Q3: What is SSH Agent Forwarding, and what are its security risks?

Agent Forwarding (ssh -A) allows you to SSH from Server A to Server B using the SSH key stored on your local laptop. It forwards the authentication request back to your laptop.
Risks: If Server A is compromised, an attacker with root access can hijack the forwarded socket and use your SSH keys to authenticate to other servers as you. Use ProxyJump (-J) instead, which is more secure.

Q4: What is the difference between ssh-keygen -t rsa -b 4096 and ssh-keygen -t ed25519?

- RSA 4096: Widely supported older algorithm. Relies on large key sizes for security. Slower during handshake.
- Ed25519: Modern Elliptic Curve algorithm. Much shorter keys (256-bit), faster, and provides better security against side-channel attacks. Ed25519 is the industry standard recommendation today.

Q5: How does sshd prevent brute-force password attacks natively?

OpenSSH uses login delays. By default, if a password authentication fails, the server imposes a delay before accepting the next attempt. Additionally, parameters like MaxAuthTries (default 6) limit the number of failed attempts per connection before disconnecting. For robust protection, administrators use tools like fail2ban to ban IPs after repeated failures.
