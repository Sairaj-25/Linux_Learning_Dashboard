LINUX/RHEL INTERVIEW QUESTIONS

1. Linux Basics

Q1. What is Linux? How is it different from Unix?

Answer:
Linux is a free, open-source operating system kernel created by Linus Torvalds in 1991. It is Unix-like but not Unix. Unix is a proprietary OS developed at Bell Labs in the 1970s. Linux was built from scratch to be Unix-compatible but is independently developed.

Key differences:

Feature Linux Unix
License Open source (GPL) Proprietary
Cost Free Paid
Examples RHEL, Ubuntu, Debian Solaris, AIX, HP-UX
Hardware Runs on almost anything Specific hardware
Source code Publicly available Closed source

Real-time use case:
Most cloud servers (AWS EC2, GCP VMs) run Linux (Amazon Linux, RHEL, Ubuntu). Your FastAPI or Django app almost certainly runs on Linux in production.

Common mistake:
Saying “Linux is Unix” — it is Unix-like (POSIX-compliant) but not certified Unix.

Follow-up:

  • Name some Linux distributions.
  • What does POSIX mean?

Q2. What is the kernel, shell, and terminal? How are they different?

Answer:

  • Kernel: The core of the operating system. It manages CPU, memory, processes, and device communication. Users never directly interact with the kernel.
  • Shell: A command interpreter that takes user commands and passes them to the kernel. Examples: bash, zsh, sh, fish.
  • Terminal: A program that provides a text interface (window) to access the shell. Examples: GNOME Terminal, PuTTY, iTerm2.

Analogy:
Terminal = car dashboard, Shell = steering wheel, Kernel = engine.

Command:

echo $SHELL        # shows current shell
cat /etc/shells    # lists all available shells
uname -r           # shows the kernel version

Real-time use case:
When you SSH into a production Linux server, the SSH client gives you a terminal, bash is your shell, and the kernel runs your FastAPI process.

Follow-up:

  • What is the difference between bash and sh?
  • What is a login shell vs non-login shell?

Q4. What is the difference between root user and a normal user?

Answer:

  • Root user (UID=0): Superuser with unlimited privileges. Can read, write, execute any file, install software, modify system configurations, and kill any process.
  • Normal user (UID >= 1000 on RHEL): Limited privileges. Can only access their own files and directories. Cannot install system packages or modify system configs without sudo.

Command:

whoami           # shows current user
id               # shows UID, GID, and groups
sudo whoami      # runs as root
su -             # switch to root (needs root password)

Real-time use case:
Your Django app should run as a non-root user (e.g., appuser) for security. Root should only be used for system admin tasks.

Common mistake:
Running production applications as root — a security risk.

Follow-up:

  • What is sudo and how is it different from su?
  • What file controls sudo access?

Q6. What is the difference between absolute and relative paths?

Answer:

  • Absolute path: Starts from root /. Always the same regardless of current directory. Example: /home/devuser/project/app.py
  • Relative path: Starts from the current directory. Depends on where you are. Example: ./project/app.py or ../logs/

Command:

pwd                  # print working directory (shows absolute path)
cd /etc/nginx        # absolute path
cd ../logs           # relative path (go up one, then into logs)
ls -la               # . is current dir, .. is parent dir

Real-time use case:
In a cron job or systemd service, always use absolute paths because the working directory may not be what you expect.

Common mistake:
Using relative paths in cron jobs causes “file not found” errors.

Follow-up:

  • What does . and .. represent?

3. File Permissions and Ownership

Q9. Explain Linux file permissions. What does -rwxr-xr-- mean?

Answer:
Permissions have 10 characters:

  • Position 1: file type (- = file, d = dir, l = link)
  • Positions 2-4: owner permissions (rwx)
  • Positions 5-7: group permissions (r-x)
  • Positions 8-10: others permissions (r–)

-rwxr-xr-- means:

  • Regular file
  • Owner: read + write + execute (rwx = 7)
  • Group: read + execute (r-x = 5)
  • Others: read only (r-- = 4)
Symbol Meaning Numeric
r Read 4
w Write 2
x Execute 1
- No permission 0

Command:

ls -la file.sh
stat file.sh

Q10. How do you use chmod, chown, and chgrp?

Answer:

  • chmod — change file permissions
  • chown — change file owner
  • chgrp — change file group

Command:

# chmod — symbolic mode
chmod u+x script.sh          # add execute for owner
chmod g-w file.txt           # remove write for group
chmod o=r file.txt           # set others to read only
chmod a+x script.sh          # add execute for all

# chmod — numeric mode
chmod 755 script.sh          # rwxr-xr-x
chmod 644 config.txt         # rw-r--r--
chmod 600 id_rsa             # rw------- (SSH private key)
chmod -R 755 /var/www/html   # recursive

# chown
chown devuser file.txt               # change owner
chown devuser:developers file.txt    # change owner and group
chown -R nginx:nginx /var/www/html   # recursive

# chgrp
chgrp developers project/

Real-time use case:
When deploying a FastAPI app, you set /opt/myapp owned by appuser and give only the necessary permissions. Nginx config files should be 644 (root readable).

Common mistake:
Using chmod 777 — gives everyone full access, a serious security risk.

  • What permissions should SSH private keys have?

Q11. What are SUID, SGID, and Sticky Bit?

Answer:
These are special permission bits beyond rwx.

SUID (Set User ID) — numeric 4 in front:

  • On executables: runs with the file owner’s permissions, not the caller’s.
  • Example: /usr/bin/passwd runs as root even when called by a normal user.
chmod u+s /usr/bin/program    # add SUID
chmod 4755 program            # numeric
ls -la /usr/bin/passwd        # shows: -rwsr-xr-x

SGID (Set Group ID) — numeric 2 in front:

  • On executables: runs with the file’s group permissions.
  • On directories: new files inherit the directory’s group.
chmod g+s /shared/dir         # add SGID on directory
chmod 2775 /shared/dir        # numeric
ls -la                        # shows: drwxrwsr-x

Sticky Bit — numeric 1 in front:

  • On directories: only the file owner (or root) can delete files, even if others have write access.
  • Classic use: /tmp directory
chmod +t /shared/dir          # add sticky bit
chmod 1777 /tmp               # numeric
ls -la /                      # shows: drwxrwxrwt for /tmp

Real-time use case:
On a shared project directory, SGID ensures all new files belong to the devteam group. Sticky bit on /tmp stops users from deleting each other’s temp files.

Common mistake:
Confusing SUID on directories (it’s mostly ignored on dirs in Linux).

Follow-up:

  • Find all SUID files on the system. How?
  • Why is SUID potentially dangerous?

Q12. What is umask? How does it work?

Answer:
umask defines the default permission mask applied when creating new files or directories. It subtracts from the maximum permissions.

  • Default maximum for files: 666 (no execute by default)
  • Default maximum for directories: 777
  • Common umask: 022

Calculation:

  • New file: 666 - 022 = 644 → -rw-r--r--
  • New dir: 777 - 022 = 755 → drwxr-xr-x

Command:

umask              # view current umask
umask 027          # set umask (owner=all, group=r+x, others=none)
umask 0022         # typical production setting

Real-time use case:
In a secure environment, you might set umask 027 so new files are not readable by others, which is important for config files with credentials.

Q13. What are ACLs (Access Control Lists)? Why are they needed?

Answer:
Standard Linux permissions only allow one owner, one group, and others. ACLs allow you to set permissions for multiple users and groups on a single file.

Command:

# Install if not present
dnf install acl

# Set ACL
setfacl -m u:john:rw file.txt          # give john read+write
setfacl -m g:devteam:rx /opt/app/      # give devteam read+execute
setfacl -R -m u:john:rwx /opt/app/     # recursive

# View ACL
getfacl file.txt

# Remove ACL
setfacl -x u:john file.txt
setfacl -b file.txt                    # remove all ACLs

Real-time use case:
You have a shared /data directory. The reports group needs read access, the etl_user needs write access, but the base group is data. Instead of changing group ownership, use ACLs.

Common mistake:
Filesystem must be mounted with acl option, or ACLs won’t work.

4. User and Group Management

Q14. How do you create, modify, and delete users in Linux?

Answer:

# Create user
useradd username                          # basic
useradd -m -s /bin/bash -c "Dev User" devuser   # with home, shell, comment
useradd -u 1500 -g developers devuser     # specific UID and group

# Set/change password
passwd devuser

# Modify user
usermod -aG sudo devuser           # add to supplementary group
usermod -s /bin/zsh devuser        # change shell
usermod -l newname oldname         # rename user
usermod -L devuser                 # lock account
usermod -U devuser                 # unlock account

# Delete user
userdel devuser                    # delete user
userdel -r devuser                 # delete user + home directory

Real-time use case:
When deploying a new microservice, you create a dedicated service account: useradd -r -s /sbin/nologin appservice (system user, no login shell).

Common mistake:
Forgetting -a in usermod -aG — without -a, it replaces all supplementary groups instead of appending.

Follow-up:

  • What is the difference between a system user and a regular user?
  • What does -s /sbin/nologin mean?

Q15. Explain /etc/passwd, /etc/shadow, and /etc/group.

Answer:

/etc/passwd — stores user account info (world-readable):

username:x:UID:GID:comment:home_dir:shell
devuser:x:1001:1001:Dev User:/home/devuser:/bin/bash

Fields: username, password placeholder (x), UID, GID, GECOS, home dir, shell

/etc/shadow — stores encrypted passwords (readable only by root):

username:$6$hash...:18000:0:99999:7:::

Fields: username, encrypted password, last change, min days, max days, warn days, inactive, expire

/etc/group — stores group info:

groupname:x:GID:member1,member2
developers:x:1002:devuser,john

Command:

cat /etc/passwd
sudo cat /etc/shadow
cat /etc/group
getent passwd devuser     # query user info
id devuser                # show UID, GID, groups

Q17. How do you configure sudo access?

Answer:
Sudo allows a normal user to run commands as root (or another user). Configuration is in /etc/sudoers (edit with visudo).

Command:

visudo                            # safe editor for sudoers
# or edit /etc/sudoers.d/custom file

# Syntax: user/group  hosts=(run_as) commands
devuser ALL=(ALL) ALL             # full sudo access
devuser ALL=(ALL) NOPASSWD: ALL   # without password prompt
devuser ALL=(ALL) /bin/systemctl  # only specific command
%developers ALL=(ALL) ALL         # entire group

# Grant sudo
usermod -aG wheel devuser         # wheel group has sudo on RHEL

Real-time use case:
A CI/CD pipeline user needs to restart services: ciuser ALL=(ALL) NOPASSWD: /bin/systemctl restart myapp

Common mistake:
Editing /etc/sudoers directly without visudo — a syntax error locks you out of sudo.

5. Process Management

Q18. How do you view running processes in Linux?

Answer:

ps aux                    # all processes, BSD style
ps -ef                    # all processes, UNIX style
ps aux | grep nginx       # filter by name
top                       # interactive, real-time view
htop                      # improved top (must install)
pgrep nginx               # get PID by name

ps aux columns:

  • USER: process owner
  • PID: process ID
  • %CPU, %MEM: resource usage
  • VSZ: virtual memory
  • RSS: resident (physical) memory
  • STAT: process state (R=running, S=sleeping, Z=zombie, D=disk wait)
  • COMMAND: command name

Real-time use case:
When a FastAPI/Gunicorn server is slow, run ps aux --sort=-%mem | head to find the top memory-consuming processes.

Q19. How do you kill a process?

Answer:

kill PID               # sends SIGTERM (15) — graceful stop
kill -9 PID            # sends SIGKILL (9) — force kill
kill -15 PID           # explicit SIGTERM
killall nginx          # kill all processes named nginx
pkill -f "gunicorn"    # kill by pattern match
pkill -u devuser       # kill all processes of a user

Signal types:

Signal Number Meaning
SIGTERM 15 Graceful termination (default)
SIGKILL 9 Force kill (cannot be ignored)
SIGHUP 1 Reload config (many daemons respond to this)
SIGINT 2 Interrupt (Ctrl+C)
SIGSTOP 19 Pause process

Real-time use case:
A stuck Gunicorn worker is blocking requests. First try kill -15 PID to allow cleanup; if it doesn’t respond within 10 seconds, use kill -9 PID.

Common mistake:
Always using kill -9 first — this skips cleanup handlers and can corrupt data or leave temp files.

  • What is the difference between SIGTERM and SIGKILL?
  • Can a process ignore SIGKILL?

Q20. What is a zombie process? What is an orphan process?

Answer:

Zombie process:
A process that has finished execution but still has an entry in the process table because its parent hasn’t read its exit status (wait() syscall). Shows as Z in ps.

  • Cannot be killed directly
  • Goes away when parent reads exit status or parent dies
  • If too many zombies accumulate, they can exhaust PID space

Orphan process:
A process whose parent has died. It is adopted by init/systemd (PID 1), which will clean it up properly.

Command:

ps aux | grep Z         # find zombie processes
ps -ef --ppid 1         # find processes adopted by init

Real-time use case:
A Python script spawns child processes but doesn’t call wait(). The children finish but become zombies. Fix by using subprocess.wait() or os.wait() in the parent.

Follow-up:

  • How do you get rid of a zombie process?
  • What is process adoption?

Q21. What is nice and renice? How do you set process priority?

Answer:
Nice values control CPU scheduling priority. Range: -20 (highest priority) to +19 (lowest priority).

  • Default nice value: 0
  • Only root can set negative nice values (higher priority)
nice -n 10 ./backup.sh           # start with nice value 10
renice -n 5 -p 1234              # change priority of PID 1234
renice -n -5 -p 1234             # increase priority (needs root)
ps -eo pid,ni,comm               # view PIDs with nice values

Real-time use case:
Running a heavy database backup during business hours — use nice -n 19 to give it the lowest priority so production traffic is unaffected.

Follow-up:

  • What is the relation between nice value and CPU priority?
  • Can a normal user increase their process priority?

Q22. What are foreground and background jobs? What is nohup?

Answer:

  • Foreground: Command runs in the terminal, blocking it until done.
  • Background: Command runs in the background, terminal is free.
  • nohup: Runs command immune to hangups (SIGHUP), continues even after terminal is closed.
./longscript.sh &          # run in background
jobs                        # list background jobs
fg %1                       # bring job 1 to foreground
bg %1                       # send job 1 to background
Ctrl+Z                      # suspend foreground job
Ctrl+C                      # terminate foreground job

# nohup
nohup ./longscript.sh &                     # output to nohup.out
nohup ./longscript.sh > output.log 2>&1 &   # custom output file

Real-time use case:
Starting a data pipeline script over SSH and then disconnecting: nohup python etl_job.py > etl.log 2>&1 &. Without nohup, the script dies when you disconnect.

Common mistake:
Forgetting & with nohup — it still runs in foreground until you manually press Ctrl+Z.

6. Systemd and Services

Q23. What is systemd? What are the main systemctl commands?

Answer:
systemd is the init system and service manager used in modern Linux (RHEL 7+, Ubuntu 16+). It manages services, mounts, timers, and more. It replaces the older SysV init.

Key systemctl commands:

systemctl start nginx           # start service
systemctl stop nginx            # stop service
systemctl restart nginx         # stop + start
systemctl reload nginx          # reload config without restart
systemctl status nginx          # show status and recent logs
systemctl enable nginx          # auto-start at boot
systemctl disable nginx         # remove auto-start
systemctl is-active nginx       # check if running
systemctl is-enabled nginx      # check if enabled

systemctl list-units            # list all units
systemctl list-unit-files       # list all unit files
systemctl daemon-reload         # reload systemd after unit file change
systemctl --failed              # show failed services

Real-time use case:
After deploying a new FastAPI app as a systemd service, run systemctl enable --now myapp to both start and enable it.

Common mistake:
Editing a .service file without running systemctl daemon-reload — systemd won’t pick up your changes.

Follow-up:

  • What is the difference between enable and start?
  • What is daemon-reload used for?

Q24. How do you create a custom systemd service unit?

Answer:
Create a file at /etc/systemd/system/myapp.service:

[Unit]
Description=MyFastAPI Application
After=network.target

[Service]
Type=simple
User=appuser
WorkingDirectory=/opt/myapp
ExecStart=/opt/myapp/venv/bin/gunicorn -w 4 -k uvicorn.workers.UvicornWorker main:app
ExecReload=/bin/kill -s HUP $MAINPID
Restart=always
RestartSec=5
StandardOutput=journal
StandardError=journal
Environment=ENVIRONMENT=production

[Install]
WantedBy=multi-user.target

systemctl daemon-reload
systemctl enable --now myapp
systemctl status myapp
journalctl -u myapp -f         # follow logs

Real-time use case:
Every FastAPI/Django production deployment needs a systemd service to auto-restart on crash and start on reboot.

Follow-up:

  • What does After=network.target mean?
  • What is the difference between Restart=always and Restart=on-failure?

Q25. How do you use journalctl?

Answer:
journalctl reads logs from systemd’s journal.

journalctl                          # all logs
journalctl -u nginx                 # logs for specific service
journalctl -u nginx -f              # follow (like tail -f)
journalctl -u nginx --since "1 hour ago"
journalctl -u nginx -n 100          # last 100 lines
journalctl -p err                   # only errors
journalctl --since "2024-01-01" --until "2024-01-02"
journalctl -k                       # kernel messages only
journalctl -b                       # logs from current boot
journalctl -b -1                    # logs from previous boot
journalctl --disk-usage             # check journal size
journalctl --vacuum-size=100M       # clean up old logs

Real-time use case:
Your FastAPI app crashed at 2am. Run journalctl -u myapp --since "2024-01-15 01:00" --until "2024-01-15 03:00" to investigate.

Follow-up:

  • Where are journal files stored on disk?
  • How do you make journal logs persistent across reboots?

7. Package Management (RHEL)

Q26. What is the difference between rpm, yum, and dnf?

Answer:

Tool Purpose
rpm Low-level package manager. Installs .rpm files but does NOT resolve dependencies.
yum High-level package manager. Resolves dependencies, manages repos. Legacy (RHEL 7).
dnf Replacement for yum (RHEL 8+). Faster, better dependency resolution, same syntax. 
# rpm — direct package operations
rpm -ivh package.rpm             # install
rpm -Uvh package.rpm             # upgrade
rpm -e packagename               # remove
rpm -qa                          # list all installed
rpm -qi nginx                    # info about package
rpm -ql nginx                    # list files in package
rpm -qf /etc/nginx/nginx.conf    # which package owns this file

# dnf — recommended for RHEL 8+
dnf install nginx                # install
dnf remove nginx                 # remove
dnf update                       # update all
dnf update nginx                 # update specific
dnf search nginx                 # search
dnf info nginx                   # package info
dnf list installed               # list installed packages
dnf provides /usr/bin/python3    # which package provides a file
dnf history                      # transaction history
dnf history undo 5               # undo transaction 5

Real-time use case:
Before deploying a Python backend on RHEL, you’d run dnf install python3 python3-pip git to install dependencies.

Common mistake:
Using rpm -ivh for packages with dependencies — it will fail. Use dnf install package.rpm instead to resolve dependencies.

Q27. How do you manage repositories in RHEL/Rocky Linux?

Answer:
Repositories are sources from which packages are downloaded. Repo files are in /etc/yum.repos.d/.

# List repos
dnf repolist
dnf repolist all             # including disabled

# Add EPEL (Extra Packages for Enterprise Linux)
dnf install epel-release     # on Rocky/AlmaLinux

# Manage repos
dnf config-manager --enable epel
dnf config-manager --disable epel

# Manual repo file example

# [myrepo]
# name=My Custom Repo
# baseurl=https://repo.example.com/rhel8/
# enabled=1
# gpgcheck=1
# gpgkey=https://repo.example.com/RPM-GPG-KEY

# Clean cache
dnf clean all
dnf makecache

Real-time use case:
To install Python 3.11 on Rocky Linux 8, you might add a software collection repo or use EPEL.

Q29. What is LVM and how does it work?

Answer:
LVM (Logical Volume Manager) provides flexible disk management. It adds a layer of abstraction between physical disks and filesystems.

Components:

  • PV (Physical Volume): Raw disk or partition prepared for LVM (/dev/sdb)
  • VG (Volume Group): Pool of storage made from one or more PVs
  • LV (Logical Volume): A “virtual partition” carved from a VG. This is what you format and mount.
# Step-by-step LVM setup
pvcreate /dev/sdb                         # create PV
vgcreate datavg /dev/sdb                  # create VG
lvcreate -L 20G -n datalv datavg          # create 20G LV

mkfs.xfs /dev/datavg/datalv               # format
mount /dev/datavg/datalv /data            # mount

# View
pvs / pvdisplay
vgs / vgdisplay
lvs / lvdisplay

# Extend LV (without downtime on XFS)
lvextend -L +10G /dev/datavg/datalv       # extend by 10G
xfs_growfs /data                          # grow XFS filesystem
# For ext4: resize2fs /dev/datavg/datalv

# Add new disk to existing VG
pvcreate /dev/sdc
vgextend datavg /dev/sdc

Real-time use case:
Your database data directory is 80% full. Add a new EBS volume, create a PV from it, extend the VG, extend the LV, and grow the filesystem — all without unmounting.

Q34. What is SELinux? How do you manage it?

Answer:
SELinux (Security-Enhanced Linux) is a mandatory access control (MAC) system that provides fine-grained security policies. It is enforced at the kernel level and restricts what processes can do, even as root.

SELinux modes:

  • Enforcing: Policies are enforced. Violations are blocked and logged.
  • Permissive: Violations are logged but NOT blocked. Good for debugging.
  • Disabled: SELinux is off.
getenforce               # show current mode
sestatus                 # detailed status

# Temporarily change mode
setenforce 0             # set permissive
setenforce 1             # set enforcing

# Permanently change (edit /etc/selinux/config)
# SELINUX=enforcing | permissive | disabled
# Requires reboot for disabled/enabled change

# Troubleshooting
ausearch -c nginx --raw | audit2allow -M mynginx   # generate policy from denials
semodule -i mynginx.pp                              # install custom policy
restorecon -Rv /var/www/html                        # restore default SELinux labels
ls -laZ /var/www/html                               # view SELinux context

# Check SELinux denials
grep "avc:  denied" /var/log/audit/audit.log

Real-time use case:
Nginx can’t read files in /opt/myapp/static/ even though file permissions are correct — SELinux is blocking it. Run restorecon -Rv /opt/myapp/static/ or check with ls -laZ.

Common mistake:
Disabling SELinux entirely instead of troubleshooting it. Set it to permissive, find the denial, and fix the label or policy.

10. SSH and Remote Access

Q35. How does SSH key-based authentication work?

Answer:
SSH uses asymmetric cryptography:

  1. Generate a key pair: private key (stays on client) + public key (goes to server)
  2. Server stores your public key in ~/.ssh/authorized_keys
  3. When you connect, the server sends a challenge encrypted with your public key
  4. Only your private key can decrypt it — proving your identity without a password
# Generate key pair
ssh-keygen -t ed25519 -C "devuser@laptop"    # modern, recommended
ssh-keygen -t rsa -b 4096                    # RSA (legacy compatible)

# Copy public key to server
ssh-copy-id devuser@server-ip
# or manually
cat ~/.ssh/id_ed25519.pub >> ~/.ssh/authorized_keys  # on server

# Connect
ssh devuser@server-ip
ssh -i ~/.ssh/custom_key devuser@server-ip
ssh -p 2222 devuser@server-ip                 # custom port

# SSH config file (~/.ssh/config)
Host myserver
    HostName 192.168.1.100
    User devuser
    IdentityFile ~/.ssh/id_ed25519
    Port 22

Real-time use case:
CI/CD pipeline (GitHub Actions, GitLab CI) uses SSH keys to deploy code to production servers — no passwords, fully automated.

Common mistake:
Wrong permissions on SSH files cause silent failures. Required permissions:

  • ~/.ssh/ → 700
  • ~/.ssh/authorized_keys → 600
  • ~/.ssh/id_rsa (private key) → 600

Q46. What is a shebang? Why is it important?

Answer:
The shebang (#!) is the first line of a script. It tells the OS which interpreter to use to execute the script.

#!/bin/bash        # use bash

Without a shebang, the OS uses the default shell, which might not be bash. #!/usr/bin/env bash is more portable across different systems where bash might be at different paths.

Common mistake:
Forgetting to chmod +x the script or having a Windows line ending (\r\n) which breaks the shebang on Linux.

Q48. What are exit codes? How do you use them?

Answer:
Every command exits with a status code. 0 = success, non-zero = failure.

ls /etc/passwd
echo $?        # 0 — success

ls /nonexistent
echo $?        # 2 — error (file not found)

# Exit codes in scripts
#!/bin/bash
if ! ping -c 1 8.8.8.8 &>/dev/null; then
    echo "Network unreachable"
    exit 1        # exit with failure
fi
echo "Network OK"
exit 0            # exit with success

# Using exit status in conditions
grep "error" logfile.txt
if [ $? -eq 0 ]; then
    echo "Errors found"
fi

# Shorthand
grep -q "error" logfile.txt && echo "Errors found" || echo "No errors"

Common exit codes:

  • 0 = Success
  • 1 = General error
  • 2 = Misuse of shell command
  • 126 = Command not executable
  • 127 = Command not found
  • 130 = Terminated with Ctrl+C

20. Bash — I/O and Redirection

Q52. Explain stdin, stdout, stderr, and redirection operators.

Answer:
Every process has three standard streams:

  • stdin (0): Standard input (keyboard by default)
  • stdout (1): Standard output (terminal by default)
  • stderr (2): Standard error (terminal by default)
# Redirection operators
command > file.txt          # redirect stdout to file (overwrite)
command >> file.txt         # redirect stdout to file (append)
command 2> error.log        # redirect stderr to file
command 2>> error.log       # append stderr to file
command &> all.log          # redirect both stdout and stderr
command > out.log 2>&1      # redirect stderr to stdout (then to file)
command 2>/dev/null         # discard errors (send to null device)
command > /dev/null 2>&1    # discard all output

# stdin redirection
command < input.txt          # read stdin from file
command <<EOF                # here document (heredoc)
line1
line2
EOF
command <<< "string"         # here string

# Pipes — connect stdout of one to stdin of next
ps aux | grep nginx
cat /etc/passwd | awk -F: '{print $1}' | sort
cat access.log | grep "500" | wc -l

# Practical examples
./backup.sh > /var/log/backup.log 2>&1        # log all output
./deploy.sh >> /var/log/deploy.log 2>&1 &     # background with logging

Real-time use case:
In cron jobs, always redirect output: 0 2 * * * /opt/backup.sh >> /var/log/backup.log 2>&1

Common mistake:
2>&1 > file.log is WRONG (order matters!) — stderr goes to terminal, stdout to file. Correct: > file.log 2>&1

28. Tips to Remember

Linux Permissions — The Quick Recall

chmod 755 → rwxr-xr-x → "I can do anything, others can read and run"
chmod 644 → rw-r--r--  → "I can read/write, others can only read"
chmod 600 → rw-------  → "Only I can read/write" (SSH keys, private configs)
chmod 777 → rwxrwxrwx  → "DANGER: everyone can do anything"

Process Signals — The Holy Trinity

  • kill -15 (SIGTERM) — “Please stop gracefully” → try this first
  • kill -9 (SIGKILL) — “Die now, no choice” → last resort
  • kill -1 (SIGHUP) — “Re-read your config” → for daemons

Cron Timing — Read Left to Right

Min Hour Day Month Weekday

  • 0 2 * * * = Every day at 2:00 AM
  • */5 * * * * = Every 5 minutes
  • 0 0 * * 0 = Every Sunday at midnight

Directory Purpose Quick Recall

  • /etc = Everything Configuration
  • /var = Variable data (logs, spool)
  • /tmp = Temporary (cleared on boot)
  • /proc = Process info (virtual FS)
  • /opt = Optional third-party software

Bash Script Error Handling Header

#!/bin/bash

trap 'echo "Error at line $LINENO"' ERR

Put this at the top of every serious script.

Q: “What is the difference between kill -9 and kill -15?”
A:

“kill -15 sends SIGTERM, which asks the process to terminate gracefully — the process can catch this signal, save state, and clean up before exiting. kill -9 sends SIGKILL, which the kernel enforces immediately and the process cannot catch or ignore. Always try -15 first; only use -9 if the process doesn’t respond, since -9 can leave temp files, corrupt locks, or skip important cleanup handlers.”

Q: “Explain the difference between > and >> and 2>&1.”
A:

“> redirects stdout to a file, overwriting it. >> appends stdout to a file. 2>&1 redirects file descriptor 2 (stderr) to wherever file descriptor 1 (stdout) currently points. So cmd > file.log 2>&1 means: redirect stdout to file.log, then redirect stderr to stdout (which is now file.log), so both go to the same file.”

Q: “How does SSH key-based authentication work?”
A:

“You generate a key pair with ssh-keygen — a private key stays on your client machine and a public key goes to ~/.ssh/authorized_keys on the server. When you connect, the server picks up your public key, encrypts a random challenge with it, and sends it to you. Your SSH client decrypts the challenge using your private key and sends back a response. If it matches, you’re authenticated without ever sending a password. This is why key-based auth is both more secure and more convenient for automation.”

Q: “What does set -euo pipefail do?”
A:

“set -e makes the script exit immediately if any command fails (non-zero exit code). set -u treats any reference to an unset variable as an error. set -o pipefail makes a pipeline fail if any command in it fails, not just the last one. Together these three options make Bash scripts fail loudly and early instead of continuing silently with bad data — which is essential for production scripts.”
