Q 1.1 Beginner
Disk full alert — Find and clean up large old log files without causing service disruption
finddurmtruncate
🧩 Scenario

Your monitoring system fires a "Disk 90% full" alert on a production web server at 2 AM. The application is still running. You need to identify what is consuming space and free up disk immediately without restarting the application.

❓ Problem Statement

Identify the top disk consumers, locate large log files, and safely clean them up while the application is live.

🛠 Step-by-Step Solution
1
Check which filesystem is full
BASH
df -h
# Filesystem      Size  Used Avail Use% Mounted on
# /dev/xvda1       50G   45G  2.5G  95% /
# tmpfs            16G     0   16G   0% /dev/shm
2
Find top-level directories consuming most space
BASH
sudo du -hd1 / 2>/dev/null | sort -rh | head -10
# 28G  /var
# 9G   /opt
# 4G   /home
3
Drill down into /var
BASH
sudo du -hd2 /var | sort -rh | head -10
# 24G  /var/log
# 18G  /var/log/nginx
4
Find all files larger than 1GB in /var/log
BASH
sudo find /var/log -type f -size +1G -exec ls -lh {} \;
5
Find files open by running processes (CRITICAL — cannot just rm them!)
BASH
sudo lsof /var/log/nginx/access.log
# nginx  1234  www-data  4w  REG  202,1  5368709120  nginx/access.log
# File is OPEN by nginx PID 1234 — do NOT delete!
6
Safely truncate open log files (does NOT disrupt nginx)
BASH
# Truncate to 0 bytes — nginx continues writing to the same file descriptor
sudo truncate -s 0 /var/log/nginx/access.log

# Alternative: redirect empty output
> /var/log/nginx/access.log

# Safe delete of OLD rotated logs (not currently open)
sudo find /var/log -name "*.gz" -mtime +7 -delete
sudo find /var/log -name "*.log.[0-9]" -mtime +3 -delete
7
Verify space reclaimed
BASH
df -h /
🔄 Alternative Solutions
  • journalctl --vacuum-size=500M — reduce systemd journal to 500MB
  • docker system prune -af — if Docker images/layers are consuming space
  • sudo apt-get clean or yum clean all — clear package cache
  • Check pip/conda caches: pip cache purge, conda clean --all
⚠️ Common Mistakes
  • Deleting a file that's currently open — disk space is NOT freed until process closes the file
  • Using rm on large log files instead of truncate — same problem with open files
  • Running du on NFS mounts without -x flag — can take hours
💡 Interview Tips
  • Always check lsof | grep deleted — files deleted but held open by a process are invisible to du but still consume disk blocks
  • Interviewers expect you to know the truncate trick — it's a classic production gotcha
  • Mention df -i to check inode exhaustion — disk can be "full" with 0% block usage if inodes are exhausted
Q 1.2 Intermediate
Archive and transfer 90 days of application logs to remote backup server with compression
findtarrsyncssh
🧩 Scenario

Your company's compliance policy requires retaining application logs for 1 year. Local disk is too expensive. You need to archive logs older than 90 days, compress them, and sync to a remote NFS backup server nightly via cron.

❓ Problem Statement

Create a script that finds logs older than 90 days, archives them with timestamp, transfers to backup server, and removes local copies after successful transfer.

🛠 Step-by-Step Solution
1
Find log files older than 90 days
BASH
find /var/log/app -name "*.log" -mtime +90 -type f
2
Archive them into a dated tarball
BASH
# Create archive with date stamp
ARCHIVE=/tmp/app_logs_$(date +%Y%m%d).tar.gz

find /var/log/app -name "*.log" -mtime +90 -type f | \
    tar -czf $ARCHIVE -T -

# Verify archive integrity before deletion
tar -tzf $ARCHIVE > /dev/null && echo "Archive OK"
3
Transfer to backup server with verification
BASH
rsync -avz --checksum --remove-source-files \
    $ARCHIVE backup@10.0.1.200:/mnt/backup/logs/

# --checksum: verify data integrity after transfer
# --remove-source-files: delete local after successful transfer
4
Remove original log files after successful rsync
BASH
if [ $? -eq 0 ]; then
    find /var/log/app -name "*.log" -mtime +90 -delete
    echo "$(date): Archive and cleanup complete" >> /var/log/log_archive.log
else
    echo "$(date): rsync FAILED — logs NOT deleted" >> /var/log/log_archive.log
    exit 1
fi
🔄 Alternative Solutions
  • find + xargs tar — for very large numbers of files: find /var/log -mtime +90 | xargs tar -czf archive.tar.gz
  • scp — simpler but no checksum verification, no resume on failure
  • logrotate — the proper production tool for log rotation; configure with compress + dateext + postrotate hooks for archiving
⚠️ Common Mistakes
  • Deleting files before verifying the archive is valid — always verify tar with -t before deleting
  • Not checking rsync exit code before deleting source files
  • Using cp instead of rsync — no checksum, no resume capability
💡 Interview Tips
  • The pattern: find → archive → verify → transfer → verify transfer → delete is the safe production workflow
  • Mention logrotate as the proper tool — shows you know the right approach, not just ad-hoc commands
  • Always check exit codes — if [ $? -eq 0 ] before destructive operations
Q 1.3 Advanced
Find all Python files modified in the last 24 hours, count lines, and generate a report
findxargswcawk
🧩 Scenario

A team lead wants a daily report of all Python files changed by developers in the last 24 hours — showing filename, line count, and size — sorted by most recently modified. This needs to run from CI/CD pipeline.

🛠 Step-by-Step Solution
1
Find recently modified .py files
BASH
find /opt/project -name "*.py" -mtime -1 -type f
2
Count lines in each file using xargs
BASH
find /opt/project -name "*.py" -mtime -1 -type f | \
    xargs -I{} wc -l {} | \
    sort -rn
3
Generate full report with filename, lines, size, last modified
BASH
echo "=== Python File Change Report: $(date) ==="
printf "%-60s %8s %10s %20s\n" "FILE" "LINES" "SIZE" "MODIFIED"
echo "$(printf '%0.s-' {1..102})"

find /opt/project -name "*.py" -mtime -1 -type f \
    -printf "%T@ %p\n" | \
    sort -rn | \
    awk '{print $2}' | \
    while read file; do
        lines=$(wc -l < "$file")
        size=$(du -sh "$file" | cut -f1)
        mtime=$(stat -c "%y" "$file" | cut -d. -f1)
        printf "%-60s %8d %10s %20s\n" "$file" "$lines" "$size" "$mtime"
    done
💡 Interview Tips
  • xargs -P 4 runs 4 processes in parallel — useful for processing large numbers of files faster
  • Use find -print0 | xargs -0 for filenames with spaces
  • stat -c "%y" gives modification time; -c "%s" gives size in bytes
Q 1.4 Intermediate
Hard links vs Symbolic links — production scenario with shared configuration files
lninodesymlink
🧩 Scenario

Multiple microservices need to share the same database configuration file. When the config is updated, all services should see the change immediately. You need to choose between hard links and symbolic links.

🛠 Step-by-Step Solution
1
Create the master config file
BASH
mkdir -p /etc/shared-config
cat > /etc/shared-config/db.conf << EOF
DB_HOST=prod-db.internal
DB_PORT=5432
DB_USER=app_user
EOF
2
Create symbolic links for each service
BASH
# Symbolic link — recommended for cross-directory config sharing
ln -s /etc/shared-config/db.conf /opt/service-a/config/db.conf
ln -s /etc/shared-config/db.conf /opt/service-b/config/db.conf
ln -s /etc/shared-config/db.conf /opt/service-c/config/db.conf
3
Verify links and understand the difference
BASH
ls -lai /etc/shared-config/db.conf /opt/service-a/config/db.conf
# 123456 -rw-r--r-- 2 root root 64 /etc/shared-config/db.conf
# 789012 lrwxrwxrwx 1 root root 31 /opt/service-a/config/db.conf -> /etc/shared-config/db.conf

# Hard link demo (same inode!):
ln /etc/shared-config/db.conf /etc/shared-config/db.conf.hard
ls -lai /etc/shared-config/
# 123456 -rw-r--r-- 2 root root 64 db.conf      ← link count = 2
# 123456 -rw-r--r-- 2 root root 64 db.conf.hard ← SAME inode 123456!
🔄 Hard Links vs Soft Links — Key Differences
  • Hard link: Same inode, same filesystem only, survives deletion of original — ln source dest
  • Soft/Symbolic link: Different inode, works across filesystems, breaks if original deleted — ln -s source dest
  • Cannot create hard links for directories (prevents circular references)
  • Use symlinks for config sharing — more visible, cross-filesystem capable
⚠️ Common Mistakes
  • Creating symlink with relative path and then moving it — breaks the link. Always use absolute paths for config symlinks
  • Thinking rm on a hard-linked file deletes data — data persists until all hard links are removed
💡 Interview Tips
  • Classic interview Q: "If you delete the original file of a symlink, what happens?" — the symlink becomes broken (dangling)
  • Find broken symlinks: find /opt -type l -! -e
  • ls -l shows l for symlink, number after permissions is link count for hard links

Q 2.1 Intermediate
Web deployment fails — permission denied errors after deploying as root, app runs as www-data
chmodchownfindumask
🧩 Scenario

A developer deployed new code as the root user. Now the nginx/gunicorn application (running as www-data) throws "Permission denied" errors trying to read static files and write to upload directories.

❓ Problem Statement

Fix file ownership and permissions correctly — web files readable by www-data, upload directory writable, config files protected, without making everything 777.

🛠 Step-by-Step Solution
1
Check current ownership and permissions
BASH
ls -la /var/www/html/
# -rw------- 1 root root  index.html    ← www-data cannot read!
# drwx------ 1 root root  uploads/      ← www-data cannot write!
2
Fix ownership to www-data
BASH
sudo chown -R www-data:www-data /var/www/html/
3
Set correct permissions: directories=755, files=644 using find
BASH
# Directories: owner rwx, group+others rx (need x to traverse)
sudo find /var/www/html -type d -exec chmod 755 {} \;

# Regular files: owner rw, group+others r
sudo find /var/www/html -type f -exec chmod 644 {} \;

# Upload directory needs write permission for www-data
sudo chmod 775 /var/www/html/uploads/
4
Protect sensitive config files
BASH
# .env files — readable only by owner
sudo chmod 600 /var/www/html/.env
sudo chown www-data:www-data /var/www/html/.env

# Make scripts executable but not writable by web process
sudo find /var/www/html -name "*.sh" -exec chmod 750 {} \;
5
Set umask for www-data so new files have correct permissions by default
BASH
# In /etc/profile.d/app.sh or systemd service file:
# UMask=0022 means: files=644, dirs=755 (default for www-data)
grep UMask /lib/systemd/system/nginx.service
🔄 Alternative Solutions (ACL for fine-grained control)
BASH
# ACL: Give www-data read access to /var/www/html without changing ownership
sudo setfacl -R -m u:www-data:rx /var/www/html/

# Give www-data write access only to uploads
sudo setfacl -m u:www-data:rwx /var/www/html/uploads/

# Set default ACL so new files inherit permissions
sudo setfacl -m d:u:www-data:rx /var/www/html/

# View ACL
getfacl /var/www/html/
⚠️ Common Mistakes
  • Using chmod 777 — never acceptable in production. Always specific permissions
  • Recursively chowning without understanding impact — chown -R on / is catastrophic
  • Forgetting execute permission on directories — files inside are inaccessible without x on parent
💡 Interview Tips
  • Standard web permissions: dirs=755, static files=644, writable dirs=775, secrets=600
  • ACLs shine when you need multiple users with different permissions beyond owner/group/other
  • stat filename shows all permission info including inode, size, timestamps in one view
Q 2.2 Advanced
Security audit — find all SUID/SGID binaries and world-writable files on the server
findSUIDSGIDsecurity
🧩 Scenario

Your security team requires a monthly audit report of all SUID/SGID binaries and world-writable files on production servers. These are common privilege escalation vectors that must be tracked against a baseline.

🛠 Step-by-Step Solution
1
Find all SUID (Set User ID) binaries — run as owner, not caller
BASH
sudo find / -perm -4000 -type f 2>/dev/null | sort
# /usr/bin/passwd    ← needs root to modify /etc/shadow
# /usr/bin/sudo
# /usr/bin/su
# /usr/sbin/ping     ← needs raw socket access
# Any UNEXPECTED entries here = SECURITY RISK!
2
Find all SGID (Set Group ID) files
BASH
sudo find / -perm -2000 -type f 2>/dev/null | sort
3
Find world-writable files (potential backdoor locations)
BASH
# Files world-writable (excluding /tmp, /proc, /sys)
sudo find / -path /proc -prune -o -path /sys -prune -o \
    -path /tmp -prune -o -perm -o+w -type f -print 2>/dev/null
4
Generate and compare against baseline
BASH
# Create baseline
sudo find / -perm -4000 -type f 2>/dev/null | sort > /root/suid_baseline.txt

# Monthly audit — compare with baseline
sudo find / -perm -4000 -type f 2>/dev/null | sort > /tmp/suid_current.txt
diff /root/suid_baseline.txt /tmp/suid_current.txt
# Lines starting with > = NEW SUID files added since baseline — INVESTIGATE!
💡 Interview Tips
  • SUID on shell binaries (bash, dash, sh) is an immediate red flag — attackers plant these for persistence
  • -perm -4000 means "SUID bit set" (4000 = SUID, 2000 = SGID, 1000 = sticky bit)
  • CIS Benchmark and RHCSA exams specifically test knowledge of SUID/SGID auditing

5 Questions
Q 3.1 Intermediate
Onboard a new DevOps engineer with proper sudo access, SSH key setup, and password policy
useraddusermodpasswdchagesudo
🧩 Scenario

A new DevOps engineer "Alice" joins the team. She needs: a Linux account on 5 production servers, membership in the devops and docker groups, sudo privileges for specific commands only, SSH key authentication, and a 90-day password expiry policy.

🛠 Step-by-Step Solution
1
Create user account with home directory and shell
BASH
sudo useradd -m -s /bin/bash -c "Alice DevOps" alice
# -m: create home directory
# -s: set login shell
# -c: comment/description

# Set temporary password (force change on first login)
sudo passwd alice
sudo passwd -e alice    # -e: expire immediately, force change at login
2
Add to required groups
BASH
# Add to supplementary groups (devops, docker)
sudo usermod -aG devops,docker alice
# CRITICAL: -aG = APPEND to groups. Without -a, existing groups are REPLACED!

# Verify group membership
id alice
groups alice
3
Configure sudo access for specific commands only
BASH
# ALWAYS edit sudoers with visudo (syntax validation!)
sudo visudo -f /etc/sudoers.d/alice

# Add to file /etc/sudoers.d/alice:
# alice ALL=(ALL) NOPASSWD: /usr/bin/systemctl, /usr/bin/docker
# alice ALL=(ALL) /usr/bin/apt-get, !/usr/bin/apt-get install curl
4
Set up SSH key authentication
BASH
# Create .ssh directory with correct permissions
sudo mkdir -p /home/alice/.ssh
sudo chmod 700 /home/alice/.ssh

# Add her public key
echo "ssh-ed25519 AAAA...her_public_key alice@laptop" | \
    sudo tee /home/alice/.ssh/authorized_keys

# Strict permissions on authorized_keys
sudo chmod 600 /home/alice/.ssh/authorized_keys
sudo chown -R alice:alice /home/alice/.ssh
5
Set password expiry policy with chage
BASH
# Set password to expire in 90 days, warn at 7 days
sudo chage -M 90 -W 7 -I 14 alice
# -M 90: max password age = 90 days
# -W 7:  warn 7 days before expiry
# -I 14: lock account if inactive 14 days after expiry

# View password aging info
sudo chage -l alice
⚠️ Common Mistakes
  • usermod -G groups (without -a) REPLACES all supplementary groups — user loses existing group memberships
  • Directly editing /etc/sudoers instead of visudo — a syntax error can lock out ALL sudo access
  • Not setting correct permissions on .ssh directory — SSH will refuse key auth if .ssh is group/world readable
💡 Interview Tips
  • The -aG distinction is one of the most commonly tested Linux admin questions
  • RHCSA exam specifically tests: useradd, usermod -aG, chage, visudo, and SSH key setup
  • Mention that group changes take effect at next login — use newgrp groupname for immediate effect in current session
Q 3.2 Advanced
Find all users with UID 0 (root equivalents) and audit sudo access — security hardening
awkpasswdsudosecurity
🧩 Scenario

Your security team audited the server and wants a report of: all users with UID=0 (root equivalents), all accounts that can run sudo, accounts with no password, and accounts with expired passwords.

🛠 Step-by-Step Solution
BASH
# 1. Find all UID=0 accounts (root equivalents — should be ONLY root!)
awk -F: '$3 == 0 {print "UID-0 account:", $1}' /etc/passwd

# 2. Find users with NO password (!! or empty in /etc/shadow)
sudo awk -F: '$2 == "" || $2 == "!!" {print "No password:", $1}' /etc/shadow

# 3. Find accounts not logged in for 90+ days
sudo lastlog | awk 'NR>1 && $0 !~ "Never" {print}'

# 4. List all users who can run sudo
sudo grep -r "ALL=\|sudo" /etc/sudoers /etc/sudoers.d/ 2>/dev/null

# 5. Find all human users (UID >= 1000)
awk -F: '$3 >= 1000 && $3 != 65534 {print $1, $3, $6}' /etc/passwd

# 6. Find users with shell access (not /sbin/nologin)
grep -v "\(nologin\|false\)" /etc/passwd | awk -F: '{print $1, $7}'
💡 Interview Tips
  • Service accounts should have /sbin/nologin or /bin/false as shell — prevents interactive login
  • UID 0 accounts besides root are a critical security finding in any audit
  • RHCE/CIS Benchmark: lock inactive accounts with usermod -L username and set expiry with chage -E 0 username

Q 4.1 Advanced
Production database server running out of space at 3 AM — extend LVM volume online without downtime
LVMlvextendresize2fsvgextend
🧩 Scenario

PostgreSQL on a production server is failing with "No space left on device" at 3 AM. /var/lib/postgresql is on an LVM logical volume. A new 100GB EBS volume has been attached to the server. The database CANNOT be stopped.

❓ Problem Statement

Extend the LVM volume while PostgreSQL is running, add new disk to the volume group, grow the filesystem online — zero downtime.

🛠 Step-by-Step Solution
1
Identify current state
BASH
df -h /var/lib/postgresql
# /dev/mapper/vg0-lv_pgsql   50G   50G    0  100% /var/lib/postgresql

lvs    # Show LVs
vgs    # Show VGs and FREE SPACE
pvs    # Show PVs
lsblk  # Identify new attached disk (e.g., /dev/xvdb)
2
Initialize new disk as LVM physical volume
BASH
sudo pvcreate /dev/xvdb
# Physical volume "/dev/xvdb" successfully created
3
Add new PV to existing volume group
BASH
sudo vgextend vg0 /dev/xvdb
# Volume group "vg0" successfully extended

vgs  # Verify — VFree should now show 100G
4
Extend logical volume AND filesystem in one command (ONLINE!)
BASH
# -r flag: resize filesystem automatically after LV extension
sudo lvextend -L +100G -r /dev/vg0/lv_pgsql
# Size of logical volume vg0/lv_pgsql changed from 50 GiB to 150 GiB
# Filesystem resize successful — NO DOWNTIME!

# Verify — PostgreSQL never stopped
df -h /var/lib/postgresql
# /dev/mapper/vg0-lv_pgsql  150G  50G  100G  34% /var/lib/postgresql
🔄 Without the -r flag (manual filesystem resize)
BASH
# Extend LV only
sudo lvextend -L +100G /dev/vg0/lv_pgsql

# Then grow filesystem manually:
# For ext4 (can be done ONLINE on mounted filesystem):
sudo resize2fs /dev/vg0/lv_pgsql

# For XFS (also online):
sudo xfs_growfs /var/lib/postgresql

# IMPORTANT: XFS cannot be SHRUNK — plan carefully
⚠️ Common Mistakes
  • Trying to shrink XFS — it's not supported. Plan ahead or use ext4 if shrinking is needed
  • Shrinking LV before shrinking filesystem — data corruption! Always: shrink filesystem first, then LV
  • Forgetting -r flag — extending LV without growing filesystem leaves the extra space unusable
💡 Interview Tips
  • This scenario is RHCSA exam content — know the 4-step flow: pvcreate → vgextend → lvextend -r
  • ext4 and XFS can both be grown online (while mounted). Only ext4 can be shrunk (offline only)
  • Always verify with vgs that VFree is sufficient before attempting to extend
Q 4.2 Intermediate
Mount NFS share persistently in /etc/fstab and handle boot failure gracefully with nofail
mountfstabNFSblkid
🧩 Scenario

A data engineering team needs an NFS share from the file server (10.0.1.100:/data/shared) mounted on all compute nodes at /mnt/shared-data. The mount must persist after reboot, and servers should still boot successfully even if the NFS server is temporarily unavailable.

🛠 Step-by-Step Solution
1
Test the NFS mount manually first
BASH
sudo mkdir -p /mnt/shared-data
sudo mount -t nfs 10.0.1.100:/data/shared /mnt/shared-data
df -h /mnt/shared-data  # Verify it works
2
Add to /etc/fstab with production-safe options
BASH
sudo nano /etc/fstab
# Add line:
10.0.1.100:/data/shared  /mnt/shared-data  nfs  defaults,_netdev,nofail,soft,timeo=15,retrans=3  0 0

# Options explained:
# _netdev:  wait for network before mounting (critical for NFS at boot)
# nofail:   don't fail boot if NFS unavailable
# soft:     timeout instead of hanging indefinitely  
# timeo=15: 1.5 second NFS timeout
# retrans=3: retry 3 times before giving up
3
Verify fstab is correct BEFORE rebooting
BASH
sudo umount /mnt/shared-data
sudo mount -a              # Mount all in fstab — shows errors immediately
sudo findmnt --verify     # Verify fstab syntax
df -h /mnt/shared-data
⚠️ Common Mistakes
  • Missing _netdev — system tries to mount before network is up, fails at boot
  • Missing nofail — if NFS server is down, server gets stuck in emergency mode at boot
  • Not testing with mount -a before rebooting — a bad fstab entry renders the server unbootable
💡 Interview Tips
  • The nofail + _netdev combination is the correct production pattern for any network-dependent mount
  • For local block devices, use UUID= instead of /dev/sdX — device names can change on reboot
  • sudo blkid /dev/sdb1 gets the UUID for fstab entries

Q 5.1 Intermediate
Kubernetes cluster upgrade fails — package held at wrong version, need to pin and upgrade carefully
apt-getapt-markapt-cachedpkg
🧩 Scenario

During a Kubernetes upgrade, apt-get upgrade accidentally upgraded kubeadm from 1.28 to 1.29. The worker nodes need to stay on 1.28 until the control plane is updated. You need to: rollback kubeadm to 1.28, pin all kube packages at 1.28, then perform a controlled upgrade later.

🛠 Step-by-Step Solution
1
Check available versions of kubeadm
BASH
apt-cache madison kubeadm
# kubeadm | 1.29.0-1.1 | https://pkgs.k8s.io ...
# kubeadm | 1.28.4-1.1 | https://pkgs.k8s.io ...
# kubeadm | 1.28.3-1.1 | https://pkgs.k8s.io ...
2
Downgrade kubeadm to specific version
BASH
sudo apt-get install -y kubeadm=1.28.4-1.1 --allow-downgrades
kubeadm version
3
Pin all kube packages to prevent accidental upgrade
BASH
sudo apt-mark hold kubeadm kubelet kubectl
# kubeadm set on hold
# kubelet set on hold
# kubectl set on hold

# Verify held packages
sudo apt-mark showhold
# kubeadm
# kubelet
# kubectl
4
When ready to upgrade — unhold, upgrade specific version, re-hold
BASH
# Unhold
sudo apt-mark unhold kubeadm kubelet kubectl

# Upgrade to specific version
sudo apt-get install -y \
    kubeadm=1.29.0-1.1 \
    kubelet=1.29.0-1.1 \
    kubectl=1.29.0-1.1

# Re-hold at new version
sudo apt-mark hold kubeadm kubelet kubectl
🔄 RHEL/CentOS equivalent with yum-versionlock
BASH
# Install versionlock plugin
sudo yum install -y yum-versionlock
# or for dnf:
sudo dnf install -y python3-dnf-plugin-versionlock

# Lock versions
sudo yum versionlock add kubeadm kubelet kubectl

# List locked packages
sudo yum versionlock list

# Remove lock
sudo yum versionlock delete kubeadm
💡 Interview Tips
  • apt-mark hold is the standard Kubernetes upgrade safety mechanism — all K8s docs reference it
  • Always check apt-cache policy package-name before upgrading — see exactly which version will be installed and from which repository
  • In Docker best practices: always pin versions in Dockerfile: apt-get install nginx=1.24.*
Q 5.2 Intermediate
Find which package owns a missing library causing application crash — rpm/dpkg forensics
rpmdpkglddldconfig
🧩 Scenario

After a server migration, a Python application crashes with: ImportError: libpq.so.5: cannot open shared object file. You need to find which package provides this library and install it.

🛠 Step-by-Step Solution
1
On Debian/Ubuntu — find which package provides the library
BASH
# Search for package providing libpq.so.5
apt-cache search libpq
apt-file search libpq.so.5    # Requires: apt-get install apt-file
# libpq5: /usr/lib/x86_64-linux-gnu/libpq.so.5

# Install it
sudo apt-get install libpq5
2
On RHEL/CentOS — find package providing the library
BASH
yum provides '*/libpq.so.5'
# postgresql-libs-13.3.x86_64 provides libpq.so.5

# Or with rpm for already-installed packages:
rpm -qf /usr/lib64/libpq.so.5
# postgresql-libs-13.3-1.el8.x86_64
3
Verify library is found by linker
BASH
ldconfig -p | grep libpq
# If empty: sudo ldconfig  (update library cache)

# Check what libraries an application needs
ldd /usr/bin/myapp
# libpq.so.5 => /usr/lib/x86_64-linux-gnu/libpq.so.5 ✓
# libssl.so.3 => not found ← next problem to fix!
💡 Interview Tips
  • ldd binary shows ALL shared libraries an application needs — great for diagnosing missing library errors
  • rpm -qf /path/to/file is the RPM equivalent of dpkg -S /path/to/file — both identify the owner package
  • After installing libraries: always run sudo ldconfig to refresh the dynamic linker cache
