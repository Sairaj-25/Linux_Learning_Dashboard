Linux Permission Model
Understanding rwx before learning chmod/chown
Owner (u)
rwx = 7
Read + Write + Execute
Group (g)
r-x = 5
Read + Execute only
Others (o)
r-- = 4
Read only
TEXT — Permission Reference
# r = 4 (read)  w = 2 (write)  x = 1 (execute)
#
# Common values:  7=rwx  6=rw-  5=r-x  4=r--  0=---
#
# Standard permissions:
777 = rwxrwxrwx  (NEVER in production — security risk)
755 = rwxr-xr-x  (scripts, directories)
644 = rw-r--r--  (regular files)
640 = rw-r-----  (config files — owner+group only)
600 = rw-------  (SSH keys, secrets, .env files)
700 = rwx------  (private directories)
400 = r--------  (read-only: certificates)
#
# Special bits:
4000 = SUID  (run as file owner)
2000 = SGID  (run as file group / inherit dir group)
1000 = Sticky (only owner can delete in shared dir)
chmod
Change Mode
Change file/directory permissions — octal and symbolic modes
1Command Overview

chmod (Change Mode) modifies the permission bits of files and directories. It accepts two notation styles: octal (755, 644) for setting absolute permissions, and symbolic (u+x, g-w, o=r) for relative changes. In production, chmod is used for securing SSH keys, fixing web server permissions, making scripts executable, and hardening file access.

2Syntax
BASH
chmod [OPTIONS] MODE FILE...

  MODE (octal):    chmod 755 file.sh
  MODE (symbolic): chmod u+x,g-w,o=r file.sh
    u = user/owner   g = group   o = others   a = all
    + = add          - = remove   = = set exactly
    r = read(4)      w = write(2) x = execute(1)
3Commonly Used Options
Flag/UsageDescriptionExample
-RRecursive — apply to all files in directorychmod -R 755 /var/www/html/
-vVerbose — show each file changedchmod -v 644 *.conf
-cLike verbose but only show changed fileschmod -c 755 script.sh
--referenceUse permissions from another file as referencechmod --reference=ref.sh target.sh
Symbolic Mode Examples
u+xAdd execute for ownerchmod u+x deploy.sh
g-wRemove write for groupchmod g-w /etc/config
o=rSet others to read-only exactlychmod o=r report.txt
a+rAdd read for ALL (owner+group+others)chmod a+r public.txt
u=rwx,g=rx,o=Set exact permissions for all groupschmod u=rwx,g=rx,o= dir/
Special Bits
4755SUID bit — runs as file owner (e.g., passwd)chmod 4755 /usr/bin/myapp
2755SGID bit — runs as group / dir files inherit groupchmod 2775 /shared/project/
1777Sticky bit — only owner can delete (e.g., /tmp)chmod 1777 /tmp
4Practical Examples
BASH
# Make script executable
chmod +x deploy.sh
chmod 755 deploy.sh    # Owner: full, Group+Others: read+execute

# Secure SSH private key (REQUIRED — SSH refuses 644!)
chmod 600 ~/.ssh/id_rsa
chmod 700 ~/.ssh/

# Web server permissions (standard pattern)
find /var/www/html -type d -exec chmod 755 {} \;   # Directories
find /var/www/html -type f -exec chmod 644 {} \;   # Files
chmod 640 /var/www/html/.env   # Protect secrets

# Shared directory — SGID ensures new files inherit group
chmod 2775 /opt/shared-project/
ls -ld /opt/shared-project/
# drwxrwsr-x  (s = SGID bit set)

# Remove all permissions from others (secure file)
chmod o= sensitive_data.csv

# Add execute for all — quick way to allow script execution
chmod a+x /usr/local/bin/myutil

# Sticky bit on shared directory (/tmp-like)
chmod 1777 /shared/uploads/    # Everyone can write, only owner can delete own files
6Interview Questions & Answers
Q1What is the difference between octal and symbolic chmod notation? When do you prefer each?
A:Octal (chmod 755) sets absolute permissions — all three groups are set simultaneously. Symbolic (chmod u+x) makes relative changes without affecting other bits. Use octal when setting from scratch (new files, deployments). Use symbolic in scripts when you only want to toggle specific bits — e.g., chmod +x script.sh adds execute without changing read/write.
Q2Explain SUID, SGID, and Sticky bit with real-world examples.
A:SUID (4xxx): File runs as file owner — /usr/bin/passwd has SUID root, so any user can change their own password (needs to write /etc/shadow). SGID (2xxx) on dir: New files created inside inherit the directory's group — critical for shared project directories where files should belong to "devteam" not the creator's primary group. Sticky (1xxx): On directories like /tmp — everyone can write but only the file owner and root can delete a file. Prevents users from deleting each other's temp files.
Q3SSH refuses to connect saying "WARNING: UNPROTECTED PRIVATE KEY FILE!" How do you fix it?
A:The SSH private key has permissions too open. Fix: chmod 600 ~/.ssh/id_rsa && chmod 700 ~/.ssh/. SSH requires the private key to be readable ONLY by the owner (600 = rw-------). If the key is world or group readable, SSH considers it a security risk and refuses to use it.
7Common Mistakes & Tips
❌
chmod 777 is never acceptable in production — it gives everyone write access, making the file a security vulnerability. If nginx is compromised, it can overwrite any 777 file.
⚠️
chmod -R on /var/www without checking — recursively applying 755 to ALL files makes config files and database files potentially world-readable. Always separate file permissions (644) from directory permissions (755) using find.
✅
Pro tip: Use stat -c "%a %n" file to quickly get the octal permissions of a file. Use ls -l | grep "^-..s" to find SUID binaries.
chown
Change Owner
Change file owner and/or group — service isolation principle
1Command Overview

chown changes the owner and optionally the group of files/directories. In production, it enforces the principle of least privilege — services run as dedicated users and own only their files. Web servers own web root, databases own data directories, ML services own model files.

2Syntax
BASH
chown [OPTIONS] OWNER[:GROUP] FILE...

  chown user file          ← change owner only
  chown user:group file    ← change both owner and group
  chown :group file        ← change group only (same as chgrp)
  chown user: file         ← change owner, set group to user's login group
3Commonly Used Options
FlagDescriptionExample
-RRecursive — change owner for all files in dirchown -R www-data:www-data /var/www/
-vVerbose — show each changechown -v alice:devops report.csv
-hChange symlink itself, not its targetchown -h user symlink
--referenceUse ownership from reference filechown --reference=/etc/app/cfg file
--fromChange only if current owner matcheschown --from=old:old new:new file
4Practical Examples
BASH
# Web server deployment
sudo chown -R www-data:www-data /var/www/html/

# Database files — owned by postgres service user
sudo chown -R postgres:postgres /var/lib/postgresql/

# ML model files — owned by ml service account
sudo chown -R mlservice:mlteam /opt/models/

# Docker socket — add user to docker group instead of chown!
sudo chown root:docker /var/run/docker.sock
sudo chmod 660 /var/run/docker.sock

# Change only the group (same as chgrp)
sudo chown :developers /shared/project/

# After running deployment as wrong user — fix ownership
sudo chown -R appuser:appgroup /opt/app/

# Change owner using UID/GID numbers (when name not available)
sudo chown 1001:1002 /data/file.csv
6Interview Questions & Answers
Q1Why does nginx/apache/postgres run as a dedicated user instead of root?
A:Principle of least privilege — if the service is compromised, the attacker gets the permissions of that service user, not root. A compromised www-data account can only read/write files owned by www-data. It cannot access /etc/shadow, other users' home directories, or system files. This containment is fundamental to Linux security design.
Q2After deploying with root, the application gets "Permission denied" errors. How do you fix?
A:The deployment created files owned by root, but the application runs as e.g. www-data. Fix: sudo chown -R www-data:www-data /var/www/html/ then find /var/www/html -type d -exec chmod 755 {} \; && find /var/www/html -type f -exec chmod 644 {} \;
❌
Never run chown -R on / — always specify a full non-root path. Recursive chown from root destroys the entire system's permission structure.
chgrp
Change Group
Change group ownership — collaborate without changing file owner
1Command Overview

chgrp changes only the group ownership of a file — leaving the owner unchanged. While chown :group file achieves the same result, chgrp is more explicit and readable when only the group needs changing. Used for shared project directories where multiple users need access through a common group.

4Practical Examples
BASH
# Change group of a directory to 'developers'
sudo chgrp developers /opt/project/

# Recursive — all files inherit group
sudo chgrp -R dataengineers /data/warehouse/

# Docker socket — allow docker group access
sudo chgrp docker /var/run/docker.sock

# Give data team read access to ML models
sudo chgrp -R datateam /opt/models/
sudo chmod -R g+r /opt/models/

# Equivalent using chown
sudo chown :developers /opt/project/   # same as chgrp developers
💡
chgrp vs chown :group — Both achieve the same result. Use chgrp for clarity when only changing group. Use chown user:group when changing both simultaneously. In scripts, chown :group is preferred for compactness.
umask
User Mask
Set default file creation permissions — what gets removed by default
1Command Overview

umask defines the default permissions REMOVED when creating new files/directories. It's a mask — bits set in umask are subtracted from maximum permissions. Files default to 666, directories to 777, and umask removes the specified bits. umask 022 → files get 644, directories get 755.

2How umask Works
TEXT — umask Calculation
# Max permissions:  Files=666   Directories=777
# Subtract umask:   666 - 022 = 644  (files get rw-r--r--)
#                  777 - 022 = 755  (dirs  get rwxr-xr-x)
#
# Common umask values:
022  → files=644  dirs=755  (standard, most systems)
027  → files=640  dirs=750  (more secure, others get nothing)
077  → files=600  dirs=700  (very secure, private)
002  → files=664  dirs=775  (group collaborative environment)
4Practical Examples
BASH
# Check current umask
umask
# 0022

umask -S   # Symbolic display
# u=rwx,g=rx,o=rx

# Set umask for current session
umask 027   # Files=640, Dirs=750 (more secure)

# Set permanently for a user (add to ~/.bashrc or ~/.profile)
echo "umask 027" >> ~/.bashrc

# System-wide default (affects all users)
# Edit: /etc/profile or /etc/login.defs (UMASK setting)

# Service-specific umask in systemd unit file
# [Service]
# UMask=0027  ← nginx creates files with 640

# DevOps: verify new files have expected permissions
umask 027 && touch testfile && ls -l testfile && rm testfile
# -rw-r----- 1 user user 0 Jan 15 10:00 testfile
6Interview Questions & Answers
Q1What does umask 022 mean and what permissions will new files have?
A:umask 022 means bits 022 (write permissions for group and others) are removed from new files. New files: 666 - 022 = 644 (rw-r--r--). New directories: 777 - 022 = 755 (rwxr-xr-x). This is the Linux standard — owner can read/write, group and others can only read.
Q2Why would a production application run with umask 027 instead of 022?
A:umask 027 prevents "others" (any user not in the file's group) from reading any files created by the application. Files get 640, directories get 750. Critical for: log files containing sensitive data, config files with credentials, financial/healthcare application data. The "others" bits (last digit) become 0 with umask 027.
setfacl
Set ACL
Set Access Control Lists — granular permissions beyond owner/group/other
1Command Overview

setfacl sets Access Control Lists (ACLs) — an extension to standard Linux permissions that allows granting different permissions to multiple specific users/groups on the same file. The traditional model supports only one owner and one group; ACLs allow: "alice gets read, bob gets read+write, devteam gets read, others get nothing."

In production, ACLs are used for: shared NFS mounts where multiple teams need different access, ML model files shared between data scientists and production systems, and compliance environments requiring fine-grained audit trails.

2Syntax
BASH
setfacl [OPTIONS] ACLS FILE...

  ACL format:  [d:]TYPE:NAME:PERMS
    d: = default ACL (new files inherit)
    TYPE: u (user), g (group), o (other), m (mask)
    NAME: username, groupname, or empty for owner/owning-group
    PERMS: rwx combinations
3Key Options
FlagDescriptionExample
-m ACLSModify/add ACL entrysetfacl -m u:alice:rwx file
-x ACLSRemove specific ACL entrysetfacl -x u:alice file
-bRemove ALL ACL entriessetfacl -b file
-RRecursive — apply to directory treesetfacl -Rm u:alice:rx /data/
-dSet default ACL (new files inherit)setfacl -m d:u:alice:rx /dir/
-kRemove default ACL onlysetfacl -k /dir/
4Practical Examples
BASH
# Grant specific user read+write to a file (without changing owner)
setfacl -m u:alice:rw /data/report.csv

# Grant a group read-only access to ML models
setfacl -Rm g:datascientists:rx /opt/models/

# Set default ACL — new files in dir inherit these permissions
setfacl -m d:u:alice:rwx,d:g:devteam:rx /shared/project/

# Grant multiple users different permissions
setfacl -m u:alice:rwx,u:bob:rx,u:charlie:r /data/db.sqlite

# View ACL
getfacl /data/report.csv
# file: data/report.csv
# owner: admin
# group: admin
# user::rw-
# user:alice:rw-     ← ACL entry
# group::r--
# mask::rw-
# other::---

# Remove specific user's ACL
setfacl -x u:alice /data/report.csv

# Remove ALL ACLs (revert to standard permissions)
setfacl -b /data/report.csv

# Copy ACLs from one dir to another
getfacl /source/dir | setfacl --set-file=- /dest/dir
6Interview Questions & Answers
Q1When would you use ACLs instead of traditional permissions?
A:Traditional permissions support ONE owner + ONE group. Use ACLs when: (1) Multiple users need different access levels to the same file/directory — e.g., alice=rwx, bob=rx, charlie=r. (2) A user not in the file's group needs access without changing the group. (3) Default ACLs needed so all new files in a dir automatically get specific permissions. (4) NFS shares accessed by teams with different access needs.
Q2What is the "mask" entry in ACL output?
A:The mask is the maximum effective permissions that can be granted to any named user or group (not the owner). mask::rw- means no matter what you grant via ACL, the effective permission cannot exceed rw-. When you chmod g+x file on an ACL file, it updates the mask. This is why after chmod operations, ACL effective permissions might show "effective:---" or reduced permissions.
⚠️
When you see a + at the end of ls -l permissions (e.g., -rw-rw-r--+), the file has ACLs. Always check with getfacl before assuming only standard permissions apply.
getfacl
Get ACL
Get/display Access Control Lists — audit and backup permissions
1Command Overview

getfacl displays the ACL (Access Control List) for files and directories. It shows standard permissions AND any extended ACL entries. Essential for security auditing, permission debugging, and backing up/restoring ACL configurations.

4Practical Examples
BASH
# View ACL of a file
getfacl /etc/app/config.yaml

# Recursive — view ACLs of all files in directory
getfacl -R /shared/project/ > acl_backup.txt

# Backup and restore ACLs (server migration)
getfacl -R /data/ > data_acls.txt
setfacl --restore=data_acls.txt   # Restore on new server

# Omit comments (clean output for scripting)
getfacl -c file.txt

# Check if a specific user has ACL access
getfacl /data/model.pkl | grep "alice"
6Interview Questions & Answers
Q1How do you backup and restore ACLs when migrating a server?
A:Backup: getfacl -R /data/ > data_acls.txt. Restore on new server: setfacl --restore=data_acls.txt. This preserves all extended permissions that rsync or cp would miss. For complete migration: backup ACLs, copy files with rsync, restore ACLs.
✅
Security audit tip: getfacl -R /sensitive/data/ | grep -v "^#" | grep ":" — shows only actual ACL entries without standard permissions — useful for finding unexpected access grants.

Final Interview Tips — Sections 2 & 3

The #1 gotcha: usermod -G vs usermod -aG — without -a, ALL existing groups are replaced. This is asked in virtually every Linux interview. Always use -aG.

New group doesn't work: Group changes need a new login. Interviewers test this — the answer is newgrp groupname for immediate effect without logout.

Permission model shortcut: 755 for directories/executables, 644 for files, 600 for secrets. Know WHY — directories need x to be traversable, secrets need 600 so only owner reads them.

ACLs in modern environments: Whenever you see "multiple teams need different permissions on the same resource," the answer is ACLs — setfacl -m u:user:rwx. Traditional permissions can't do this elegantly.

Security audit flow: find / -perm -4000 (SUID) → find / -nouser (orphaned) → lastb (failed logins) → sudo -l (who can do what). Know this flow cold.
