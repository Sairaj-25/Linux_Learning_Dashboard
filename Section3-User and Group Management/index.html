useradd
Add User
Create a new user account — low-level, scriptable
1Command Overview

useradd is the low-level command for creating user accounts. Unlike adduser, it doesn't ask interactive questions — making it ideal for automation, scripts, Ansible playbooks, and Dockerfiles. It reads defaults from /etc/default/useradd and /etc/login.defs.

3Commonly Used Options
FlagDescriptionExample
-mCreate home directoryuseradd -m alice
-s SHELLSet login shelluseradd -s /bin/bash alice
-d DIRCustom home directory pathuseradd -d /opt/alice alice
-g GROUPSet primary group (name or GID)useradd -g developers alice
-G GROUPSAdd to supplementary groupsuseradd -G docker,sudo alice
-c COMMENTGECOS field (full name, description)useradd -c "Alice Smith" alice
-u UIDSet specific UIDuseradd -u 1050 alice
-e DATEAccount expiry date (YYYY-MM-DD)useradd -e 2024-12-31 contractor
-f DAYSDays after password expiry to disableuseradd -f 7 alice
-rCreate system account (UID < 1000, no home)useradd -r nginx
-MDo NOT create home directoryuseradd -M -s /sbin/nologin svcuser
4Practical Examples
BASH
# Create standard interactive user
sudo useradd -m -s /bin/bash -c "Alice Smith DevOps" alice

# Create user with groups, set password
sudo useradd -m -s /bin/bash -G docker,sudo,developers alice
sudo passwd alice

# Create service account (no home, no shell, no login)
sudo useradd -r -M -s /sbin/nologin -c "Nginx Web Server" nginx

# Create ML service user with custom UID (for consistency across servers)
sudo useradd -u 2001 -m -s /bin/bash -g mlteam -G docker mlservice

# Create temporary contractor account expiring on Dec 31
sudo useradd -m -s /bin/bash -e 2024-12-31 -c "Temp Contractor" contractor

# Verify user was created
id alice
grep alice /etc/passwd
ls -la /home/alice/
6Interview Questions & Answers
Q1What is the difference between useradd and adduser?
A:useradd is the low-level binary — no prompts, no defaults, exactly what you specify. Perfect for automation. adduser is a high-level Perl/shell script (Debian/Ubuntu) that wraps useradd — it prompts interactively, automatically creates home directory, sets skeleton files, and is more user-friendly. For scripts and Ansible: use useradd. For manual one-off creation on Debian: use adduser.
Q2How do you create a service account that cannot log in interactively?
A:sudo useradd -r -M -s /sbin/nologin -c "Service Account" servicename. The -r flag creates a system account (UID below 1000), -M skips home directory creation, and -s /sbin/nologin prevents interactive login. If someone tries to su - servicename they get "This account is currently not available."
✅
Defaults file: View/edit useradd defaults with useradd -D or cat /etc/default/useradd. Set default shell, home base dir, and expiry here to ensure consistency across all new accounts.
adduser
Add User (Interactive)
High-level user creation — interactive, sets up home with skeleton files
1Command Overview

adduser is a user-friendly wrapper (Debian/Ubuntu) over useradd. It prompts for password, full name, and other details, automatically creates the home directory with skeleton files from /etc/skel, and is the recommended command for manual user creation on Debian systems.

4Practical Examples
BASH
# Interactive user creation (Debian/Ubuntu)
sudo adduser alice
# Adding user 'alice' ...
# Adding new group 'alice' (1001) ...
# Adding new user 'alice' (1001) with group 'alice' ...
# Creating home directory '/home/alice' ...
# Copying files from '/etc/skel' ...
# New password: [prompt]
# Full Name []: Alice Smith
# ...

# Add existing user to a group using adduser
sudo adduser alice docker
# "Adding user alice to group docker"

# Non-interactive adduser (via --quiet + GECOS data)
echo "alice:password:::" | sudo adduser --gecos "" --quiet --disabled-password alice
💡
On RHEL/CentOS, adduser is a symlink to useradd. On Debian/Ubuntu it's a separate script. Always check which system you're on. For cross-platform scripts, use useradd.
userdel
Delete User
Remove user account — with or without home directory
1Command Overview

userdel removes a user account from the system. By default it only removes the account from /etc/passwd and /etc/shadow — the home directory is preserved. Use -r to also remove the home directory and mail spool.

3Key Options
FlagDescriptionExample
-rRemove home directory and mail spooluserdel -r alice
-fForce — even if user is logged inuserdel -f alice
-ZRemove SELinux user mappinguserdel -Z alice
4Practical Examples
BASH
# Remove user but KEEP home directory (data preserved)
sudo userdel alice

# Remove user AND home directory (complete cleanup)
sudo userdel -r alice

# Production best practice: lock first, delete later
sudo usermod -L alice          # Lock account first
sudo usermod -s /sbin/nologin alice  # Disable shell
# Wait for confirmation no data is needed...
sudo userdel -r alice          # Then delete

# Find files owned by deleted user (orphaned files)
sudo find / -nouser -ls 2>/dev/null
sudo find / -user 1001 -ls 2>/dev/null  # By UID if user gone
6Interview Questions & Answers
Q1An employee leaves the company. What's the proper procedure before deleting their account?
A:1. Lock account immediately: usermod -L username. 2. Change shell to nologin: usermod -s /sbin/nologin username. 3. Archive home directory: tar -czf /backup/username_$(date +%Y%m%d).tar.gz /home/username/. 4. Transfer ownership of any shared files. 5. Revoke SSH keys and sudo access. 6. After data retention period: userdel -r username. 7. Find orphaned files: find / -nouser.
usermod
Modify User
Modify existing user accounts — the most versatile user admin command
1Command Overview

usermod modifies an existing user account's properties. It's the Swiss Army knife of user administration — changing username, shell, groups, home directory, and account lock/unlock. The most commonly tested in interviews is the -aG distinction.

3Commonly Used Options

Group Management

-G GROUPSSET supplementary groups (REPLACES all existing!)usermod -G docker,sudo alice
-g GROUPChange primary groupusermod -g developers alice

-l NEWNAMERename user (login name only)usermod -l alice_smith alice
-d DIRChange home directoryusermod -d /newhome/alice alice
-mMove home directory contents to new locationusermod -d /newhome -m alice
-s SHELLChange login shellusermod -s /bin/zsh alice
-c COMMENTChange GECOS comment fieldusermod -c "Alice Smith Lead" alice
-u UIDChange UIDusermod -u 1100 alice
Account Status
-LLOCK account (prepends ! to password hash)usermod -L alice
-UUNLOCK account (removes ! prefix)usermod -U alice
-e DATESet account expiry dateusermod -e 2024-12-31 alice
-f DAYSDays after password expiry before disableusermod -f 14 alice
4Practical Examples
BASH
# Add user to docker group (CRITICAL: always use -aG not -G!)
sudo usermod -aG docker alice
# New group takes effect on NEXT login
# For current session: newgrp docker

# Add user to multiple groups at once
sudo usermod -aG docker,sudo,developers,databricks alice

# Lock account (employee on leave)
sudo usermod -L alice
sudo passwd -S alice   # Verify: shows 'L' for locked

# Unlock account
sudo usermod -U alice

# Change shell (e.g., after zsh install)
sudo usermod -s /usr/bin/zsh alice

# Disable interactive login for service account
sudo usermod -s /sbin/nologin nginx

# Rename user (login only, home dir still old name)
sudo usermod -l alice_smith alice
# Rename home too:
sudo usermod -l alice_smith -d /home/alice_smith -m alice

# Set account expiry for contractor
sudo usermod -e 2024-06-30 contractor_bob
6Interview Questions & Answers
Q1What is the critical difference between usermod -G and usermod -aG?
A:This is one of the most commonly tested Linux interview questions. -G groups REPLACES all supplementary groups — if alice is in docker,sudo,developers and you run usermod -G databricks alice, she's now ONLY in databricks. She lost docker, sudo, and developers! -aG groups APPENDS — adds to existing groups. ALWAYS use -aG when adding a user to a group.
Q2You added alice to the docker group but she still gets "permission denied." Why?
A:Group changes take effect at NEXT login — the current session still has the old group membership. Solutions: (1) Log out and log back in. (2) newgrp docker — creates a new shell with the new group active. (3) su - alice — refresh session. Verify groups in current session: id.
passwd
Password
Change user password — lock, unlock, expire accounts
1Command Overview

passwd changes user passwords and manages password states. Regular users can change their own password; root can change any password. The command reads from and writes to /etc/shadow (hashed passwords) via SUID mechanism.

3Key Options
FlagDescriptionExample
-lLock account (prepend ! to hash)passwd -l alice
-uUnlock account (remove !)passwd -u alice
-dDelete password (passwordless — dangerous!)passwd -d alice
-eExpire immediately — force change at next loginpasswd -e alice
-SShow password statuspasswd -S alice
-n DAYSMinimum days before password can be changedpasswd -n 7 alice
-x DAYSMaximum days before password must be changedpasswd -x 90 alice
-w DAYSWarning days before expirypasswd -w 14 alice
-i DAYSInactivity days after expiry before lockingpasswd -i 30 alice
4Practical Examples
BASH
# Change own password
passwd

# Root changes alice's password
sudo passwd alice

# Set password non-interactively (for automation/scripts)
echo 'alice:SecureP@ss123' | sudo chpasswd
# chpasswd reads user:password pairs from stdin

# Force password change at next login (new employee onboarding)
sudo passwd -e alice

# Lock account
sudo passwd -l alice

# Check password status
sudo passwd -S alice
# alice P 2024-01-10 0 90 14 30
#       ^                        
# P=has password, L=locked, NP=no password

# Set 90-day max age, 7-day warning
sudo passwd -x 90 -w 7 alice
6Interview Questions & Answers
Q1How does the passwd command work even for non-root users to modify /etc/shadow?
A:/usr/bin/passwd has the SUID bit set (ls -l /usr/bin/passwd shows -rwsr-xr-x). When any user runs it, it executes as root (the file owner), allowing it to write to /etc/shadow. It validates that you can only change your own password (unless you are root). This is the SUID mechanism in practice.
✅
Non-interactive password setting: Use chpasswd for automation — echo 'user:password' | sudo chpasswd. This avoids the interactive prompt. In Ansible: use the password_hash filter instead of cleartext.
chage
Change Age
Password expiry and account aging policy — compliance and security
1Command Overview

chage (Change Age) manages user password aging policies. It sets when passwords expire, how long they're valid, warning periods, and when accounts become inactive. Critical for security compliance (PCI-DSS, HIPAA, SOC2) requiring regular password rotation.

3Key Options
FlagDescriptionExample
-lList current aging infochage -l alice
-M DAYSMaximum password age (days before forced change)chage -M 90 alice
-m DAYSMinimum password age (can't change before N days)chage -m 7 alice
-W DAYSWarning days before expirychage -W 14 alice
-I DAYSInactive days after expiry before account lockschage -I 30 alice
-E DATEAccount expiry date (YYYY-MM-DD or -1 for never)chage -E 2024-12-31 alice
-d DATELast password change date (0 = force change now)chage -d 0 alice
-iInteractive mode — prompts for all valueschage alice
4Practical Examples
BASH
# View complete password aging info
sudo chage -l alice
# Last password change           : Jan 10, 2024
# Password expires                : Apr 09, 2024   (90 days)
# Password inactive               : May 09, 2024   (30 days after expiry)
# Account expires                 : Dec 31, 2024
# Minimum days between changes    : 7
# Maximum days between changes    : 90
# Number of days of warning       : 14

# Set 90-day max, 7-day min, 14-day warning (security policy)
sudo chage -M 90 -m 7 -W 14 alice

# Force password change at next login (new employee)
sudo chage -d 0 alice

# Set account to never expire
sudo chage -E -1 alice

# Apply policy to all users (for compliance)
for user in $(getent passwd | awk -F: '$3 >= 1000 && $3 != 65534 {print $1}'); do
    sudo chage -M 90 -W 14 $user
done
6Interview Questions & Answers
Q1How do you enforce a 90-day password policy across all users for compliance?
A:Two approaches: (1) chage per user. (2) Set system default in /etc/login.defs — set PASS_MAX_DAYS 90, PASS_MIN_DAYS 7, PASS_WARN_AGE 14 — these apply to NEW accounts. For existing accounts, loop with chage -M 90 username. Then audit compliance: for user in $(cut -d: -f1 /etc/passwd); do chage -l $user; done | grep "Password expires"
id & whoami
Identity
Display user and group identity — scripting and verification
1Command Overview

id prints the UID, GID, and all group memberships of the current or specified user. whoami simply prints the effective username. Both are essential in scripts to verify execution context, and in debugging to confirm user identity after su or sudo.

4Practical Examples
BASH
# id — show full identity including all groups
id
# uid=1001(alice) gid=1001(alice) groups=1001(alice),4(adm),27(sudo),999(docker)

id alice         # Check another user's identity
id -u            # Print only UID (useful in scripts)
id -g            # Print only GID
id -G            # Print all group IDs (numeric)
id -Gn           # Print all group NAMES

# whoami — effective username
whoami
# alice

# Script pattern: check if running as root
if [[ $(id -u) -ne 0 ]]; then
    echo "ERROR: This script must run as root"
    exit 1
fi

# Verify user has docker group before running docker command
if ! id -Gn | grep -q "docker"; then
    echo "Not in docker group — run: sudo usermod -aG docker $USER"
    exit 1
fi
6Interview Questions & Answers
Q1What's the difference between id and whoami?
A:whoami prints the effective username. id prints comprehensive identity: UID, primary GID, and ALL supplementary groups. In scripts, use id -u to check if root (UID=0), and id -Gn to check group membership without needing sudo.
su & sudo
Switch User / SuperUser Do
Elevate privileges — su switches user, sudo executes as another user
1Command Overview

su (Switch User) switches to another user account — you need THAT user's password (except root can su to anyone). sudo (SuperUser Do) executes a command with elevated privileges — you need YOUR OWN password, and access is controlled by /etc/sudoers. In modern production: prefer sudo over su — it audits every command in /var/log/auth.log.

3Key Options
CommandDescriptionExample
su — Switch User
su usernameSwitch to user (environment NOT reset)su alice
su - usernameLogin shell — full environment of target usersu - alice
su -Switch to root with full root environmentsu -
su -c CMD userExecute command as user without interactive shellsu -c "id" alice
sudo — Execute as Another User
sudo CMDRun command as rootsudo apt-get update
sudo -u USER CMDRun command as specific usersudo -u postgres psql
sudo -iInteractive root shell (full environment)sudo -i
sudo -sShell as root (current environment)sudo -s
sudo -lList what sudo commands current user can runsudo -l
sudo -kInvalidate sudo timestamp (require password again)sudo -k
sudo !! Re-run last command with sudosudo !!
visudoEdit sudoers file safely (with syntax check)sudo visudo
4Practical Examples
BASH
# su examples
su - alice          # Full login as alice (prompts for alice's password)
su -                # Switch to root (prompts for root password)

# sudo basic usage
sudo apt-get update
sudo systemctl restart nginx

# Run as postgres user (common for DB admin)
sudo -u postgres psql -d mydb

# Get root shell

# View what you can run with sudo
sudo -l

#     (ALL : ALL) NOPASSWD: /usr/bin/systemctl
#     (ALL : ALL) /usr/bin/apt-get
6Interview Questions & Answers
Q1What is the difference between su - and su (without dash)?
A:su username switches user but inherits the current user's environment (HOME, PATH, etc.). su - username (login shell) creates a completely fresh environment as if you logged in as that user — loads their ~/.bashrc, ~/.profile, sets their HOME, PATH, and all environment variables. In production, always use su - to avoid environment conflicts that cause "command not found" errors.
Q2Why should you always use visudo to edit the sudoers file?
A:visudo validates the sudoers syntax before saving. If you directly edit /etc/sudoers with vi/nano and introduce a syntax error, ALL sudo access is broken — no one can use sudo, potentially locking everyone out of root access. visudo catches errors before committing. Also, if two admins edit simultaneously, visudo uses file locking to prevent corruption.
Q3How do you grant a user sudo access for specific commands only?
A:">sudo visudo then add: alice ALL=(ALL) NOPASSWD: /usr/bin/systemctl, /usr/bin/docker. This allows alice to run systemctl and docker with sudo, without a password prompt. The NOPASSWD is optional — remove it to require alice's password. This follows the principle of least privilege — alice can restart services and manage containers, but cannot rm system files or modify /etc/passwd.
❌
Never share root password — use sudo instead. sudo creates an audit trail in /var/log/auth.log with exact commands, timestamps, and usernames. Shared root password has no accountability.
last & who
Login History
Track user logins — audit trail and active session monitoring
1Command Overview

last shows login history from /var/log/wtmp — who logged in, from where, and for how long. who shows currently logged-in users. Both are critical for security auditing, detecting unauthorized access, and compliance reporting.

4Practical Examples
BASH
═══ last — Login history ═══
last                     # All recent logins
last alice               # Alice's login history
last -n 20              # Last 20 logins
last -F                  # Full date/time format
last -a                  # Show hostname in last column
# alice    pts/1  10.0.1.50   Mon Jan 15 09:30  still logged in
# bob      pts/0  10.0.1.25   Mon Jan 15 08:00 - 09:00 (01:00)

lastb                    # Failed login attempts (bad logins)
lastlog                  # Most recent login for each user

═══ who — Current logins ═══
who                      # Currently logged-in users
# alice    pts/1  2024-01-15 09:30 (10.0.1.50)
# bob      pts/0  2024-01-15 08:00 (10.0.1.25)

who am i                 # Current login session info
w                        # More detailed: shows what users are running
# USER  TTY   FROM      LOGIN@  IDLE  JCPU  PCPU  WHAT
# alice pts/1 10.0.1.50 09:30   5:00  0.05  0.02  vim config.yaml

# Security audit: detect logins outside business hours
last -F | grep -E "[0-1][0-9]:00|2[0-3]:[0-9]"

# Find all users who logged in from external IPs
last -a | grep -v "^reboot\|^shutdown\|10\.0\.\|192\.168\."
6Interview Questions & Answers
Q1How do you find all failed login attempts for security auditing?
A:sudo lastb — shows bad (failed) login attempts from /var/log/btmp. Also: grep "Failed password" /var/log/auth.log | awk '{print $(NF-3)}' | sort | uniq -c | sort -rn | head -20 — finds top IPs with failed SSH attempts (potential brute force). For blocking: fail2ban automates this.

Groups are the core mechanism for granting shared access to resources. Services, teams, and functional roles map to Linux groups. A user can be in multiple groups simultaneously, and their effective access is the union of all their groups' permissions.

groups · newgrp · gpasswd
Group Tools
List groups, switch active group, manage group passwords & members
1Command Overview
  • groups: Displays all groups a user belongs to — simple output, quick check.
  • newgrp: Switches the active/effective group for the current session — new group is immediately active without logging out. Creates a new shell with that group effective.
  • gpasswd: Administers the /etc/group and /etc/gshadow files — set group passwords, add/remove members, designate group administrators.
4Practical Examples
BASH
═══ groups — List group memberships ═══
groups                   # Current user's groups
# alice : alice adm sudo docker developers

groups alice             # Specific user's groups
groups alice bob charlie  # Multiple users at once

═══ newgrp — Switch active group ═══
# Problem: added to docker group, but current session doesn't know
id  # docker not shown — need to re-login or...
newgrp docker          # Immediately activates docker group!
id  # Now shows docker in groups
docker ps              # Works now without logout
# exit  ← returns to previous shell

═══ gpasswd — Manage group membership ═══
# Add user to group (alternative to usermod -aG)
sudo gpasswd -a alice developers

# Remove user from group
sudo gpasswd -d bob developers

# Add group administrator (can manage group members)
sudo gpasswd -A alice developers   # alice can now add/remove devs

# Set group password (rarely used — prefer sudo/ACLs)
sudo gpasswd developers

# View group members
getent group developers
# developers:x:1050:alice,bob,charlie,diana
6Interview Questions & Answers
Q1How do you immediately apply a new group membership without logging out?
A:Two methods: (1) newgrp groupname — creates a new shell with the group active immediately. Type exit to return to previous shell. (2) exec su - $USER — restarts the login shell with all new group memberships. Or simply log out and back in. In scripts: use sg groupname -c "command" to run a command with a specific group active.
Q2What is the difference between adding a user with gpasswd -a vs usermod -aG?
A:Both add a user to a supplementary group, but the mechanism differs. usermod -aG group user modifies /etc/passwd group supplementary field. gpasswd -a user group modifies the /etc/group member list directly. The net result is identical. gpasswd also supports group administrators who can manage group membership without root — via gpasswd -A.
