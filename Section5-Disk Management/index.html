lsblk
Block Device Listing
List block devices in a tree-like format — disks, partitions, LVM, RAID
1Command Overview

lsblk reads the /sys virtual filesystem to list all block devices (hard drives, SSDs, NVMe, USB drives, LVM volumes, RAID arrays) in a hierarchical tree. It's the fastest, safest way to get a full picture of storage topology without any disk access.

In DevOps and cloud engineering, lsblk is the first command run after attaching a new EBS volume, NVMe disk, or SAN LUN — to confirm the kernel has recognised it and to identify its device name before partitioning or formatting.

2Syntax
BASH
lsblk [OPTIONS] [device...]

  device  — Optional: restrict output to specific device (e.g., /dev/sda)
  OPTIONS — Flags controlling output format and columns
3Commonly Used Options & Flags
FlagDescriptionExample
-aShow all devices including empty oneslsblk -a
-fShow filesystem type, UUID, and mount pointlsblk -f
-o COLSSpecify output columnslsblk -o NAME,SIZE,TYPE,MOUNTPOINT
-pPrint full device paths (/dev/sda1 not sda1)lsblk -p
-nNo header line (clean for scripting)lsblk -n
-dDon't print partitions — disks onlylsblk -d
-e LISTExclude device types by major numberlsblk -e 7 (exclude loopback)
-JJSON output (for automation)lsblk -J | jq '.blockdevices'
-bPrint sizes in byteslsblk -b -o NAME,SIZE
-sInverse — show device dependencies (slaves)lsblk -s /dev/sda1

5Real-World Use Cases
  • New volume verification: After attaching AWS EBS, GCP Persistent Disk, or Azure Managed Disk — lsblk confirms the device name before partitioning.
  • LVM topology audit: See which physical disks back which VGs and LVs in a single tree view.
  • CI/CD disk provisioning scripts: lsblk -J output is parsed by Python/Ansible to auto-detect and mount available data volumes.
  • Kubernetes node setup: Verify NVMe local storage is available for local-path-provisioner before joining the cluster.
6Interview Questions & Answers
Q1What is the difference between lsblk and fdisk -l?
A:lsblk reads from /sys — it's read-only, fast, and safe to run as any user (with sudo for full detail). It shows device topology including LVM and RAID. fdisk -l opens each device file and reads partition tables — requires root, may wake sleeping disks, but shows partition table details (type, sectors, start/end) that lsblk doesn't. Use lsblk first for overview, fdisk -l for partition-level details.
Q2You attach an EBS volume to an EC2 instance. How do you find its device name?
A:Run lsblk before and after attaching, or simply lsblk -d -o NAME,SIZE,MODEL after attachment. AWS attaches NVMe EBS volumes as nvme1n1, nvme2n1, etc. The AWS console shows the device name (e.g., /dev/sdf) but the kernel may map it to /dev/nvme1n1. Always use lsblk to confirm the actual kernel device name.
7Common Mistakes & Pro Tips
✅
Pro Tip — lsblk -f before any disk operation Always run lsblk -f before formatting — confirms there's no existing filesystem (FSTYPE blank). Formatting a disk with data loses everything.
💡
Column Reference Most useful columns: NAME,SIZE,TYPE,FSTYPE,UUID,LABEL,MOUNTPOINT,MODEL,SERIAL,ROTA — ROTA=0 means SSD/NVMe, ROTA=1 means HDD (rotational).
fdisk
Partition Table Manipulation
Interactive partition table manipulator for MBR and GPT disks
1Command Overview

fdisk is an interactive text-based partition editor. It reads and modifies partition tables on block devices — primarily MBR (Master Boot Record, max 2TB, max 4 primary partitions) and GPT (GUID Partition Table, supports disks >2TB, up to 128 partitions).

In production environments, fdisk is used to prepare raw disks before filesystem creation — carving out partitions for OS boot, swap, data volumes, and LVM physical volumes. For disks >2TB, use parted or gdisk instead.

🚨
DESTRUCTIVE — No Undo! fdisk writes changes directly to the partition table. Deleting a partition loses ALL data on it. Always verify the device name twice. In production, schedule maintenance windows and have backups before any fdisk operation.
2Syntax
BASH
fdisk [OPTIONS] DEVICE

Read-only (safe):
  fdisk -l              # List all disk partition tables
  fdisk -l /dev/sda     # List specific disk

Interactive (requires root):
  sudo fdisk /dev/sdb   # Open fdisk for /dev/sdb

gCreate new GPT tableDestroys existing table!

5Real-World Use Cases
  • Provisioning data nodes: New bare-metal Kafka, Elasticsearch, or database servers receive raw disks — fdisk creates LVM partitions before volume group setup.
  • Boot partition creation: Creating /boot and /boot/efi partitions when reinstalling RHEL/Ubuntu on physical hardware.
  • Swap partition setup: Create a dedicated swap partition (type 82) on servers with memory-intensive ML model inference workloads.
6Interview Questions & Answers
Q1What is the difference between MBR and GPT partition tables?
A:MBR (Master Boot Record): Legacy format. Max disk size 2TB. Max 4 primary partitions (or 3 primary + 1 extended with logical partitions inside). GPT (GUID Partition Table): Modern standard. Supports disks up to 9.4ZB. Up to 128 partitions. Has backup partition table at disk end. Required for UEFI boot. Use GPT for all new deployments — fdisk now defaults to GPT on new disks.
Q2What happens if you press 'w' in fdisk by mistake?
A:The partition table is written to disk immediately and is irreversible. Data recovery may be possible with tools like testdisk if the data hasn't been overwritten, but there are no guarantees. Prevention: always use p to print and verify the partition layout before pressing w. In production, take a partition table backup first: sfdisk -d /dev/sda > partition_backup.txt.
Q3When would you use parted instead of fdisk?
A:Use parted when: (1) disk size >2TB — fdisk has limitations with very large disks, (2) you need scripted/non-interactive partitioning — parted supports one-liner commands, (3) you need to resize an existing partition without deleting it, (4) you need to align partitions optimally for SSDs/NVMe (parted has optimal alignment checks). fdisk is fine for typical server disks under 2TB.
7Common Mistakes & Pro Tips
❌
Mistake — Wrong device name fdisk /dev/sda on a production OS disk is catastrophic. Triple-check the device: lsblk, fdisk -l, compare sizes. In automated scripts, add device validation: assert device is not mounted as root.
✅
Pro Tip — Backup partition table first sudo sfdisk -d /dev/sda > /backup/sda_partition_table.txt — saves the partition table as a script that can be replayed with sfdisk /dev/sda < partition_backup.txt.
⚠️
partprobe after fdisk After writing changes, run sudo partprobe /dev/sdX to signal the kernel to re-read the partition table without requiring a reboot. Without this, the new partitions won't be visible to the OS.
parted
Advanced Partition Manipulation
GNU parted — supports GPT, large disks, scripting, and partition resizing
1Command Overview

parted is a more advanced partition editor than fdisk. Key advantages: supports disks >2TB, can resize partitions without data loss (for some filesystems), works in both interactive and non-interactive (scriptable) modes, and performs partition alignment verification critical for SSD performance.

In modern DevOps workflows, parted is preferred over fdisk for cloud disk provisioning scripts because it accepts commands as arguments — enabling fully automated, non-interactive partition creation in Terraform, Ansible, and cloud-init user data.

2Syntax
BASH
# Interactive mode
sudo parted /dev/sdb

# Non-interactive (scripted) mode — preferred in automation
sudo parted -s /dev/sdb COMMAND [ARGUMENTS]
sudo parted --script /dev/sdb mklabel gpt

  -s / --script  — Non-interactive: don't prompt for confirmation
  -a optimal    — Align partitions to optimal boundaries (SSD-friendly)
3Key parted Subcommands
CommandDescriptionExample
printShow partition tableparted /dev/sda print
mklabel TYPECreate new partition table (gpt/msdos)parted -s /dev/sdb mklabel gpt
mkpartCreate a partitionparted -s /dev/sdb mkpart primary ext4 0% 100%
rm NUMDelete partition number NUMparted /dev/sdb rm 2
resizepartResize a partition (expand end)parted /dev/sda resizepart 3 100%
set NUM FLAGSet partition flag (boot, lvm, swap, raid)parted -s /dev/sdb set 1 lvm on
align-checkVerify partition is optimally alignedparted /dev/sdb align-check optimal 1
unitSet unit (s=sectors, MB, GB, %)parted /dev/sdb unit MB print
4Practical Examples
BASH — Automated disk setup (cloud-init / Ansible)
# Full disk setup in ONE scriptable line (no prompts)
sudo parted -s /dev/sdb \
    mklabel gpt \
    mkpart primary ext4 0% 100%
sudo partprobe /dev/sdb
sudo mkfs.ext4 /dev/sdb1

# Create 3 partitions: boot (1G), swap (8G), data (rest)
sudo parted -s -a optimal /dev/sdb \
    mklabel gpt \
    mkpart primary fat32 1MiB 1GiB \
    mkpart primary linux-swap 1GiB 9GiB \
    mkpart primary ext4 9GiB 100% \
    set 1 esp on \
    set 2 swap on \
    set 3 lvm on

# Expand last partition to fill disk (after disk resize)
sudo parted /dev/sda resizepart 3 100%
sudo resize2fs /dev/sda3          # Then grow filesystem

# Print partition table in machine-readable format
sudo parted /dev/sda print machine
6Interview Questions & Answers
Q1Scenario: You have an AWS EC2 instance and expanded the root EBS volume from 20GB to 50GB. How do you use the extra space?
A:Three steps: 1. Expand the partition: sudo growpart /dev/xvda 1 (or use parted /dev/xvda resizepart 1 100%). 2. Grow the filesystem: sudo resize2fs /dev/xvda1 (ext4) or sudo xfs_growfs / (XFS). 3. Verify: df -h /. No reboot required — can be done live on a running system.
7Common Mistakes & Pro Tips
✅
Pro Tip — Always use -a optimal parted -a optimal aligns partitions to 1MiB boundaries, which aligns to SSD erase blocks and NVMe optimal I/O boundaries. Misaligned partitions can lose 50%+ of SSD performance.
⚠️
parted mkpart vs fdisk n parted's mkpart does NOT automatically create a filesystem — it only creates the partition entry. Always run mkfs after parted. A common mistake is assuming the partition is formatted after mkpart.
blkid
Block Device Identification
Locate and display attributes (UUID, LABEL, TYPE) of block devices
1Command Overview

blkid queries block devices and prints their attributes: UUID (Universally Unique Identifier), filesystem type, partition label, and partition UUID. The UUID is a persistent identifier — unlike device names (/dev/sda1) which can change if disks are added or reordered, UUIDs stay constant.

In DevOps, blkid is the standard tool for getting the UUID before writing /etc/fstab entries. Using UUID in fstab is the production best practice — prevents boot failures from device name changes after disk additions or kernel updates.

2Syntax & Key Flags
BASH
blkid [OPTIONS] [DEVICE...]

  No args   — List ALL block devices with attributes
  DEVICE    — Query specific device only
FlagDescriptionExample
-o FORMATOutput format: full, value, list, udev, exportblkid -o value -s UUID /dev/sda1
-s TAGShow specific tag only (UUID, TYPE, LABEL)blkid -s TYPE /dev/sda1
-t TAG=VALSearch for device with specific attributeblkid -t TYPE=ext4
-pLow-level probing (bypass cache)blkid -p /dev/sda1
-u LISTFilter by usage: filesystem,raid,cryptoblkid -u filesystem
4Practical Examples
BASH
# Show all block device attributes
sudo blkid
# /dev/sda1: UUID="a1b2c3d4-e5f6-..." TYPE="ext4" PARTUUID="..."
# /dev/sda2: UUID="8765-4321"          TYPE="vfat" PARTLABEL="EFI"

# Get ONLY the UUID of a specific partition (for fstab scripting)
sudo blkid -s UUID -o value /dev/sdb1
# a1b2c3d4-e5f6-7890-abcd-ef1234567890

# Find all ext4 filesystems
sudo blkid -t TYPE=ext4

# Script: auto-generate fstab entry for new data partition
UUID=$(sudo blkid -s UUID -o value /dev/sdb1)
echo "UUID=${UUID} /data ext4 defaults,nofail 0 2" | \
    sudo tee -a /etc/fstab
6Interview Questions & Answers
Q1Why should /etc/fstab use UUID instead of device names like /dev/sda1?
A:Device names (/dev/sda1) are assigned by the kernel based on detection order and can change — adding a second disk might rename sda to sdb causing boot failure. UUIDs are embedded in the filesystem itself and never change regardless of disk position. This is especially critical in cloud environments where EBS/block device names can change after snapshots or instance type changes. Always use UUID= or LABEL= in fstab.
7Common Mistakes & Pro Tips
✅
Pro Tip — Scripting fstab entries The pattern UUID=$(blkid -s UUID -o value /dev/sdX1) extracts the UUID as a clean string for automation. Combine with echo "UUID=$UUID /mount ext4 defaults,nofail 0 2" | tee -a /etc/fstab for fully automated mount setup in cloud provisioning scripts.
df
Disk Free / Filesystem Space
Report filesystem disk space usage — mounted filesystems overview
1Command Overview

df (disk free) reports the amount of disk space used and available on all mounted filesystems. It queries the kernel's mount table — showing total capacity, used space, available space, and mount points. Unlike du, df shows filesystem-level totals, not individual file/directory sizes.

In production, df is checked in monitoring scripts, Prometheus exporters, cron jobs, and on-call runbooks. "Disk full" incidents (df showing 100% usage) are among the most common production outages — they cause application crashes, database corruption, and complete service loss.

2Syntax
BASH
df [OPTIONS] [FILE or FILESYSTEM...]

  No args   — Show all mounted filesystems
  FILE      — Show filesystem containing that file
  FILESYSTEM— Show specific mount point or device
3Commonly Used Flags
FlagDescriptionExample
-hHuman-readable (KB, MB, GB, TB)df -h
-HHuman-readable using powers of 1000 (not 1024)df -H
-TShow filesystem type (ext4, xfs, tmpfs, nfs...)df -Th
-iShow inode usage instead of block usagedf -i
-t TYPEShow only filesystems of specific typedf -t ext4
-x TYPEExclude filesystem type from outputdf -x tmpfs -x devtmpfs
-aInclude pseudo/virtual filesystemsdf -a
--totalAdd grand total row at bottomdf -h --total
-PPOSIX output (stable for scripting)df -P | awk 'NR>1'
4Practical Examples
Standard Usage
BASH
# Human-readable overview — most common command
df -h
# Filesystem      Size  Used Avail Use% Mounted on
# /dev/sda2        99G   45G   49G  48% /
# /dev/sda1       976M  151M  759M  17% /boot
# tmpfs            16G     0   16G   0% /dev/shm

Automation & Monitoring
BASH — Production disk monitoring script
# Alert if any filesystem is over 85% full
df -h | awk 'NR>1 && $5+0 > 85 {
    printf "ALERT: %s is %s full (threshold: 85%%)\n", $6, $5
}'

# Get specific filesystem usage as integer (for scripts)
DISK_USAGE=$(df -P / | awk 'NR==2 {gsub(/%/,"",$5); print $5}')
if [[ ${DISK_USAGE} -gt 90 ]]; then
    echo "CRITICAL: Root disk ${DISK_USAGE}% full!"
    # send_pagerduty_alert
fi

df -Th | grep -E '^/dev/' | sort -k6

# Total used space across all real filesystems

5Real-World Use Cases
  • Production monitoring: Prometheus node_exporter exposes node_filesystem_avail_bytes (sourced from df). Alert rules fire when df would show >85% used.
  • ML training server health checks: Training jobs write large checkpoint files — df -h /checkpoints in cron scripts alert before disk fills up and corrupts the checkpoint.
  • Log rotation triggers: /var/log filling up is a common production issue. df-based alerting triggers logrotate or cleanup before the filesystem fills completely.
  • Container host monitoring: Docker's image/overlay filesystem can fill /var/lib/docker — df -h /var/lib/docker is the first check when Kubernetes pods fail to start.
6Interview Questions & Answers
Q1Scenario: df shows disk is 100% full but du shows much less usage. What could cause this?
A:This is the classic deleted-but-open files problem. When a process deletes a file but still has it open (via a file descriptor), the OS marks the inode as deleted but doesn't release the disk blocks until the file descriptor is closed. du counts only directory-linked files (deleted files aren't counted), while df shows actual block usage including these "ghost" files. Fix: Find the process holding the open file: lsof | grep deleted | sort -k7 -rn | head. Either restart the process or wait for it to close the file.
Q2df shows 45% used but the system is reporting "No space left on device". Why?
A:Inode exhaustion. Run df -i — if IUse% shows 100%, the inode table is full even though disk blocks are available. Each file requires one inode. This happens when a process creates millions of small files (log entries, temp files, email spool). Fix: Find directory with millions of files: find / -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head. Delete small files to free inodes. Long-term fix: format with more inodes using mkfs.ext4 -N <count> or switch to XFS which has dynamic inodes.
Q3
Q3:What is the difference between df and du?
A:df queries the filesystem metadata (kernel mount table) — it reports at the filesystem/partition level including reserved blocks and overhead, instantly. du actually traverses the directory tree counting file sizes — it works at the file/directory level and can be slow on large filesystems. Use df to check overall filesystem fullness; use du to find WHAT is consuming the space.
7Common Mistakes & Pro Tips
❌
Mistake — Ignoring inode usage df -h shows only block usage by default. Always check df -i when debugging "no space" errors — inode exhaustion is a common production gotcha, especially on systems processing millions of small files.
✅
Pro Tip — Reserved space By default, mkfs.ext4 reserves 5% of disk space for root. On large data disks (1TB+), this wastes 50GB. Reduce with tune2fs -m 1 /dev/sdX1 (reduce to 1%). Only do this on data disks, not root filesystem.
du
Disk Usage — File/Directory Level
Estimate file space usage — find what is consuming disk space
1Command Overview

du (disk usage) traverses the directory tree and reports the disk space consumed by files and directories. While df tells you a filesystem is full, du tells you what is full — pinpointing the directories consuming the most space. It is the essential "disk space forensics" tool.

In AI/ML engineering, du is critical for managing model checkpoints, training datasets, Docker image caches, and Python package environments that frequently grow to hundreds of GBs.

2Syntax
BASH
du [OPTIONS] [FILE or DIRECTORY...]

  No args     — Summarize current directory recursively
  DIRECTORY   — Summarize given path recursively
3Commonly Used Flags
FlagDescriptionExample
-hHuman-readable (K, M, G)du -h /var
-sSummary — total only for each argument (no recursion)du -sh /var/log
-aShow all files, not just directoriesdu -ah /etc | sort -h
-cAdd grand total at the enddu -shc /home/*
--max-depth=NLimit recursion to N levelsdu -h --max-depth=2 /var
-d NSame as --max-depth (shorthand)du -hd1 /
--exclude=PATExclude files matching patterndu -sh --exclude='*.pyc' /app
-xStay on same filesystem (don't cross mount points)du -sx /
--apparent-sizeShow apparent size, not disk usage (for sparse files)du -sh --apparent-size file
-lCount hardlinked files multiple timesdu -hl /data
4Practical Examples
Finding Disk Hogs — The Core Use Case
BASH
# TOP 10 largest directories in /var (sorted)
sudo du -h --max-depth=2 /var | sort -rh | head -20

# Summary of ALL top-level directories — find biggest
sudo du -hd1 / 2>/dev/null | sort -rh | head -15

# How big is each user's home directory?
sudo du -shc /home/* | sort -rh

# Single directory total
du -sh /var/log
du -sh /opt/conda/envs/ml_env    # ML conda environment size

# All directories under /var, 2 levels deep
du -h --max-depth=2 /var | sort -rh | head -20
AI/ML Specific — Model & Dataset Management

docker system df                 # Better: Docker-native usage
docker system prune              # Clean up unused layers

# Python venv sizes — find bloated environments
du -shc ~/.virtualenvs/* 2>/dev/null | sort -rh
du -shc ~/miniconda3/envs/* | sort -rh

# Log directory analysis — find which app logs are biggest

# Disk usage excluding compiled Python files

6Interview Questions & Answers
Q1Scenario: Production disk is 95% full. Walk me through how you investigate.
A:1. df -h — identify which filesystem is full. 2. df -i — check if it's inode exhaustion instead. 3. sudo du -hd1 /problem_mount 2>/dev/null | sort -rh | head -15 — find top directories. 4. Drill down: sudo du -hd2 /problem_dir | sort -rh | head. 5. lsof | grep deleted — check for deleted-but-open files. 6. Immediate relief: clear old logs /var/log/*.gz, Docker cleanup docker system prune, rotate logs. 7. Long-term: set up monitoring alerts at 80% threshold.
7Common Mistakes & Pro Tips
❌
Mistake — du on network filesystems du traverses every file, which can take hours on large NFS mounts or generate massive network traffic. Always use -x flag to stay on local filesystem: du -sx /.
✅
Pro Tip — ncdu for interactive exploration ncdu (ncurses disk usage) is a TUI version of du — navigable with arrow keys, much faster than the classic du | sort pipeline. Install: apt-get install ncdu. Use: sudo ncdu /.
mkfs
Filesystem Creation & Repair
Build Linux filesystems — mkfs, mkswap, fsck, e2fsck
1Command Overview

mkfs (make filesystem) creates a new filesystem on a partition or block device. It is a frontend for filesystem-specific tools: mkfs.ext4, mkfs.xfs, mkfs.btrfs, mkfs.vfat. mkswap prepares a swap area. fsck/e2fsck check and repair filesystems.

🚨
IRREVERSIBLY DESTRUCTIVE mkfs completely overwrites all data on the target device. mkfs.ext4 /dev/sda (instead of /dev/sda1) destroys the entire disk including partition table. There is NO undo. Verify device name multiple times before running.
mkfs — Create Filesystems
BASH
# Create ext4 filesystem (most common Linux FS)
sudo mkfs.ext4 /dev/sdb1

# With label (human-readable name — usable in fstab)
sudo mkfs.ext4 -L data_vol /dev/sdb1

# XFS — high-performance for large files (data engineering, ML storage)
sudo mkfs.xfs /dev/sdb1

# XFS with label
sudo mkfs.xfs -L mldata /dev/sdb1

# Btrfs — modern FS with snapshots and RAID built-in
sudo mkfs.btrfs /dev/sdb1

BASH

# Boot to rescue mode or unmount first

# Force fsck on next reboot (system disk)

sudo touch /forcefsck         # Alternative method
6Interview Questions & Answers
Q1When would you choose XFS over ext4?
A:Choose XFS when: working with large files (video, ML datasets, genomic data, database files), high-throughput sequential workloads, filesystems >16TB (ext4 max is 1EB but XFS performs better at scale), or when using RHEL/CentOS (XFS is the default). Choose ext4 when: general-purpose use, systems with many small files, need for easy recovery with e2fsck, or when you need to shrink the filesystem (XFS cannot shrink). For Kafka data directories or Elasticsearch indices — XFS is typically preferred in production.
Q2Scenario: A server crashed and the filesystem is corrupted. How do you repair it?
A:1. Boot to rescue/single-user mode or boot from live USB. 2. UNMOUNT the affected filesystem — NEVER run fsck on a mounted FS. 3. sudo e2fsck -fp /dev/sdXN — auto-repair trivial errors. 4. If errors remain: sudo e2fsck -fy /dev/sdXN — answer yes to all prompts. 5. If corruption is severe, backup readable files first with ddrescue before running fsck. 6. After repair, remount and verify: mount /dev/sdXN /mnt && ls /mnt.
7Common Mistakes & Pro Tips
❌
Critical Mistake — Running fsck on mounted filesystem Running fsck on a mounted filesystem causes severe corruption — never do this. The filesystem MUST be unmounted first. Exception: journaling check tools can run on read-only mounted filesystems.
✅
Pro Tip — Filesystem choice for Docker overlay XFS with ftype=1 is required for Docker overlay2 storage driver (default since Docker 17.06). Always format Docker storage as: mkfs.xfs -n ftype=1 /dev/sdX. Missing this causes Docker to fall back to slower devicemapper.
mount / umount
Filesystem Mounting
Attach/detach filesystems + /etc/fstab for persistent mounts
1Command Overview

mount attaches a filesystem to a directory in the Linux directory tree (the mount point), making it accessible. umount detaches it. /etc/fstab is the configuration file defining mounts that persist across reboots — every permanent mount in production is configured here.

In cloud environments, mounting EBS volumes, NFS shares, EFS (AWS), GCS FUSE, S3 FUSE mounts for ML training data, and Kubernetes persistent volumes all rely on these fundamentals.

5Real-World Use Cases
  • AWS EBS volume mounting: New EBS → lsblk to find device → mkfs.xfs → mkdir /data → mount → blkid for UUID → add to /etc/fstab with nofail.
  • ML training data on NFS: Mount NFS/EFS storage with _netdev,nofail,noatime for training dataset access from multiple GPU nodes.
  • Docker bind mounts: mount --bind /host/data /container/data is the underlying mechanism for Docker -v volume mounts.
  • Kubernetes PersistentVolumes: The kubelet uses mount/umount to attach/detach CSI volumes to pods — understanding this is essential for K8s storage troubleshooting.
6Interview Questions & Answers
Q1What happens if /etc/fstab has an error and why is nofail important?
A:A bad fstab entry causes the system to drop into emergency/rescue mode at boot — the boot process halts waiting for the missing or corrupt device. This is catastrophic for production servers and cloud instances where no console access is available. The nofail option tells systemd to continue booting even if the specified device is unavailable. Always add nofail to data disks, NFS mounts, and optional volumes. Always test with sudo mount -a and sudo findmnt --verify before rebooting after fstab changes.
Q2What does noatime do and why is it a common production optimisation?
A:Normally Linux updates the access time (atime) metadata on every file READ — generating a write I/O for every read operation, effectively doubling disk writes. noatime disables this update, eliminating these write operations. On systems with many small file accesses (web servers, package managers, ML frameworks reading model weights), noatime can reduce disk I/O by 20-50%. It's a standard mount option in high-performance production deployments. Add to fstab: defaults,noatime.
7Common Mistakes & Pro Tips
❌
Mistake — fstab error locks out server Never reboot after changing fstab without testing first. Run sudo mount -a to catch errors immediately. A typo in fstab can render a cloud instance unreachable — requiring console/rescue mode access.
✅
Pro Tip — Always use nofail for data disks Root filesystem: no nofail (you want it to fail loudly). Data disks: always add nofail. NFS/network mounts: add both nofail and _netdev. This single habit prevents most fstab-related boot failures.
LVM
Logical Volume Management
pvcreate · pvdisplay · vgcreate · vgdisplay · lvcreate · lvdisplay · lvextend · lvreduce · vgremove · lvremove
1Command Overview — LVM Architecture

LVM (Logical Volume Manager) is an abstraction layer between physical storage and filesystems that enables flexible, live disk management — grow/shrink volumes without unmounting, add disks to existing volumes, create snapshots, and stripe across multiple disks. It is the standard storage approach for production Linux servers.

Complete LVM Setup — New Disk to Mounted Volume
BASH — Full LVM provisioning workflow
═══ STEP 1: Prepare the physical disks ═══
sudo fdisk /dev/sdb    # Create partition, set type to 8e (Linux LVM)
# OR use parted:

═══ STEP 3: Create Volume Group ═══

═══ STEP 4: Create Logical Volumes ═══
sudo lvcreate -L 50G -n lv_models vg_data    # Fixed 50GB
sudo lvcreate -l 100%FREE -n lv_datasets vg_data  # Rest of VG

Expanding LVM — Most Important Production Task
BASH — Live LV extension (no downtime required)
═══ SCENARIO: /opt/datasets is 90% full, add 100GB ═══

# Option A: Extend from existing free space in VG
vgs vg_data                    # Check VG free space
sudo lvextend -L +100G /dev/vg_data/lv_datasets
sudo xfs_growfs /opt/datasets  # Grow XFS filesystem ONLINE
# For ext4: sudo resize2fs /dev/vg_data/lv_datasets
df -h /opt/datasets            # Verify new size

# Option B: Add new disk to VG first, then extend LV
# 1. Add new disk as PV
sudo pvcreate /dev/sdd1
# 2. Extend VG
sudo vgextend vg_data /dev/sdd1
# 3. Extend LV to use all new space
sudo lvextend -l +100%FREE /dev/vg_data/lv_datasets
# 4. Grow filesystem
sudo xfs_growfs /opt/datasets

# One-liner: extend LV AND filesystem together (-r flag)
sudo lvextend -L +100G -r /dev/vg_data/lv_datasets
# -r = resize filesystem automatically after LV extension
LVM Snapshots — For Backups & Safe Upgrades
BASH
# Create snapshot before risky operation
sudo lvcreate -s -L 10G -n lv_root_snap /dev/vg_system/lv_root
# -s = snapshot, -L = snapshot reserve size, -n = snapshot name

# List snapshots
lvs -a | grep snap

# Mount snapshot (read-only backup inspection)
sudo mount -o ro /dev/vg_system/lv_root_snap /mnt/snap

# Rollback: merge snapshot back (REVERTS all changes!)
sudo umount /
sudo lvconvert --merge /dev/vg_system/lv_root_snap

# Remove snapshot when no longer needed
sudo lvremove /dev/vg_system/lv_root_snap
5Real-World Use Cases
  • Kubernetes worker nodes: RHEL/Ubuntu K8s nodes use LVM for flexible container runtime storage allocation — docker, containerd, and kubelet data directories on separate LVs.
  • Database servers: PostgreSQL, MySQL data directories on LVM LVs with snapshots for consistent backups without stopping the database.
  • ML training farms: Model checkpoints and dataset caches on LVM with easy online expansion as training data grows — add a new disk, vgextend, lvextend, xfs_growfs, done.
  • Zero-downtime volume expansion: Production databases need more space at 3AM — LVM allows online expansion without unmounting or stopping the service.
6Interview Questions & Answers
Q1What is the difference between lvextend and lvresize?
A:lvextend only grows an LV — attempting to shrink with lvextend fails. lvresize can both grow and shrink. lvreduce only shrinks. Best practice: use lvextend -r or lvresize -r with the -r flag which automatically resizes the filesystem after changing the LV size — combining two steps into one. Always grow LV before filesystem; always shrink filesystem before LV.
Q2Scenario: A production server running out of disk space at 2AM. The root LV is 95% full. Walk through your resolution.
A:1. df -h — confirm it's the LVM root LV. 2. vgs — check if VG has free extents. If yes: 3a. sudo lvextend -r -L +20G /dev/vg_system/lv_root — grow instantly, no downtime. If VG is full: 3b. pvcreate /dev/sdb && vgextend vg_system /dev/sdb && lvextend -r -l +100%FREE /dev/vg_system/lv_root. If no new disk: 3c. Immediate relief: journalctl --vacuum-size=1G, apt-get clean, remove old kernels: apt-get autoremove. The beauty of LVM is that expansion is online, live, and safe.
Q3Why is shrinking an LV more dangerous than growing it?
A:Growing is safe — you're adding new empty space. Shrinking truncates the storage, potentially cutting off data. The process must be: (1) OFFLINE — unmount the filesystem first, (2) shrink the filesystem to less than the target size: resize2fs /dev/vg/lv NEW_SIZE, (3) THEN shrink the LV: lvreduce -L NEW_SIZE /dev/vg/lv. If you shrink the LV BEFORE the filesystem, you truncate the filesystem's data, causing severe corruption. XFS CANNOT be shrunk at all — plan XFS sizes carefully upfront.
Q4What are LVM snapshots and how are they used in production?
A:LVM snapshots capture the state of an LV at a point in time using copy-on-write (COW) — only blocks that change after the snapshot are copied. Production uses: (1) consistent database backups without stopping postgres/mysql — snapshot the LV, back up from the snapshot, delete snapshot, (2) safe OS upgrades — snapshot root LV, run apt-get upgrade, merge snapshot to rollback if it breaks, (3) testing — snapshot production data, run destructive tests on the snapshot, remove it when done. Snapshots are NOT a substitute for backups — they live on the same disk.
7Common Mistakes & Pro Tips
❌
Mistake — Shrinking XFS XFS filesystems CANNOT be shrunk. If you need a smaller XFS volume, you must backup, recreate with mkfs.xfs, and restore. Plan XFS volume sizes carefully and allocate conservatively — you can always grow later.
❌
Mistake — lvreduce before resize2fs Always shrink the filesystem first, THEN shrink the LV. Reverse order causes data truncation and filesystem corruption. Order: umount → e2fsck -f → resize2fs (smaller) → lvreduce.
✅
Pro Tip — Use -r flag for lvextend sudo lvextend -r -L +20G /dev/vg/lv — the -r flag automatically runs resize2fs or xfs_growfs after extending the LV. One command instead of two, reduces human error.
✅
Pro Tip — pvs/vgs/lvs for scripting Use pvs, vgs, lvs instead of pvdisplay, vgdisplay, lvdisplay in scripts — compact, parseable output. Add --noheadings -o name,size,vgname for clean CSV-like output for automation.
